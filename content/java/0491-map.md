## 1. What it is

`Stream.map(function)` returns a new stream where each element is the result of applying the given `Function<T, R>` to the corresponding element of the source stream. It's a one-to-one transformation: exactly one output element per input element, potentially of a completely different type (`Stream<T>` becomes `Stream<R>`). Like `filter`, it's an intermediate, lazy operation.

## 2. Why & when

`map` is the streams equivalent of transforming every element of a collection in a loop and collecting the results into a new list — but expressed as one declarative step. You reach for it whenever you need to convert each element into something else: extracting a field from an object, converting units, wrapping a value, parsing a string into a number. Because it preserves stream length (one input, one output, always), it's the tool for *transformation*, distinct from `filter` (which changes length by dropping elements) and `flatMap` (which can change length by expanding each element into several).

## 3. Core concept

```java
import java.util.stream.*;

List<Integer> lengths = Stream.of("apple", "kiwi", "fig")
        .map(String::length)
        .toList(); // [5, 4, 3]
```

`map` applies the function to each element independently, preserving order and count — the output stream has exactly as many elements as the input.

## 5. Runnable example

Scenario: converting a shopping cart's line items into display strings — evolved from a single-field extraction, through a multi-field transformation into a new type, to a version that chains multiple `map` steps into a small transformation pipeline.

### Level 1 — Basic

```java
import java.util.*;
import java.util.stream.*;

public class MapBasic {
    record Item(String name, double price) {}

    public static void main(String[] args) {
        List<Item> cart = List.of(
                new Item("Book", 12.99),
                new Item("Pen", 1.50)
        );

        List<String> names = cart.stream()
                .map(Item::name)
                .toList();

        System.out.println("Items: " + names);
    }
}
```

**How to run:** `java MapBasic.java`

Expected output:
```
Items: [Book, Pen]
```

`.map(Item::name)` transforms each `Item` into just its `name` field — a `Stream<Item>` becomes a `Stream<String>`, one name per item, same order and count as the source.

### Level 2 — Intermediate

```java
import java.util.*;
import java.util.stream.*;

public class MapToDisplayLine {
    record Item(String name, double price) {}
    record DisplayLine(String text) {}

    public static void main(String[] args) {
        List<Item> cart = List.of(
                new Item("Book", 12.99),
                new Item("Pen", 1.50)
        );

        List<DisplayLine> lines = cart.stream()
                .map(item -> new DisplayLine(item.name() + " - $" + String.format("%.2f", item.price())))
                .toList();

        lines.forEach(line -> System.out.println(line.text()));
    }
}
```

**How to run:** `java MapToDisplayLine.java`

Expected output:
```
Book - $12.99
Pen - $1.50
```

The real-world concern this adds: instead of extracting a single existing field, each `Item` is now transformed into a brand-new type (`DisplayLine`) built by combining *multiple* fields (`name` and formatted `price`) into one computed string — showing that the mapping function can be arbitrarily complex as long as it's a pure one-input-to-one-output transformation.

### Level 3 — Advanced

```java
import java.util.*;
import java.util.stream.*;

public class MapPipeline {
    record Item(String name, double price, double taxRate) {}
    record PricedLine(String name, double finalPrice) {}

    public static void main(String[] args) {
        List<Item> cart = List.of(
                new Item("Book", 12.99, 0.0),
                new Item("Laptop", 999.00, 0.08),
                new Item("Pen", 1.50, 0.08)
        );

        List<String> receipt = cart.stream()
                .map(item -> new PricedLine(item.name(), item.price() * (1 + item.taxRate()))) // apply tax
                .map(line -> new PricedLine(line.name(), Math.round(line.finalPrice() * 100) / 100.0)) // round to cents
                .map(line -> line.name() + ": $" + line.finalPrice()) // format for display
                .toList();

        receipt.forEach(System.out::println);
    }
}
```

**How to run:** `java MapPipeline.java`

Expected output:
```
Book: $12.99
Laptop: $1078.92
Pen: $1.62
```

This chains **three** `.map(...)` calls into a small pipeline, each doing one focused transformation: first applying tax to get a raw `PricedLine`, then rounding that price to the nearest cent (still a `PricedLine`), then formatting the final result as a display `String`. Splitting the work into named, single-purpose `map` steps like this keeps each transformation easy to reason about independently, rather than cramming tax, rounding, and formatting into one dense lambda.

## 6. Walkthrough

Execution starts in `main` of the Level 3 example. `cart` holds three items, each with a `price` and a `taxRate`.

`cart.stream()` begins the pipeline. The first `.map(item -> new PricedLine(item.name(), item.price() * (1 + item.taxRate())))` runs per element: for `Item("Book", 12.99, 0.0)`, it computes `12.99 * (1 + 0.0) = 12.99`, producing `PricedLine("Book", 12.99)`. For `Item("Laptop", 999.00, 0.08)`, it computes `999.00 * 1.08 = 1078.92`, producing `PricedLine("Laptop", 1078.92)`. For `Item("Pen", 1.50, 0.08)`, it computes `1.50 * 1.08 = 1.62`, producing `PricedLine("Pen", 1.62)`.

The second `.map(line -> new PricedLine(line.name(), Math.round(line.finalPrice() * 100) / 100.0))` rounds each `finalPrice` to two decimal places: `Math.round(12.99 * 100) / 100.0 = Math.round(1299.0) / 100.0 = 1299 / 100.0 = 12.99` (unchanged, already exact). `Math.round(1078.92 * 100) / 100.0 = 107892 / 100.0 = 1078.92` (unchanged). `Math.round(1.62 * 100) / 100.0 = 162 / 100.0 = 1.62` (unchanged) — in this particular data set the rounding step happens not to change anything, but it guards against floating-point noise from the tax multiplication in general.

The third `.map(line -> line.name() + ": $" + line.finalPrice())` formats each `PricedLine` into its final display `String`: `"Book: $12.99"`, `"Laptop: $1078.92"`, `"Pen: $1.62"`.

```
Item(Book,12.99,0.0)   -> tax -> PricedLine(Book,12.99)   -> round -> 12.99   -> format -> "Book: $12.99"
Item(Laptop,999,0.08)  -> tax -> PricedLine(Laptop,1078.92)-> round -> 1078.92-> format -> "Laptop: $1078.92"
Item(Pen,1.50,0.08)    -> tax -> PricedLine(Pen,1.62)     -> round -> 1.62    -> format -> "Pen: $1.62"
```

`.toList()` collects the three formatted strings, and `receipt.forEach(System.out::println)` prints each on its own line, in the original cart order.

## 7. Gotchas & takeaways

> `map` must be one-to-one: the mapping function returns exactly one value per input element. If a source element should sometimes contribute *zero* or *multiple* output elements (e.g. splitting a string into words, or skipping invalid entries), `map` is the wrong tool — reach for `flatMap` instead, which can expand or collapse element counts.

- `map(function)` transforms each element independently into exactly one output element, preserving the original order and count.
- The result type can differ entirely from the source type (`Stream<Item>` to `Stream<String>`), since `map` takes a `Function<T, R>`.
- Chaining multiple `.map(...)` calls, each doing one focused transformation, is often clearer than one dense lambda that does everything at once — each step stays independently readable and testable.
- `map` is lazy like `filter` — nothing runs until a terminal operation drives the pipeline.
- Don't reach for `map` when an input element could map to zero or more than one output element — that's `flatMap`'s job, not `map`'s.
