## 1. What it is

G1 ("Garbage First") is the default garbage collector on modern HotSpot JVMs (since Java 9), designed to give reasonably low, *predictable* pause times on large heaps without giving up much throughput. Instead of the traditional fixed young/old generation split occupying two large contiguous blocks, G1 divides the entire heap into many small, equally-sized regions (typically 1–32 MB each, chosen automatically based on heap size) and assigns each region a role — Eden, Survivor, or Old — dynamically, rather than fixing generation boundaries in advance. Its defining trick is choosing which regions to collect based on how much garbage each one contains, always picking the most garbage-rich regions first ("garbage-first," hence the name) — this lets G1 reclaim the most memory for the least collection effort, and lets it collect incrementally, a handful of regions at a time, rather than an entire generation at once, which is what makes its pauses both shorter and more predictable than [Parallel GC](0930-parallel-gc.md)'s.

## 2. Why & when

G1 targets a genuinely different goal than Parallel GC's raw throughput or Serial GC's minimal footprint: a **configurable, soft pause-time target** (`-XX:MaxGCPauseMillis`, defaulting to 200ms), which it tries to meet by choosing how many regions to collect in any given increment — collect fewer regions per pause to stay under the target, at the cost of needing more total pauses. This makes it the right general-purpose default for most server applications with heaps from roughly a few hundred megabytes up to many gigabytes, where some pause exists but an unbounded multi-second stop-the-world event (as Serial or Parallel GC could produce on a large heap) is unacceptable. It is not the right choice when pause times must be kept in the single-digit-millisecond range regardless of heap size — G1's remaining stop-the-world phases (evacuation pauses) still scale somewhat with the number of regions being evacuated, which is exactly the gap [ZGC](0933-zgc.md) and [Shenandoah](0934-shenandoah.md) close by making even the actual object-moving phase mostly concurrent.

## 3. Core concept

```
Heap divided into many small, equal-sized REGIONS (not two fixed generations):

[E][E][S][O][O][E][ ][O][E][O][S][ ][O][E][ ][O] ...
 E=Eden  S=Survivor  O=Old  [ ]=unused/free

G1 collection cycle:
  1. Concurrent marking -- trace liveness across old-gen regions, MOSTLY without pausing
  2. Pick the N most garbage-rich regions that fit the pause-time budget
     (MaxGCPauseMillis) -- "garbage first"
  3. Evacuate (copy) survivors out of those N regions into fresh ones -- STOP-THE-WORLD,
     but only for those N regions, not the whole generation
  4. Reclaimed regions become free -- naturally compacted, since evacuation is a
     copying-style operation
```

Because collection happens region-by-region rather than generation-at-once, G1 can bound each individual pause by simply choosing fewer regions to evacuate, trading pause frequency for pause duration to hit its target.

## 5. Runnable example

Scenario: observe G1's region-based, pause-target-driven behavior — starting with a baseline default-G1 workload, then adding retained old-generation data to trigger concurrent marking and mixed collections, then tuning the pause-time target explicitly to see G1 trade pause frequency for pause duration.

### Level 1 — Basic

```java
public class G1Baseline {
    public static void main(String[] args) {
        for (int i = 0; i < 3_000_000; i++) {
            byte[] garbage = new byte[64];
        }
        System.out.println("done -- check -Xlog:gc for 'Pause Young (Normal) (G1 Evacuation Pause)' events");
    }
}
```

**How to run:** `java -Xlog:gc -XX:+UseG1GC G1Baseline.java` (JDK 17+; G1 is already the default on JDK 9+, the flag is explicit here for clarity).

Expected output shape:
```
[0.02s][info][gc] GC(0) Pause Young (Normal) (G1 Evacuation Pause) ... 6M->1M(64M) 1.6ms
[0.04s][info][gc] GC(1) Pause Young (Normal) (G1 Evacuation Pause) ... 7M->1M(64M) 1.3ms
...
done -- check -Xlog:gc for 'Pause Young (Normal) (G1 Evacuation Pause)' events
```

Even the young-generation pauses are labeled "G1 Evacuation Pause" rather than a fixed-generation copy, reflecting that G1 treats every collection, young or mixed, as evacuating a chosen set of regions.

### Level 2 — Intermediate

```java
import java.util.*;

public class G1MixedCollections {
    public static void main(String[] args) {
        List<byte[]> retained = new ArrayList<>();
        for (int i = 0; i < 300_000; i++) {
            retained.add(new byte[300]);
            if (i % 5 == 0 && !retained.isEmpty()) {
                retained.remove(retained.size() / 2); // create garbage-rich old regions
            }
        }
        System.out.println("retained: " + retained.size());
        System.out.println("check the log for 'Pause Young (Concurrent Start)' then 'Pause Young (Mixed)'");
    }
}
```

**How to run:** `java -Xlog:gc -Xmx200m -XX:+UseG1GC G1MixedCollections.java` (JDK 17+).

Expected output shape:
```
[0.05s][info][gc] GC(0) Pause Young (Normal) (G1 Evacuation Pause) ... 6M->1M(200M) 1.5ms
...
[0.30s][info][gc] GC(15) Pause Young (Concurrent Start) (G1 Evacuation Pause) ... 90M->40M(200M) 4.2ms
[0.35s][info][gc] GC(16) Pause Young (Mixed) (G1 Evacuation Pause) ... 95M->35M(200M) 3.8ms
retained: 300000
check the log for 'Pause Young (Concurrent Start)' then 'Pause Young (Mixed)'
```

The real-world concern added: sustained old-generation retention (with the deliberate mid-list removals creating garbage-rich old regions) triggers G1's concurrent marking cycle ("Concurrent Start"), which then feeds into "Mixed" collections — pauses that evacuate both young regions and a selected set of the most garbage-rich old regions together, exactly G1's "garbage first" strategy in action.

### Level 3 — Advanced

```java
import java.util.*;

public class G1PauseTargetTuning {
    public static void main(String[] args) {
        List<byte[]> retained = new ArrayList<>();
        long start = System.nanoTime();
        for (int i = 0; i < 500_000; i++) {
            retained.add(new byte[300]);
            if (i % 5 == 0 && retained.size() > 10) {
                retained.remove(retained.size() / 3);
            }
        }
        long elapsedMs = (System.nanoTime() - start) / 1_000_000;
        System.out.println("retained: " + retained.size() + ", elapsed: " + elapsedMs + "ms");
    }
}
```

**How to run:** compare a tight versus loose pause-time target: `java -Xlog:gc -Xmx200m -XX:+UseG1GC -XX:MaxGCPauseMillis=20 G1PauseTargetTuning.java` versus `java -Xlog:gc -Xmx200m -XX:+UseG1GC -XX:MaxGCPauseMillis=500 G1PauseTargetTuning.java` (JDK 17+).

Expected output shape (illustrative):
```
(MaxGCPauseMillis=20):  many short pauses (~15-20ms each), more of them, elapsed: 620ms
(MaxGCPauseMillis=500): fewer, longer pauses (up to ~500ms), elapsed: 540ms
```

The production-flavored hard case: tightening the pause-time target forces G1 to evacuate fewer regions per pause (to stay under budget), producing more frequent but shorter pauses at some throughput cost, while loosening it lets G1 batch more region evacuation into each pause for better overall throughput — directly demonstrating `MaxGCPauseMillis` as a real, tunable tradeoff knob between latency and throughput, not just a hint G1 ignores.

## 6. Walkthrough

Tracing `G1MixedCollections.main` end to end:

1. The loop retains growing numbers of small objects in `retained`, while periodically removing one from the middle of the list — since G1 allocates new objects into Eden regions, ordinary young collections ("Pause Young (Normal)") continue reclaiming short-lived garbage exactly as they would under any generational collector, evacuating survivors into fresh Survivor regions.
2. As `retained` grows, more objects survive enough young collections to be promoted into Old regions — and because objects are being removed from the *middle* of the list (not just the end), specific Old regions accumulate significant amounts of garbage interspersed with live data, rather than being either entirely live or entirely dead.
3. Once the old-generation occupancy crosses an internal threshold (driven here by the `-Xmx200m` cap), G1 starts a concurrent marking cycle — this begins with a "Pause Young (Concurrent Start)" pause (a young collection that also kicks off concurrent marking, folded into the same pause for efficiency) and then proceeds mostly concurrently with the application, tracing reachability across all Old regions without stopping application threads for the bulk of that work.
4. Concurrent marking produces, for every Old region, a "liveness" estimate — how much of that region's data is actually still reachable versus garbage — which is exactly the information G1 needs to rank regions by how garbage-rich they are.
5. The next pauses are labeled "Pause Young (Mixed)": G1 evacuates not just Eden and Survivor regions (as any young collection would) but *also* a selected batch of the most garbage-rich Old regions identified during marking — chosen specifically to fit within the configured pause-time budget, reclaiming the most memory per millisecond of pause spent.
6. This mixed-collection process repeats across several pauses until the marked garbage-rich regions have all been reclaimed, at which point G1 returns to ordinary young-only collections until old-generation occupancy rises enough to trigger another concurrent marking cycle — the program's final retained count and elapsed time reflect this steady-state cycle running correctly underneath, with the log evidence above showing each phase transition explicitly.

## 7. Gotchas & takeaways

> **Gotcha:** `-XX:MaxGCPauseMillis` is a *soft target*, not a hard guarantee — G1 tries to size its region-evacuation batches to hit it, but under sufficiently extreme allocation pressure (allocating faster than concurrent marking and mixed collection can keep up) it can still fall back to a full, old-fashioned stop-the-world full GC across the entire heap, which can take far longer than the configured target; monitoring for "Pause Full" events in the GC log is the way to catch this in production.

- G1 divides the heap into many small, equally-sized regions and assigns them Eden/Survivor/Old roles dynamically, rather than using two large fixed generations.
- It always evacuates the most garbage-rich regions first ("garbage first"), which lets it reclaim more memory per unit of pause time than an undifferentiated whole-generation collection.
- `-XX:MaxGCPauseMillis` (default 200ms) is a soft, tunable pause-time target that G1 tries to hit by choosing how many regions to evacuate per pause — tightening it trades throughput for shorter, more frequent pauses.
- Concurrent marking (mostly non-pausing) determines each Old region's liveness, feeding directly into which regions get selected for "Mixed" collections alongside ordinary young evacuation.
- Under extreme allocation pressure, G1 can still fall back to a full stop-the-world collection across the entire heap — this is the failure mode worth monitoring for, since it defeats the whole point of using G1 in the first place.
- See [CMS (removed)](0931-cms-removed.md) for the collector G1 replaced and the fragmentation problem G1's incremental compaction specifically solves, and [ZGC](0933-zgc.md) / [Shenandoah](0934-shenandoah.md) for collectors that push pause times even lower by making evacuation itself mostly concurrent.
