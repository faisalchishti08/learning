## 1. What it is

Beyond the [canonical constructor](0954-record-components-canonical-constructor.md), every record automatically gets four more members generated by the compiler, purely from its declared components: an **accessor method** per component (named exactly after the component — `name()` for a component called `name`, not `getName()`), an **`equals(Object)`** that returns true if and only if the other object is the same record type and every component's value matches (using `Object.equals` for reference types, and primitive `==` for primitives — with the one notable JDK-wide exception that `Double.compare`-style semantics apply to floating-point components, so `NaN` compares equal to `NaN` in a record's `equals`, unlike with `==`), a **`hashCode()`** that combines all components' hash codes in a way guaranteed consistent with that `equals()` (equal records always produce equal hash codes, upholding the fundamental `equals`/`hashCode` contract every Java object must respect), and a **`toString()`** that prints the record's class name followed by every component's name and value in declared order (`Point[x=1, y=2]`).

## 2. Why & when

These four generated members exist because implementing them correctly, by hand, for every simple data class was both extremely repetitive and a very common source of subtle bugs — a `hashCode()` that only considers some fields (breaking the contract with a more complete `equals()`), a `toString()` someone forgot to update after adding a new field, or field-by-field `equals()` logic with a typo comparing the wrong pair of fields. Because records generate all four consistently and correctly from the same single source of truth (the component list), you get this correctness for free and, just as importantly, it stays correct automatically as the record evolves — adding a new component automatically updates `equals`, `hashCode`, and `toString` to include it, with no separate code to remember to update. Understanding these generated implementations matters practically whenever you use records as map keys or set elements (correctness there depends entirely on `equals`/`hashCode` being consistent, which records guarantee by construction), and when debugging or logging (a record's `toString()` output is immediately informative without any manual formatting work).

## 3. Core concept

```
record Point(int x, int y) {}

// Auto-generated, roughly equivalent to:
public int x() { return x; }
public int y() { return y; }

public boolean equals(Object o) {
    if (this == o) return true;
    if (!(o instanceof Point other)) return false;
    return this.x == other.x && this.y == other.y;   // ALL components, by value
}

public int hashCode() {
    return Objects.hash(x, y);    // combines ALL components -- consistent with equals() above
}

public String toString() {
    return "Point[x=" + x + ", y=" + y + "]";   // ALL components, in DECLARED order
}
```

All four members are derived from exactly the same list of components, in exactly the same order — this single source of truth is precisely what guarantees they stay mutually consistent, automatically, as the record's component list changes over time.

## 5. Runnable example

Scenario: use a record as a map key, relying on its auto-generated `equals`/`hashCode`, evolving through the basic guarantee, then a realistic case using records in a `HashSet` for deduplication, then a subtler advanced case involving records that contain mutable reference-type components, exposing exactly what the generated `equals` does and doesn't protect against.

### Level 1 — Basic

```java
public class RecordEqualityBasic {
    record Point(int x, int y) {}

    public static void main(String[] args) {
        Point p1 = new Point(3, 4);
        Point p2 = new Point(3, 4);
        System.out.println("p1: " + p1);
        System.out.println("p1.equals(p2): " + p1.equals(p2));
        System.out.println("p1.hashCode() == p2.hashCode(): " + (p1.hashCode() == p2.hashCode()));
    }
}
```

**How to run:** `java RecordEqualityBasic.java` (JDK 17+).

Expected output:
```
p1: RecordEqualityBasic$Point[x=3, y=4]
p1.equals(p2): true
p1.hashCode() == p2.hashCode(): true
```

Even though `p1` and `p2` are two entirely distinct objects (created by two separate `new Point(...)` calls), the record's auto-generated `equals` compares them by *value* (both components match), returning `true`, and the auto-generated `hashCode` — derived from the exact same components — is guaranteed to match as well, upholding the `equals`/`hashCode` contract automatically.

### Level 2 — Intermediate

```java
import java.util.*;

public class RecordEqualityInSets {
    record Point(int x, int y) {}

    public static void main(String[] args) {
        Set<Point> uniquePoints = new HashSet<>();
        uniquePoints.add(new Point(1, 1));
        uniquePoints.add(new Point(2, 2));
        uniquePoints.add(new Point(1, 1)); // "duplicate" -- a DIFFERENT object, but equal VALUE

        System.out.println("unique points count: " + uniquePoints.size());
        System.out.println("contains (1,1): " + uniquePoints.contains(new Point(1, 1)));
    }
}
```

**How to run:** `java RecordEqualityInSets.java` (JDK 17+).

Expected output:
```
unique points count: 2
contains (1,1): true
```

The real-world concern added: `HashSet` relies entirely on `equals`/`hashCode` being correctly implemented and mutually consistent to deduplicate correctly — since the record generates both automatically from the same component list, adding a value-equal `Point(1, 1)` a second time is correctly recognized as a duplicate (the set stays at size 2, not 3), and `contains` correctly finds a value-equal point even though it's a different object reference than any of the ones actually stored.

### Level 3 — Advanced

```java
import java.util.*;

public class RecordEqualityWithMutableComponent {
    record Container(List<Integer> items) {}

    public static void main(String[] args) {
        List<Integer> sharedList = new ArrayList<>(List.of(1, 2, 3));
        Container c1 = new Container(sharedList);

        Set<Container> set = new HashSet<>();
        set.add(c1);
        System.out.println("contains c1 before mutation: " + set.contains(c1));

        sharedList.add(4); // mutating the LIST the record's component refers to -- the record ITSELF is unchanged,
                             // but what it's holding a reference to has changed underneath it

        System.out.println("contains c1 after mutation: " + set.contains(c1));
        System.out.println("c1: " + c1);
    }
}
```

**How to run:** `java RecordEqualityWithMutableComponent.java` (JDK 17+).

Expected output:
```
contains c1 before mutation: true
contains c1 after mutation: false
c1: RecordEqualityWithMutableComponent$Container[items=[1, 2, 3, 4]]
```

The production-flavored hard case: a record's own fields are genuinely final and cannot be reassigned, but if a component's *type* is itself mutable (here, `List<Integer>`), nothing prevents the referenced object from being mutated through some other reference (`sharedList`) that still points to the same underlying list — this doesn't violate the record's own immutability (the field still points to the same list object it always did), but it does mean `hashCode()` (computed from the list's current contents) changes after the mutation, which is exactly why `HashSet.contains` — which relies on a stable hash code to even locate the right bucket — can no longer find `c1`, despite `c1` itself being "the same object" throughout.

## 6. Walkthrough

Tracing `RecordEqualityWithMutableComponent.main` end to end:

1. `sharedList` is created as a mutable `ArrayList` containing `[1, 2, 3]`, and `c1` is constructed as a `Container` wrapping this exact list reference — the record's `items` field now holds a reference to `sharedList`, not a copy of its contents.
2. `set.add(c1)` computes `c1.hashCode()` (derived from `items.hashCode()`, which in turn is derived from the current contents of `sharedList`, `[1, 2, 3]`) to determine which internal bucket to store `c1` in, and stores it there.
3. `set.contains(c1)` (checked before mutation) recomputes `c1.hashCode()` — since nothing has changed yet, it's identical to the hash code used when `c1` was originally added, so the set correctly locates the same bucket, finds `c1` (or an equal object) there via `equals`, and returns `true`.
4. `sharedList.add(4)` mutates the list object itself, appending `4` — critically, this does *not* touch `c1`'s `items` field at all (that field still holds the exact same reference it always did); what changes is the *contents* of the object that reference points to.
5. `set.contains(c1)` (checked after mutation) recomputes `c1.hashCode()` fresh — but since `items` now contains `[1, 2, 3, 4]` instead of `[1, 2, 3]`, this freshly-computed hash code differs from the one that was used back when `c1` was originally inserted into the set; `HashSet` uses this hash code to decide which bucket to look in, and it now looks in the *wrong* bucket (the one matching the new, mutated contents) — `c1` is still physically stored in its *original* bucket, based on the *original* hash code, so the lookup fails, and `contains` returns `false`, even though `c1` genuinely is still present in the set's internal storage.
6. `System.out.println(c1)` confirms the record's own `toString()` correctly reflects the *current*, mutated state of `items` (`[1, 2, 3, 4]`) — demonstrating precisely that a record's immutability guarantee is about its own fields never being reassigned, not about the transitive immutability of whatever objects those fields happen to reference; this is exactly the nuance explored further in [records & immutability](0958-records-immutability.md).

## 7. Gotchas & takeaways

> **Gotcha:** storing a record with a mutable-type component (a `List`, `Map`, array, or any other mutable object) in a `HashSet` or as a `HashMap` key is genuinely dangerous — mutating that component after insertion, even through some other reference entirely, can silently break the collection's internal lookup structure exactly as shown above, without throwing any exception or otherwise signaling anything went wrong; either use only genuinely immutable component types (like `List.of(...)`'s unmodifiable lists, or better, other records) in any record you intend to use as a hash-based key, or defensively copy mutable components in the canonical constructor.

- Records auto-generate an accessor, `equals`, `hashCode`, and `toString` for their components — all four derived from the same component list, guaranteeing they stay mutually consistent as the record evolves.
- The generated `equals` compares by value across all components (using each component type's own `equals`, or primitive comparison), and `hashCode` is guaranteed consistent with it — correctly supporting use as `HashSet` elements or `HashMap` keys, as long as component types behave well.
- A record's own fields cannot be reassigned, but if a component's type is itself mutable, the object it references can still be mutated through any other reference that also points to it — this does not violate the record's immutability but can silently corrupt hash-based collection lookups relying on that component's hash code staying stable.
- `toString()` always reflects the *current* state of each component, including any mutation of a mutable component's contents that happened after construction.
- Prefer genuinely immutable component types in any record you intend to use as a hash-based key, or defensively copy mutable components in the canonical constructor to fully guarantee the record's effective immutability.
- See [record components & canonical constructor](0954-record-components-canonical-constructor.md) for how the canonical constructor's validation and normalization interact with these generated members, and [records & immutability](0958-records-immutability.md) for the precise scope of what a record's immutability guarantee actually covers.
