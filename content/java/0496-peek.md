## 1. What it is

`Stream.peek(consumer)` returns a stream identical to the source, but runs a given `Consumer<T>` on each element as it passes through — purely as a side effect, without changing the element or the stream's contents. It's an intermediate operation, so like `filter` and `map`, it's lazy: the consumer only runs when a terminal operation actually pulls elements through the pipeline.

## 2. Why & when

`peek` exists mainly for **debugging** and **observing** a pipeline without altering it — printing each element as it flows through a chain of `filter`/`map` calls, to see exactly what survives each stage. Because it returns the stream unchanged, you can drop a `.peek(...)` call anywhere in an existing pipeline temporarily, inspect what's happening, and remove it again without touching the actual logic.

It is *not* meant for doing real work — mutating external state, populating a collection, or anything the correctness of your program depends on. The Java documentation explicitly warns that `peek`'s consumer may not run at all for some elements if the stream's terminal operation doesn't need to visit them (e.g. `findFirst()` on an infinite or long stream may short-circuit before peeking every element), so treat it strictly as an observation tool, not a mechanism to trigger required side effects.

## 3. Core concept

```java
import java.util.stream.*;

long count = Stream.of(1, 2, 3, 4, 5)
        .peek(n -> System.out.println("Saw: " + n)) // observe, unchanged
        .filter(n -> n % 2 == 0)
        .peek(n -> System.out.println("Passed filter: " + n))
        .count();
```

`peek` doesn't transform or drop anything — the elements flowing out are identical to the elements flowing in; only the side-effecting print happens in between.

## 5. Runnable example

Scenario: debugging a data-validation pipeline for incoming sensor readings — evolved from a single `peek` used to trace what enters a filter, through multiple `peek` calls tracing each pipeline stage, to a version that demonstrates why `peek` must never be relied on for required side effects.

### Level 1 — Basic

```java
import java.util.stream.*;

public class PeekBasic {
    public static void main(String[] args) {
        long validCount = Stream.of(72, -5, 98, 150, 65)
                .peek(reading -> System.out.println("Raw reading: " + reading))
                .filter(reading -> reading >= 0 && reading <= 100)
                .count();

        System.out.println("Valid readings: " + validCount);
    }
}
```

**How to run:** `java PeekBasic.java`

Expected output:
```
Raw reading: 72
Raw reading: -5
Raw reading: 98
Raw reading: 150
Raw reading: 65
Valid readings: 3
```

`.peek(...)` prints every raw reading exactly as it enters the pipeline, before `.filter(...)` decides whether to keep it. This lets you see the *input* to the filter stage, separate from its *output* (the final count of `3` valid readings between `0` and `100`).

### Level 2 — Intermediate

```java
import java.util.stream.*;

public class PeekMultiStage {
    public static void main(String[] args) {
        long validCount = Stream.of(72, -5, 98, 150, 65)
                .peek(r -> System.out.println("1. Raw: " + r))
                .filter(r -> r >= 0 && r <= 100)
                .peek(r -> System.out.println("2. Passed range check: " + r))
                .map(r -> r / 10 * 10) // round down to nearest 10
                .peek(r -> System.out.println("3. Rounded: " + r))
                .count();

        System.out.println("Valid readings: " + validCount);
    }
}
```

**How to run:** `java PeekMultiStage.java`

Expected output:
```
1. Raw: 72
2. Passed range check: 72
3. Rounded: 70
1. Raw: -5
1. Raw: 98
2. Passed range check: 98
3. Rounded: 90
1. Raw: 150
1. Raw: 65
2. Passed range check: 65
3. Rounded: 60
Valid readings: 3
```

The real-world concern this adds: three `.peek(...)` calls trace three separate pipeline stages at once. Notice each element is processed **fully through the pipeline before the next element starts** (streams process one element through the whole chain at a time, not stage by stage across all elements) — `72` prints at stage 1, then 2, then 3, before `-5` even appears at stage 1. `-5` and `150` never reach stage 2's print, since they're dropped by `.filter(...)` right after stage 1.

### Level 3 — Advanced

```java
import java.util.*;
import java.util.stream.*;

public class PeekPitfall {
    public static void main(String[] args) {
        List<Integer> readings = List.of(72, -5, 98, 150, 65);

        // WRONG: relying on peek to populate a "seen" list -- an unreliable side effect.
        List<Integer> seenWrong = new ArrayList<>();
        Optional<Integer> firstValid = readings.stream()
                .peek(seenWrong::add) // may not run for every element if the stream short-circuits
                .filter(r -> r >= 0 && r <= 100)
                .findFirst();

        System.out.println("First valid: " + firstValid.orElse(-1));
        System.out.println("Seen via peek (unreliable): " + seenWrong);

        // RIGHT: use map/forEach or a dedicated pass when a real side effect must always happen.
        List<Integer> seenCorrect = new ArrayList<>(readings);
        System.out.println("Seen via explicit copy (reliable): " + seenCorrect);
    }
}
```

**How to run:** `java PeekPitfall.java`

Expected output:
```
First valid: 72
Seen via peek (unreliable): [72]
Seen via explicit copy (reliable): [72, -5, 98, 150, 65]
```

This demonstrates the real pitfall: `.findFirst()` is a short-circuiting terminal operation — it stops pulling elements through the pipeline the instant it finds one match. Since `72` already satisfies the filter, the stream never needs to pull `-5`, `98`, `150`, or `65` at all, so `peek`'s consumer only ran once, leaving `seenWrong` with just `[72]` instead of all five readings — proving `peek` cannot be trusted to observe every element when the terminal operation doesn't require visiting all of them.

## 6. Walkthrough

Execution starts in `main` of the Level 3 example. `readings` holds five values: `72, -5, 98, 150, 65`.

`readings.stream().peek(seenWrong::add).filter(r -> r >= 0 && r <= 100).findFirst()` sets up a lazy pipeline. `findFirst()` is the terminal operation, and it's short-circuiting: it asks the pipeline for elements **one at a time** and stops as soon as one passes the filter.

The pipeline pulls the first element, `72`: `peek` runs `seenWrong.add(72)`, appending it to `seenWrong` (now `[72]`); `filter` checks `72 >= 0 && 72 <= 100`, both `true`, so `72` passes. `findFirst()` now has its answer — `72` — and stops pulling immediately. It never asks the pipeline for a second element, so `peek` is never invoked again.

```
findFirst() pulls element 1: peek adds 72 to seenWrong -> filter(72) passes -> STOP, answer found
(elements -5, 98, 150, 65 are never pulled -- peek never runs for them)
```

`firstValid` is `Optional.of(72)`, printed via `.orElse(-1)` as `"First valid: 72"`. `seenWrong`, which was meant to capture *every* reading via the side-effecting `peek`, only contains `[72]` — proof that relying on `peek` for a required side effect silently produces incomplete results whenever the terminal operation short-circuits.

The final two lines show the reliable alternative: `new ArrayList<>(readings)` explicitly copies all five readings with no dependency on how much of a lazy pipeline happens to get evaluated — `seenCorrect` correctly contains all five values, `[72, -5, 98, 150, 65]`.

## 7. Gotchas & takeaways

> Never rely on `peek()` to perform a side effect your program's correctness depends on. Short-circuiting terminal operations (`findFirst`, `findAny`, `anyMatch`, `limit` combined with a terminal op, etc.) may stop pulling elements before every element has passed through `peek`, silently skipping its consumer for the remainder of the stream — exactly as shown in Level 3.

- `peek(consumer)` observes each element with a side-effecting action but does not change the stream's elements or count.
- It's an intermediate, lazy operation — its consumer only runs as elements are actually pulled through by a terminal operation, and only for the elements that get pulled.
- It's best used for temporary debugging/tracing of a pipeline, safely added or removed without changing the pipeline's actual behavior.
- Short-circuiting terminal operations (`findFirst`, `anyMatch`, etc.) can cause `peek`'s consumer to run for only some — or even none — of the elements, since the pipeline stops early once an answer is found.
- For side effects your logic actually depends on (populating a list, updating a counter that must be accurate), use an explicit loop, `forEach` as the terminal operation, or a proper collection operation like `collect`/`toList` instead of `peek`.
