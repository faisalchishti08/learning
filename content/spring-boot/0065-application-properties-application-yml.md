## 1. What it is

`application.properties` and `application.yml` are the **primary configuration files** Spring Boot loads automatically at startup. You place them in `src/main/resources/` (or the working directory when running a packaged JAR) and Spring Boot finds them with no annotation, no XML, no explicit registration required.

Both files set the same properties — the difference is only the **syntax** used to express them. Every property that can be written in one format can be written in the other.

```properties
# application.properties
server.port=8080
spring.datasource.url=jdbc:mysql://localhost:3306/shop
spring.datasource.username=root
```

```yaml
# application.yml
server:
  port: 8080
spring:
  datasource:
    url: jdbc:mysql://localhost:3306/shop
    username: root
```

Both express the same three properties. Choose the format that suits your team.

## 2. Why & when

Before Spring Boot, configuring a Spring application required XML files or verbose Java `@Configuration` classes. Spring Boot introduced these auto-loaded files to give developers one obvious, zero-ceremony place to put configuration.

**Properties format** is the safe default:
- Simple key=value, every line self-contained.
- No indentation rules — copy-paste errors don't create silent bugs.
- Easy to grep for a specific key.
- Works well in environments where the file is auto-generated by a script.

**YAML format** shines when:
- You have many related keys sharing a prefix — YAML's nesting collapses repetition.
- You need **lists** — YAML has native list syntax; properties requires indexed keys (`list[0]=a`).
- You want to bundle **multiple profiles** in one file using YAML's `---` document separator (covered in the profile-specific files tutorial).
- The file is primarily read by humans rather than generated by scripts.

When not to use either format:
- Secrets (passwords, API keys) — keep those in env vars or a secrets manager so they don't land in version control.

## 3. Core concept

Think of both formats as **two dialects of the same language**. English and Spanish express the same ideas with different grammar. `application.properties` and `application.yml` express the same property set with different syntax.

**How they are loaded:**

Spring Boot's `ConfigDataLoader` searches for config files in this order of locations (highest priority last, meaning later entries override earlier ones within the same source tier):

1. Classpath root (`src/main/resources/`)
2. Classpath `/config/` subdirectory
3. Working directory
4. Working directory `/config/` subdirectory
5. Immediate subdirectories of working directory `/config/`

Files found in later locations override files found in earlier ones. Both `.properties` and `.yml` extensions are supported at every location. If both `application.properties` and `application.yml` exist in the same location, `.properties` wins for any key defined in both (YAML is processed first and properties processed second, so properties overwrites).

**Key format equivalence:**

| `.properties` | `.yml` |
|---|---|
| `server.port=8080` | `server:\n  port: 8080` |
| `spring.datasource.url=jdbc:…` | `spring:\n  datasource:\n    url: jdbc:…` |
| `my.list[0]=a` | `my:\n  list:\n    - a` |
| `my.list[1]=b` | `    - b` |

**YAML-only features:**
- Native list syntax (`- item`).
- Multi-document files (multiple profiles in one file with `---` separator).
- Anchors and aliases for DRY config (though rarely used in Spring Boot config).

**Properties-only consideration:**
- The `#` comment character works the same in both formats.
- In properties, colons in values don't need quoting. In YAML, `url: jdbc:mysql://…` is fine but `url: "…"` is safer when the value starts with a colon or contains special characters.

## 5. Runnable example

```java
// ConfigFilesDemo.java — Spring Boot 3.x project

import org.springframework.boot.SpringApplication;
import org.springframework.boot.autoconfigure.SpringBootApplication;
import org.springframework.boot.context.properties.ConfigurationProperties;
import org.springframework.boot.context.properties.EnableConfigurationProperties;
import org.springframework.stereotype.Component;
import org.springframework.web.bind.annotation.GetMapping;
import org.springframework.web.bind.annotation.RestController;

import java.util.List;

@SpringBootApplication
@EnableConfigurationProperties(ConfigFilesDemo.AppProps.class)
@RestController
public class ConfigFilesDemo {

    private final AppProps props;

    public ConfigFilesDemo(AppProps props) {
        this.props = props;
    }

    @GetMapping("/props")
    public String show() {
        return "port=" + props.getPort()
             + "  name=" + props.getName()
             + "  tags=" + props.getTags();
    }

    @ConfigurationProperties(prefix = "app")
    @Component
    public static class AppProps {
        private int port = 8080;
        private String name = "unnamed";
        private List<String> tags = List.of();

        public int getPort() { return port; }
        public void setPort(int port) { this.port = port; }
        public String getName() { return name; }
        public void setName(String name) { this.name = name; }
        public List<String> getTags() { return tags; }
        public void setTags(List<String> tags) { this.tags = tags; }
    }

    public static void main(String[] args) {
        SpringApplication.run(ConfigFilesDemo.class, args);
    }
}
```

**Option A — `application.properties`:**
```properties
# src/main/resources/application.properties
server.port=8080
app.port=9000
app.name=my-service
app.tags[0]=web
app.tags[1]=api
```

**Option B — `application.yml` (equivalent):**
```yaml
# src/main/resources/application.yml
server:
  port: 8080
app:
  port: 9000
  name: my-service
  tags:
    - web
    - api
```

**How to run:** pick one format (delete or omit the other). `./mvnw spring-boot:run` then `curl http://localhost:8080/props`.

Expected: `port=9000  name=my-service  tags=[web, api]`

## 6. Walkthrough

1. `@ConfigurationProperties(prefix = "app")` tells Spring Boot to bind all `app.*` properties to `AppProps` fields. This is the type-safe alternative to `@Value` for groups of related settings.
2. `app.tags[0]=web` (properties) and `- web` (YAML) both bind to the `List<String> tags` field. YAML's list syntax is cleaner when the list grows.
3. `@EnableConfigurationProperties(AppProps.class)` registers `AppProps` as a Spring bean so it can be injected.
4. In Option A, `app.tags[0]=web` and `app.tags[1]=api` are the indexed-property form. Spring Boot assembles these into a `List` in order.
5. In Option B, `tags:\n  - web\n  - api` expresses the same list. The YAML parser produces the same flat indexed representation internally.
6. The `server.port=8080` in the config file sets the embedded Tomcat port; `app.port=9000` is a custom property for the app's own logic — note they are different things.

## 7. Gotchas & takeaways

> YAML is **indentation-sensitive**. A single extra or missing space shifts a property under the wrong parent key, and the error message is often cryptic. Properties format is immune to indentation errors — each line is independent.

> If both `application.properties` and `application.yml` exist in the same classpath location, **properties wins** for any key that appears in both. This surprises teams that start with YAML and add a properties file for testing — the properties file silently overrides the YAML.

- Both formats produce the same flat property map in the `Environment`. There is no functional difference for simple scalar values.
- YAML wins for lists (`- item` vs. `key[0]=item`) and for readability when there are many keys sharing a prefix.
- Properties wins for simplicity, grep-ability, and scripted generation.
- Place either file in `src/main/resources/` for classpath loading, or alongside the JAR in the working directory for external (overriding) config.
- The `/config/` subdirectory of the working directory has higher priority than the working directory itself — use it to keep override files separate from other working-directory contents.
- `@ConfigurationProperties` is strongly preferred over many `@Value` annotations for groups of related settings: it gives you type safety, validation (`@Validated`), and IDE auto-complete.
