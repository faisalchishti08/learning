## 1. What it is

Spring Boot can automatically expose build metadata and git commit information in `/actuator/info` by reading two generated property files:
- **`META-INF/build-info.properties`** — generated by the Spring Boot Maven/Gradle plugin at build time. Contains artifact name, version, build timestamp, and group.
- **`git.properties`** — generated by the `git-commit-id` Maven/Gradle plugin. Contains branch name, commit SHA, commit time, commit message, and dirty-flag.

Both files are generated, not hand-written. Spring Boot's `BuildInfoContributor` and `GitInfoContributor` auto-detect them on the classpath.

## 2. Why & when

Every production deployment should answer: "which code is this?" Without build/git info:
- Support can't correlate a bug report to a commit.
- Ops can't verify the right build deployed without SSH-ing into the pod.
- Rolling deployments can't distinguish old vs. new pods.

Enable both in every project that uses Spring Boot Actuator. The overhead is a few KB of generated properties files.

## 3. Core concept

**Maven: generate `build-info.properties`**
```xml
<plugin>
  <groupId>org.springframework.boot</groupId>
  <artifactId>spring-boot-maven-plugin</artifactId>
  <executions>
    <execution>
      <goals><goal>build-info</goal></goals>
    </execution>
  </executions>
</plugin>
```

**Maven: generate `git.properties`**
```xml
<plugin>
  <groupId>io.github.git-commit-id</groupId>
  <artifactId>git-commit-id-maven-plugin</artifactId>
  <version>8.0.2</version>
  <executions>
    <execution>
      <goals><goal>revision</goal></goals>
    </execution>
  </executions>
</plugin>
```

**Gradle (equivalent):**
```groovy
springBoot { buildInfo() }  // generates build-info.properties
// use com.gorylenko.gradle-git-properties plugin for git.properties
```

Properties exposed in `/actuator/info`:
- **`build`**: `artifact`, `name`, `version`, `group`, `time` (ISO-8601 build timestamp).
- **`git`** (abbreviated mode, default): `branch`, `commit.id`, `commit.time`.
- **`git`** (full mode, `management.info.git.mode=full`): also `commit.message.short`, `commit.author.name`, `dirty`.

## 5. Runnable example

```java
// BuildGitInfoDemo.java — simulates reading build-info.properties and git.properties
// How to run: java BuildGitInfoDemo.java  (JDK 17+, no dependencies)
// Real Spring Boot: add the plugins; BuildInfoContributor/GitInfoContributor read these files

import java.util.*;

public class BuildGitInfoDemo {

    // Simulates build-info.properties content (generated by spring-boot-maven-plugin)
    static final Map<String, String> BUILD_INFO = Map.of(
        "build.artifact",    "order-service",
        "build.name",        "Order Service",
        "build.version",     "2.4.1",
        "build.group",       "com.example",
        "build.time",        "2025-06-01T10:00:00.000Z"
    );

    // Simulates git.properties content (generated by git-commit-id-maven-plugin)
    static final Map<String, String> GIT_INFO_SHORT = Map.of(
        "git.branch",              "main",
        "git.commit.id",           "abc1234",
        "git.commit.id.abbrev",    "abc1234",
        "git.commit.time",         "2025-05-30T08:30:00+0000",
        "git.dirty",               "false"
    );

    // management.info.git.mode=full adds these additional fields
    static final Map<String, String> GIT_INFO_FULL_EXTRA = Map.of(
        "git.commit.id.full",          "abc1234def5678901234567890abcdef12345678",
        "git.commit.message.short",    "feat: add order cancellation",
        "git.commit.author.name",      "Alice Dev",
        "git.remote.origin.url",       "https://github.com/example/order-service.git",
        "git.build.version",           "2.4.1"
    );

    static Map<String, Object> buildInfoSection(Map<String, String> props) {
        Map<String, Object> m = new LinkedHashMap<>();
        props.forEach((k, v) -> {
            // Strip "build." prefix, nest dot-separated keys
            String key = k.replace("build.", "");
            m.put(key, v);
        });
        return m;
    }

    static Map<String, Object> gitInfoSection(Map<String, String> base,
                                               Map<String, String> extra, boolean full) {
        Map<String, String> all = new LinkedHashMap<>(base);
        if (full) all.putAll(extra);
        // Build nested map from dot-separated keys
        Map<String, Object> result = new LinkedHashMap<>();
        all.forEach((k, v) -> {
            String nested = k.replace("git.", "");
            result.put(nested, v);
        });
        return result;
    }

    static void print(String label, Map<String, Object> info) {
        System.out.println("\n--- " + label + " ---");
        System.out.println("GET /actuator/info");
        info.forEach((k, v) -> {
            if (v instanceof Map<?,?> m) {
                System.out.println("  \"" + k + "\": { ... " + m.size() + " fields }");
                m.forEach((mk, mv) -> System.out.println("    \"" + mk + "\": \"" + mv + "\""));
            } else {
                System.out.println("  \"" + k + "\": \"" + v + "\"");
            }
        });
    }

    public static void main(String[] args) {
        System.out.println("=== Build & Git Info Demo ===");

        // Short mode (default)
        Map<String, Object> shortInfo = new LinkedHashMap<>();
        shortInfo.put("build", buildInfoSection(BUILD_INFO));
        shortInfo.put("git", gitInfoSection(GIT_INFO_SHORT, Map.of(), false));
        print("Default (management.info.git.mode=simple)", shortInfo);

        // Full mode
        Map<String, Object> fullInfo = new LinkedHashMap<>();
        fullInfo.put("build", buildInfoSection(BUILD_INFO));
        fullInfo.put("git", gitInfoSection(GIT_INFO_SHORT, GIT_INFO_FULL_EXTRA, true));
        print("Full (management.info.git.mode=full)", fullInfo);

        System.out.println("\n--- Plugin config summary ---");
        System.out.println("Maven — build-info:");
        System.out.println("  spring-boot-maven-plugin / goal: build-info");
        System.out.println("Maven — git.properties:");
        System.out.println("  io.github.git-commit-id:git-commit-id-maven-plugin / goal: revision");
        System.out.println("Gradle — build-info:");
        System.out.println("  springBoot { buildInfo() }");
        System.out.println("Gradle — git.properties:");
        System.out.println("  plugin: 'com.gorylenko.gradle-git-properties'");
        System.out.println("\nProperty: management.info.git.mode=full (default: simple)");
    }
}
```

**How to run:** `java BuildGitInfoDemo.java`

## 6. Walkthrough

- **`BUILD_INFO`** map simulates the `META-INF/build-info.properties` file. The Maven plugin generates it with keys like `build.version=2.4.1`. Spring's `BuildInfoContributor` parses these into the `build` section of `/actuator/info`.
- **`GIT_INFO_SHORT`** simulates the abbreviated `git.properties` output — the default `simple` mode. Only branch, abbreviated commit SHA, commit time, and dirty flag appear.
- **`GIT_INFO_FULL_EXTRA`** shows the additional fields added by `management.info.git.mode=full` — the full 40-char SHA, commit message, author name, and remote URL.
- Both maps are classpath resources checked at application startup. If either file is missing, the corresponding section is omitted from the info response — no error.
- The plugin configurations in the summary are the minimum additions needed in `pom.xml` / `build.gradle` to generate the files.

## 7. Gotchas & takeaways

> `build-info.properties` is **not generated by default** — you must add the `build-info` execution goal to `spring-boot-maven-plugin`. Without it, the `build` section of `/actuator/info` is empty.

> `management.info.git.mode=full` exposes the **remote origin URL and author name** — this may expose internal repository paths. Use `simple` mode in public-facing services.

- Check if `build-info.properties` was generated: look in `target/classes/META-INF/build-info.properties` after `mvn package`.
- If the app is built from a shallow git clone (common in CI), `git.properties` may report `dirty=true` or fail. Add `--unshallow` to the CI `git fetch` command.
- `build.time` uses the Maven build timestamp. For reproducible builds (same input → same JAR), set `project.build.outputTimestamp` in `pom.xml`.
- Combine with a custom `InfoContributor` to add deployment timestamp, deployer user, or target environment alongside the build info.
