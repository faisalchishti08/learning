## 1. What it is

Querydsl is a library that generates typed query classes from your entities (`QOrder`, `QCustomer`, and so on) at compile time, so you write filters as ordinary Java method calls on those generated types instead of as JPQL strings or Criteria API builder chains. `JpaRepository` picks this up automatically: if a repository also extends `QuerydslPredicateExecutor<T>`, you can pass a Querydsl `Predicate` straight into `findAll(predicate)`.

```java
QOrder order = QOrder.order;
Predicate predicate = order.total.gt(100).and(order.status.eq("SHIPPED"));
List<Order> results = orderRepository.findAll(predicate);
```

## 2. Why & when

Two earlier cards already cover dynamic queries: the JPQL card (raw query strings) and the Specifications card (Criteria API lambdas). Querydsl is a third option, and it sits between them — the fluency of a fluent Java API like Criteria, but without Criteria's verbose `cb.equal(root.get("x"), y)` style. Because `QOrder` is generated from the `Order` entity, a typo like `order.totall` is a compile error, not a runtime surprise discovered when a JPQL string fails to parse.

Reach for Querydsl specifically when:

- You want compile-time safety for dynamic filters (renaming an entity field should break the build, not silently return wrong results), which plain JPQL strings cannot give you.
- You find Criteria API code too verbose for straightforward property comparisons, but still want a type-checked, composable alternative to string queries.
- Your project already depends on the `querydsl-apt` annotation processor (it must run at build time to generate the `Q*` classes) — if not, adding it is the main cost of adopting Querydsl.

## 3. Core concept

```
 Build time:
   Order.java  --querydsl-apt-->  QOrder.java (generated, typed field accessors)

 Runtime:
   QOrder order = QOrder.order;
   Predicate p = order.total.gt(100).and(order.status.eq("SHIPPED"));
                         |
                         v
   interface OrderRepository extends JpaRepository<Order, Long>,
                                      QuerydslPredicateExecutor<Order> { }
                         |
                         v
   orderRepository.findAll(p)   -- translated to SQL by Querydsl + JPA provider
```

The generated `QOrder` class mirrors every field on `Order` as a typed expression object, so predicates read like plain Java rather than embedded SQL/JPQL text.

## 5. Runnable example

The scenario: filtering orders by total and status, evolving from a hand-built predicate class (standing in for a generated `QOrder`, since a single-file example cannot run an annotation processor) to composed predicates, to a repository wired with `QuerydslPredicateExecutor`.

### Level 1 — Basic

Model what `QOrder` gives you by hand: an expression object with typed field paths, used to build one predicate.

```java
import java.util.*;
import java.util.stream.*;

class Order {
    long id; double total; String status;
    Order(long id, double total, String status) { this.id = id; this.total = total; this.status = status; }
    public String toString() { return "Order{id=" + id + ", total=" + total + ", status='" + status + "'}"; }
}

// Stand-in for a Querydsl-generated QOrder: typed field accessors returning predicates.
class QOrder {
    static final QOrder order = new QOrder();
    java.util.function.Predicate<Order> totalGt(double v) { return o -> o.total > v; }
    java.util.function.Predicate<Order> statusEq(String v) { return o -> o.status.equals(v); }
}

public class QuerydslLevel1 {
    public static void main(String[] args) {
        List<Order> orders = List.of(
            new Order(1, 50, "SHIPPED"),
            new Order(2, 150, "SHIPPED"),
            new Order(3, 200, "PENDING")
        );

        QOrder order = QOrder.order;
        java.util.function.Predicate<Order> predicate = order.totalGt(100);

        List<Order> results = orders.stream().filter(predicate).collect(Collectors.toList());
        System.out.println(results);
    }
}
```

How to run: `java QuerydslLevel1.java`

`QOrder` stands in for the class the `querydsl-apt` processor would generate from an `Order` JPA entity: one typed accessor per field, each returning something predicate-like. `order.totalGt(100)` reads like a sentence — "order total greater than 100" — which is the whole appeal versus `cb.greaterThan(root.get("total"), 100)`.

### Level 2 — Intermediate

Compose two predicates with `.and()`, matching how a real Querydsl `BooleanExpression` composes.

```java
import java.util.*;
import java.util.stream.*;
import java.util.function.Predicate;

class Order {
    long id; double total; String status;
    Order(long id, double total, String status) { this.id = id; this.total = total; this.status = status; }
    public String toString() { return "Order{id=" + id + ", total=" + total + ", status='" + status + "'}"; }
}

class QOrder {
    static final QOrder order = new QOrder();
    Predicate<Order> totalGt(double v) { return o -> o.total > v; }
    Predicate<Order> statusEq(String v) { return o -> o.status.equals(v); }
}

public class QuerydslLevel2 {
    public static void main(String[] args) {
        List<Order> orders = List.of(
            new Order(1, 50, "SHIPPED"),
            new Order(2, 150, "SHIPPED"),
            new Order(3, 200, "PENDING")
        );

        QOrder order = QOrder.order;
        Predicate<Order> predicate = order.totalGt(100).and(order.statusEq("SHIPPED"));

        List<Order> results = orders.stream().filter(predicate).collect(Collectors.toList());
        System.out.println(results);
    }
}
```

How to run: `java QuerydslLevel2.java`

`Predicate.and(...)` is the JDK's own composition method, mirroring how a real Querydsl `BooleanExpression` (which `order.total.gt(100)` actually returns) exposes `.and()`/`.or()` to combine conditions — the query "orders over 100 that are SHIPPED" is now one composed object, not a hand-written conditional.

### Level 3 — Advanced

Wire the predicate into a repository-shaped abstraction that filters, and add a `null`-safe builder so optional filter criteria (e.g., a search form where any field may be blank) only contribute a condition when actually supplied — the same problem a real `QuerydslPredicateExecutor.findAll(predicate)` call solves in production.

```java
import java.util.*;
import java.util.stream.*;
import java.util.function.Predicate;

class Order {
    long id; double total; String status;
    Order(long id, double total, String status) { this.id = id; this.total = total; this.status = status; }
    public String toString() { return "Order{id=" + id + ", total=" + total + ", status='" + status + "'}"; }
}

class QOrder {
    static final QOrder order = new QOrder();
    Predicate<Order> totalGt(double v) { return o -> o.total > v; }
    Predicate<Order> statusEq(String v) { return o -> o.status.equals(v); }
}

// Stand-in for: interface OrderRepository extends JpaRepository<Order, Long>, QuerydslPredicateExecutor<Order> { }
class OrderRepository {
    private final List<Order> data;
    OrderRepository(List<Order> data) { this.data = data; }
    List<Order> findAll(Predicate<Order> predicate) {
        return data.stream().filter(predicate).collect(Collectors.toList());
    }
}

public class QuerydslLevel3 {
    // Builds a predicate from optional filter criteria, skipping any that are null.
    static Predicate<Order> buildPredicate(Double minTotal, String status) {
        QOrder order = QOrder.order;
        Predicate<Order> predicate = o -> true; // identity: no filters yet
        if (minTotal != null) predicate = predicate.and(order.totalGt(minTotal));
        if (status != null) predicate = predicate.and(order.statusEq(status));
        return predicate;
    }

    public static void main(String[] args) {
        OrderRepository repo = new OrderRepository(List.of(
            new Order(1, 50, "SHIPPED"),
            new Order(2, 150, "SHIPPED"),
            new Order(3, 200, "PENDING")
        ));

        // Simulates a search form where the user only filled in "status".
        Predicate<Order> onlyStatus = buildPredicate(null, "SHIPPED");
        System.out.println("Status only: " + repo.findAll(onlyStatus));

        // Simulates a search form where the user filled in both fields.
        Predicate<Order> both = buildPredicate(100.0, "SHIPPED");
        System.out.println("Both filters: " + repo.findAll(both));
    }
}
```

How to run: `java QuerydslLevel3.java`

Real search screens rarely supply every filter field, so a production `Predicate` builder must skip absent criteria rather than compare against `null`. `buildPredicate` starts from an always-true identity predicate and conditionally `.and()`s in each real condition — the same pattern a real Querydsl `BooleanBuilder` provides, just spelled out with `java.util.function.Predicate`.

## 6. Walkthrough

Execution starts in `main`. First, `repo` is constructed with three in-memory orders, standing in for rows the `QuerydslPredicateExecutor`-backed repository would normally fetch from a database.

Next, `buildPredicate(null, "SHIPPED")` runs: `minTotal` is `null` so its branch is skipped, and `status` is `"SHIPPED"` so `predicate` becomes `identity.and(order.statusEq("SHIPPED"))` — effectively just the status check.

Then `repo.findAll(onlyStatus)` streams the three orders through that predicate. Order 1 (`SHIPPED`) and order 2 (`SHIPPED`) pass; order 3 (`PENDING`) is filtered out. The result `[Order{id=1,...}, Order{id=2,...}]` is printed.

The same sequence repeats for `buildPredicate(100.0, "SHIPPED")`: both branches now apply, so `predicate` requires `total > 100 AND status == "SHIPPED"`. Only order 2 (`total=150`, `SHIPPED`) satisfies both; orders 1 (`total=50`) and 3 (`PENDING`) are excluded. `[Order{id=2,...}]` is printed.

```
buildPredicate(null, "SHIPPED")     buildPredicate(100.0, "SHIPPED")
        |                                    |
   status==SHIPPED                total>100 AND status==SHIPPED
        |                                    |
   [order1, order2]                       [order2]
```

In a real Spring Data setup, this same flow happens across layers: an HTTP request like `GET /orders?status=SHIPPED` arrives at a controller, which builds a `BooleanBuilder` conditionally (exactly like `buildPredicate` here), passes the resulting `Predicate` to `orderRepository.findAll(predicate)`, and Querydsl's JPA integration translates it into a `WHERE status = ?` SQL clause executed against the database — the controller never sees SQL or JPQL text, only typed Java.

## 7. Gotchas & takeaways

> Gotcha: Querydsl's `QuerydslPredicateExecutor` requires the `querydsl-apt` annotation processor to be configured in the build (Maven/Gradle) so `Q*` classes get generated at compile time — if the generated sources are missing, every reference to `QOrder` fails to compile, and the fix is a build configuration change, not a code change.

- Querydsl gives compile-time-checked dynamic queries, positioned between raw JPQL strings and the Criteria API in both safety and verbosity.
- `QOrder` (and every other `Q*` type) is generated from your entity by an annotation processor — it does not exist until the project builds at least once.
- Build predicates incrementally from an identity/true base so optional search-form fields don't force `null` checks scattered through query logic.
- A repository only gains `findAll(Predicate)` by also extending `QuerydslPredicateExecutor<T>` — extending `JpaRepository` alone is not enough.
