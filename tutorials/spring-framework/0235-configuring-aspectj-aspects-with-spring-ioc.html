Configuring AspectJ aspects with Spring IoC

← Back to checklist

1. What it is

When you use full AspectJ (compile-time or load-time weaving) rather than proxy-based Spring AOP, AspectJ manages the aspect lifecycle itself — not Spring. Aspects are singleton instances maintained by the AspectJ runtime. Configuring AspectJ aspects with Spring IoC means telling Spring to inject dependencies (beans, properties, configuration) into those AspectJ-managed singletons, bridging the two worlds.

// Spring factory method that hands the AspectJ singleton to the Spring context:
@Bean
public SecurityAspect securityAspect() {
    return SecurityAspect.aspectOf();   // ← AspectJ singleton accessor
}

aspectOf() is generated by the AspectJ compiler/weaver. It returns the single instance that AspectJ created — Spring can then configure its fields and wired beans as a normal bean.

2. Why & when

Use this pattern when:

  1. LTW aspects need Spring beans injected — e.g., an aspect that logs to a Spring-managed AuditRepository, or checks a Spring-configured feature flag.
  2. Compile-time woven aspects need configuration — the aspect's pointcut logic might be hard-coded but its dependencies (repositories, loggers, caches) come from Spring.
  3. Mixing proxy AOP and full AspectJ — some cross-cuts use Spring's proxy AOP; others (needing private/static/self-invocation coverage) use full AspectJ. Both sets of aspects should receive Spring-managed dependencies.

This is NOT needed for @Aspect @Component aspects used with @EnableAspectJAutoProxy — those are Spring-managed beans from the start.

3. Core concept

The key is Singleton.aspectOf() — a static method generated by the AspectJ compiler for every aspect declaration. It returns the one-and-only instance of the aspect that AspectJ has already created.

AspectJ runtime                    Spring IoC
──────────────                    ──────────
aspect SecurityAspect { }  ──creates──▶ singleton instance
                                        │
                          aspectOf() ◀──┤
                                        │
Spring @Bean factory method             │
  return SecurityAspect.aspectOf() ─▶   │
                                        ▼
                            Spring injects @Autowired fields
                            into the existing AspectJ instance

Spring does NOT create a new instance — it adopts the existing one and performs dependency injection on it.

For per-this / per-target instantiation models, use aspectOf(targetObject) instead of aspectOf().

5. Runnable example

Scenario: a SecurityAspect that checks roles — it needs a Spring-managed UserRepository to look up permissions. Full AspectJ (LTW) handles the weaving; Spring IoC handles the wiring.

Level 1 — Basic

SecurityAspect as an AspectJ singleton, wired via a Spring @Bean factory method.

// AspectJIocDemo.java
import org.springframework.context.annotation.*;
import org.springframework.stereotype.*;
import org.springframework.beans.factory.annotation.*;

@Configuration
@EnableLoadTimeWeaving(aspectjWeaving = EnableLoadTimeWeaving.AspectJWeaving.ENABLED)
@ComponentScan
public class AspectJIocDemo {
    // Hand the AspectJ-managed singleton to Spring for dependency injection
    @Bean
    public SecurityAspect securityAspect() {
        return SecurityAspect.aspectOf();
    }

    public static void main(String[] args) {
        var ctx = new AnnotationConfigApplicationContext(AspectJIocDemo.class);
        PaymentService svc = ctx.getBean(PaymentService.class);
        svc.charge("USER-1", 99.0);
        ctx.close();
    }
}

@Component
class PaymentService {
    public void charge(String userId, double amount) {
        System.out.println("Charging $" + amount + " for user " + userId);
    }
}

@Component
class UserRepository {
    public boolean hasRole(String userId, String role) {
        // Simulated — USER-1 is ADMIN, others are not
        return "USER-1".equals(userId) && "PAYMENT".equals(role);
    }
}

AspectJ aspect file (SecurityAspect.java — compiled/woven by AspectJ, not proxy-based):

import org.aspectj.lang.annotation.*;
import org.aspectj.lang.*;
import org.springframework.beans.factory.annotation.*;

@Aspect
public class SecurityAspect {
    @Autowired                               // injected by Spring after aspectOf() registration
    private UserRepository userRepository;

    @Before("execution(* PaymentService.charge(String, double)) && args(userId, amount)")
    public void checkRole(JoinPoint jp, String userId, double amount) {
        if (!userRepository.hasRole(userId, "PAYMENT")) {
            throw new SecurityException("User " + userId + " lacks PAYMENT role");
        }
        System.out.println("[SECURITY] " + userId + " cleared for charge of $" + amount);
    }
}

META-INF/aop.xml:

<!DOCTYPE aspectj PUBLIC "-//AspectJ//DTD//EN"
  "https://www.eclipse.org/aspectj/dtd/aspectj.dtd">
<aspectj>
  <weaver><include within="PaymentService"/></weaver>
  <aspects><aspect name="SecurityAspect"/></aspects>
</aspectj>

How to run: java -javaagent:aspectjweaver.jar -cp spring-context.jar:spring-aop.jar:aspectjweaver.jar:spring-instrument.jar:. AspectJIocDemo.java

securityAspect() calls SecurityAspect.aspectOf() — the AspectJ runtime's static accessor. Spring then @Autowired-injects UserRepository into the returned instance. The woven PaymentService.charge() calls the aspect's checkRole(), which uses the Spring-injected repository.

Level 2 — Intermediate

Add @Value property injection and @PostConstruct lifecycle callback — showing full Spring lifecycle on the AspectJ singleton.

// AspectJIocDemo.java
import org.springframework.context.annotation.*;
import org.springframework.stereotype.*;
import org.springframework.beans.factory.annotation.*;

@Configuration
@EnableLoadTimeWeaving(aspectjWeaving = EnableLoadTimeWeaving.AspectJWeaving.ENABLED)
@ComponentScan
@PropertySource("classpath:security.properties")
public class AspectJIocDemo {
    @Bean
    public SecurityAspect securityAspect() {
        return SecurityAspect.aspectOf();
    }

    public static void main(String[] args) {
        var ctx = new AnnotationConfigApplicationContext(AspectJIocDemo.class);
        ctx.getBean(PaymentService.class).charge("USER-1", 150.0);
        ctx.getBean(PaymentService.class).charge("USER-2", 50.0);
        ctx.close();
    }
}

@Component
class PaymentService {
    public void charge(String userId, double amount) {
        System.out.println("Charging $" + amount + " for " + userId);
    }
}

@Component
class UserRepository {
    public boolean hasRole(String userId, String role) {
        return "USER-1".equals(userId);
    }
}

Aspect (now with @Value and @PostConstruct):

import org.aspectj.lang.annotation.*;
import org.aspectj.lang.*;
import org.springframework.beans.factory.annotation.*;
import jakarta.annotation.*;

@Aspect
public class SecurityAspect {
    @Autowired
    private UserRepository userRepository;

    @Value("${security.max.charge:500}")        // from security.properties or default
    private double maxAllowedCharge;

    @PostConstruct
    public void init() {
        System.out.println("[SecurityAspect] initialized, maxCharge=" + maxAllowedCharge);
    }

    @Before("execution(* PaymentService.charge(String, double)) && args(userId, amount)")
    public void checkRole(JoinPoint jp, String userId, double amount) {
        if (!userRepository.hasRole(userId, "PAYMENT")) {
            throw new SecurityException("No PAYMENT role: " + userId);
        }
        if (amount > maxAllowedCharge) {
            throw new SecurityException("Amount $" + amount + " exceeds limit $" + maxAllowedCharge);
        }
        System.out.println("[SECURITY] " + userId + " cleared for $" + amount);
    }
}

security.properties: security.max.charge=200

How to run: same as Level 1 + security.properties on classpath

@PostConstruct fires after Spring injects both userRepository and maxAllowedCharge. The aspect uses both in its @Before advice. This proves the AspectJ singleton participates in the full Spring lifecycle.

Level 3 — Advanced

Per-target instantiation model — a separate aspect instance per proxied target, each wired by Spring via aspectOf(target).

// AspectJIocDemo.java
import org.springframework.context.annotation.*;
import org.springframework.stereotype.*;
import org.springframework.beans.factory.annotation.*;

@Configuration
@EnableLoadTimeWeaving(aspectjWeaving = EnableLoadTimeWeaving.AspectJWeaving.ENABLED)
@ComponentScan
public class AspectJIocDemo {
    // per-target: aspectOf(target) instead of aspectOf()
    @Bean
    @Scope("prototype")
    public TimingAspect timingAspect() {
        // For per-target, Spring cannot call aspectOf() without target object.
        // We use the singleton aspectOf() here; per-target is handled in the aspect declaration.
        return TimingAspect.aspectOf();
    }

    public static void main(String[] args) {
        var ctx = new AnnotationConfigApplicationContext(AspectJIocDemo.class);
        PaymentService pay = ctx.getBean(PaymentService.class);
        pay.charge("USER-1", 99.0);
        pay.charge("USER-2", 20.0);

        // Introspect: get the aspect and its injected MetricsCollector
        TimingAspect aspect = ctx.getBean(TimingAspect.class);
        System.out.println("Aspect class: " + aspect.getClass().getName());
        ctx.close();
    }
}

@Component
class PaymentService {
    public void charge(String userId, double amount) {
        System.out.println("Charging $" + amount + " for " + userId);
    }
}

@Component
class MetricsCollector {
    private final java.util.List<Long> timings = new java.util.ArrayList<>();

    public void record(long nanos) {
        timings.add(nanos);
        System.out.printf("[METRICS] recorded %d µs (total %d samples)%n",
            nanos / 1_000, timings.size());
    }
}

Aspect:

import org.aspectj.lang.annotation.*;
import org.aspectj.lang.*;
import org.springframework.beans.factory.annotation.*;

@Aspect
public class TimingAspect {
    @Autowired
    private MetricsCollector metrics;   // Spring-injected

    @Around("execution(* PaymentService.charge(..))")
    public Object time(ProceedingJoinPoint pjp) throws Throwable {
        long start = System.nanoTime();
        Object result = pjp.proceed();
        metrics.record(System.nanoTime() - start);
        return result;
    }
}

How to run: same classpath

TimingAspect.aspectOf() fetches the AspectJ singleton. Spring then injects MetricsCollector into it. The aspect records timings for every charge() call. ctx.getBean(TimingAspect.class) fetches the same instance Spring wired — demonstrating that Spring and AspectJ share exactly one object reference.

6. Walkthrough

Context startup sequence:

  1. -javaagent:aspectjweaver.jar starts; reads META-INF/aop.xml.
  2. Spring starts AnnotationConfigApplicationContext. Scans @Component beans: PaymentService, UserRepository, MetricsCollector.
  3. Spring calls the @Bean factory method securityAspect():
  • Executes SecurityAspect.aspectOf() — returns the AspectJ-created singleton.
  • Spring registers this instance under the bean name "securityAspect".
  1. Spring's dependency-injection phase: finds @Autowired UserRepository on the instance → injects userRepository bean.
  2. @PostConstruct init() fires → prints initialization message.

Per-call flow for paymentService.charge("USER-1", 99.0) (Level 1, LTW):

ctx.getBean(PaymentService.class)
  → returns PaymentService (woven at class load time, no proxy wrapper)

paymentService.charge("USER-1", 99.0)
  → woven bytecode runs:
      SecurityAspect.aspectOf().checkRole(thisJoinPoint, "USER-1", 99.0)
        → userRepository.hasRole("USER-1", "PAYMENT")  [Spring bean, returns true]
        → prints "[SECURITY] USER-1 cleared for charge of $99.0"
      → PaymentService.charge body:
          prints "Charging $99.0 for user USER-1"
  ← returns void

State transitions:

Request:  charge("USER-1", 99.0)
  ↓ woven aspect entry
  userId="USER-1", amount=99.0
  ↓ userRepository.hasRole() → true
  ↓ security check passes
  ↓ PaymentService body executes
  ↓ charge complete
Response: void (side effect: charge applied)

For Level 2, @Value("${security.max.charge}") adds a second guard — amount > maxAllowedCharge — derived from security.properties. If charge("USER-1", 999.0) were called with maxAllowedCharge=200, the aspect throws before the body executes.

7. Gotchas & takeaways

SecurityAspect.aspectOf() throws NoAspectBoundException if the aspect has not been woven yet — i.e., if the JVM agent is missing or aop.xml does not include the aspect. The @Bean factory method will fail at context startup with this exception, which is actually helpful: it surfaces the misconfiguration immediately rather than silently producing an unwired aspect.
Calling aspectOf() before the AspectJ weaver has loaded the aspect class returns null in some versions. Always ensure @EnableLoadTimeWeaving is in effect before the @Bean factory runs. Spring's ordered context startup normally guarantees this, but programmatic context creation can break the order.
Spring does NOT create a second instance. If you declare new SecurityAspect() anywhere in your code, you get a second Java object that AspectJ will NOT use for weaving — weaving goes through the singleton created by AspectJ, not any hand-created instance.
  • aspectOf() is the bridge: AspectJ owns the lifecycle, Spring owns the wiring.
  • All Spring lifecycle features work on the adopted instance: @Autowired, @Value, @PostConstruct, @PreDestroy, @Scope.
  • For per-this or per-target aspects, use Aspect.aspectOf(targetObject) — one instance per object, each must be separately adopted and wired.
  • Combine @EnableLoadTimeWeaving with @EnableSpringConfigured (spring-aspects) to enable @Configurable and AnnotationBeanConfigurerAspect in the same context.
  • Inspect wired aspects via ctx.getBean(SecurityAspect.class) — useful for verifying injection in integration tests.
