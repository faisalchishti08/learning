"Vault backends (KV, database, PKI, AWS, etc.)"

1. What it is

A Vault "backend" (more precisely, a secrets engine) is a pluggable module that determines what kind of secret Vault produces at a given path and how — the KV (key-value) backend stores and returns static, manually-written secrets exactly as put in; the database backend generates brand-new, short-lived database credentials on demand; the PKI backend issues freshly-signed TLS certificates on demand; the AWS backend generates temporary AWS IAM credentials on demand — and Spring Cloud Vault configures each backend type independently, since each returns a differently-shaped secret.

spring.cloud.vault.kv.enabled=true
spring.cloud.vault.kv.backend=secret

spring.cloud.vault.database.enabled=true
spring.cloud.vault.database.role=order-service-role

spring.cloud.vault.pki.enabled=true
spring.cloud.vault.pki.role=order-service-cert

2. Why & when

A static secret (a KV entry) and a dynamic secret (a freshly-generated database credential) solve genuinely different problems, and conflating them leads to weaker security than necessary: a static API key, once leaked, remains valid until someone manually rotates it, while a dynamic database credential generated by Vault's database backend is unique to the requesting application, has a built-in expiration (a lease, covered in a later card), and if leaked, naturally becomes useless once its lease expires — no manual rotation required. Choosing the right backend for a given kind of secret is a deliberate security decision: static credentials that genuinely can't be dynamically generated (a third-party SaaS API key, for instance) belong in KV; anything Vault *can* generate dynamically (database credentials, TLS certificates, cloud provider IAM credentials) is meaningfully more secure sourced from its dedicated dynamic backend instead.

Reach for a specific backend when:

  • Storing simple, static secrets that have no natural "generate a new one on demand" mechanism — a third-party API key, a webhook signing secret — belongs in the KV backend, Vault's simplest and most general-purpose secrets engine.
  • Database access is needed — the database backend can generate a brand-new, uniquely-scoped, automatically-expiring database user for each requesting application instance, rather than every instance sharing one long-lived, manually-rotated credential.
  • Internal TLS certificates are needed for service-to-service communication — the PKI backend can issue short-lived certificates signed by a Vault-managed CA on demand, avoiding the operational burden of manually managing a certificate lifecycle.
  • Cloud provider access (AWS, GCP, Azure) is needed — the corresponding cloud backend can generate temporary, scoped cloud IAM credentials, avoiding long-lived cloud credentials embedded in application configuration entirely.

3. Core concept

 KV backend:        write once, read the SAME value every time   (static: an API key)
 database backend:  EACH read generates a BRAND NEW credential   (dynamic: a fresh DB user/password pair)
 PKI backend:        EACH read generates a BRAND NEW certificate  (dynamic: a fresh signed cert)
 AWS backend:        EACH read generates BRAND NEW IAM credentials (dynamic: temporary AWS access keys)

 static secrets:  same value returned to every requester, indefinitely, until manually changed
 dynamic secrets: a UNIQUE value returned to EACH requester, automatically expiring via a lease

The backend type is a property of the *path* being read from — the same Vault server can serve KV secrets at one path and dynamic database credentials at another, simultaneously, for different applications or purposes.

5. Runnable example

The scenario: model three backend types (KV, database, PKI) as distinct read behaviors — one returning identical values across repeated reads, the other two generating fresh, unique values each time. Start with the KV backend's static-read behavior, then add the database backend's dynamic-generation behavior alongside it, then add lease-tracking metadata (expiry) to the dynamic backends, showing why dynamic secrets need lifecycle management that static ones don't.

Level 1 — Basic

The KV backend: a fixed store, read returns the same value every time.

import java.util.*;

public class VaultBackendsLevel1 {
    static class KvBackend {
        Map<String, String> store = new HashMap<>();
        void write(String path, String value) { store.put(path, value); }
        String read(String path) { return store.get(path); } // ALWAYS returns the SAME stored value
    }

    public static void main(String[] args) {
        KvBackend kv = new KvBackend();
        kv.write("secret/order-service/api-key", "sk-abc123");

        System.out.println("read 1: " + kv.read("secret/order-service/api-key"));
        System.out.println("read 2: " + kv.read("secret/order-service/api-key"));
        System.out.println("identical every time: " + kv.read("secret/order-service/api-key").equals(kv.read("secret/order-service/api-key")));
    }
}

How to run: java VaultBackendsLevel1.java

Two separate reads of the same KV path return byte-for-byte identical values — this is precisely what "static secret" means, and it's exactly the right behavior for something like a fixed third-party API key.

Level 2 — Intermediate

Add a database backend generating a genuinely new, unique credential on every single read, contrasted directly against the KV backend's unchanging behavior.

import java.util.*;

public class VaultBackendsLevel2 {
    static class KvBackend {
        Map<String, String> store = new HashMap<>();
        void write(String path, String value) { store.put(path, value); }
        String read(String path) { return store.get(path); }
    }

    record DbCredential(String username, String password) {}

    static class DatabaseBackend {
        int counter = 0;
        DbCredential generate(String role) {
            counter++;
            // a REAL Vault database backend creates an actual new DB user with a random password server-side
            return new DbCredential("v-" + role + "-" + counter, "randompass-" + UUID.randomUUID().toString().substring(0, 6));
        }
    }

    public static void main(String[] args) {
        KvBackend kv = new KvBackend();
        kv.write("secret/order-service/api-key", "sk-abc123");
        DatabaseBackend db = new DatabaseBackend();

        System.out.println("KV read 1: " + kv.read("secret/order-service/api-key"));
        System.out.println("KV read 2: " + kv.read("secret/order-service/api-key") + " (SAME as read 1)");

        DbCredential cred1 = db.generate("order-service-role");
        DbCredential cred2 = db.generate("order-service-role");
        System.out.println("DB credential 1: " + cred1.username());
        System.out.println("DB credential 2: " + cred2.username() + " (DIFFERENT username from credential 1)");
    }
}

How to run: java VaultBackendsLevel2.java

The two KV reads produce the identical string both times, while the two database backend calls produce two entirely distinct usernames (v-order-service-role-1 and v-order-service-role-2) — this is the fundamental static-versus-dynamic distinction made concrete: every application instance requesting a database credential from Vault gets its own unique credential, rather than all instances sharing one.

Level 3 — Advanced

Add lease metadata (a duration after which a dynamically-generated credential expires) to the database backend, and demonstrate checking whether a previously-issued credential has expired — the lifecycle concern dynamic secrets require that static KV secrets simply don't have.

import java.util.*;

public class VaultBackendsLevel3 {
    record DbCredential(String username, String password, long issuedAtMs, long leaseDurationMs) {
        boolean isExpired(long nowMs) { return (nowMs - issuedAtMs) > leaseDurationMs; }
    }

    static class DatabaseBackend {
        int counter = 0;
        DbCredential generate(String role, long leaseDurationMs) {
            counter++;
            return new DbCredential(
                    "v-" + role + "-" + counter,
                    "randompass-" + UUID.randomUUID().toString().substring(0, 6),
                    System.currentTimeMillis(),
                    leaseDurationMs
            );
        }
    }

    public static void main(String[] args) throws InterruptedException {
        DatabaseBackend db = new DatabaseBackend();

        DbCredential shortLived = db.generate("order-service-role", 50); // 50ms lease -- deliberately short for this demo
        System.out.println("issued credential: " + shortLived.username() + ", expired right away? " + shortLived.isExpired(System.currentTimeMillis()));

        Thread.sleep(100); // simulate time passing beyond the lease duration

        System.out.println("after waiting past the lease: expired? " + shortLived.isExpired(System.currentTimeMillis()));

        if (shortLived.isExpired(System.currentTimeMillis())) {
            // a real application must request a FRESH credential once the old one's lease expires -- covered in the next card
            DbCredential renewed = db.generate("order-service-role", 50);
            System.out.println("requested a FRESH credential after expiry: " + renewed.username());
        }
    }
}

How to run: java VaultBackendsLevel3.java

shortLived.isExpired(...) returns false immediately after issuance (no time has passed yet), but returns true after the Thread.sleep(100) call, since 100ms exceeds the credential's 50ms lease duration — this expiry check, entirely absent from the KV backend model, is exactly what makes dynamic secrets require ongoing lifecycle management (checking expiry, requesting renewal or fresh credentials), a concern the next card covers in depth.

6. Walkthrough

Trace the expiry check sequence in Level 3.

  1. db.generate("order-service-role", 50) creates a DbCredential with issuedAtMs set to the current time and leaseDurationMs = 50.
  2. shortLived.isExpired(System.currentTimeMillis()) is called immediately — nowMs - issuedAtMs is a very small number (well under a millisecond has typically elapsed just from the method calls themselves), so (nowMs - issuedAtMs) > 50 evaluates false; the credential is correctly reported as not yet expired.
  3. Thread.sleep(100) pauses execution for 100 milliseconds, simulating real time passing while the application holds and uses this credential.
  4. shortLived.isExpired(System.currentTimeMillis()) is called again — now nowMs - issuedAtMs is approximately 100 (or slightly more, accounting for the method calls' own overhead), and 100 > 50 evaluates true — the credential is correctly reported as expired.
  5. The if (shortLived.isExpired(...)) check is true, so the code calls db.generate(...) again, producing a brand-new DbCredential with a fresh username (counter has incremented again) and a fresh issuedAtMs — this second, renewed credential is unrelated to the first beyond sharing the same role; it is a completely independent grant with its own new lease starting from this moment.
t=0ms:    generate() -> credential issued, leaseDurationMs=50
t=0ms:    isExpired(now)   -> now - issuedAt ≈ 0    -> 0 > 50?   false -> NOT expired
t=100ms:  isExpired(now)   -> now - issuedAt ≈ 100  -> 100 > 50? true  -> EXPIRED
t=100ms:  generate() again -> a FRESH credential, unrelated to the expired one, new lease starts now

7. Gotchas & takeaways

Gotcha: a dynamic secret's lease duration is a hard boundary from the backing system's own perspective — for a real Vault database backend, once a generated database user's lease expires (and isn't renewed), Vault itself revokes that database user, meaning any connection still actively using those now-revoked credentials will start failing authentication, not just "the credential looks stale." Applications using dynamic database credentials need to actually renew the lease (the next card) or re-request a fresh credential before expiry, not merely check an expiry timestamp informationally.
  • The KV backend is the right choice for genuinely static secrets with no dynamic-generation equivalent; database, PKI, and cloud-provider backends are the right choice whenever the secret type Vault needs to produce *can* be dynamically and uniquely generated per requester, since dynamic secrets are meaningfully more secure by design.
  • Every dynamically-generated secret comes with a lease — a bounded validity period after which the underlying system (a database, a certificate authority, a cloud provider) invalidates that specific secret, requiring either renewal or replacement before it's actually used past that point.
  • Static and dynamic secrets can coexist at different paths on the same Vault server — an application commonly uses KV for its truly static, unavoidably-fixed secrets (a third-party API key) alongside the database backend for its actual database credentials, simultaneously.
  • Understanding which backend a given secret comes from determines what operational behavior an application needs around it: a KV secret needs no lifecycle management beyond occasional manual rotation, while any dynamically-generated secret needs active lease renewal or periodic re-issuance to remain usable.
