"Identifier generation"

← Back to checklist

1. What it is

Identifier generation is how an entity's primary key gets its value — for JPA specifically, @GeneratedValue (used throughout this section) delegates to one of several strategies (IDENTITY, letting the database auto-increment; SEQUENCE, using a database sequence object; AUTO, letting the JPA provider choose; TABLE, a portable but slower table-based counter), while Spring Data's own Persistable<ID> interface addresses a related but distinct problem: correctly telling save() whether a given entity instance is new (needs an INSERT) or already exists (needs an UPDATE), which matters most for manually-assigned (rather than database-generated) identifiers.

@Entity
public class Order implements Persistable<UUID> {
    @Id private UUID id = UUID.randomUUID(); // assigned in Java, not by the database
    @Transient private boolean isNew = true;

    @Override public UUID getId() { return id; }
    @Override public boolean isNew() { return isNew; }
}

2. Why & when

The earlier CrudRepository card's gotcha flagged this exact issue: save()'s insert-versus-update decision, for JPA, is normally based on whether the ID is null. That works perfectly for @GeneratedValue-populated IDs (always null before the first save), but breaks down for manually-assigned IDs (a UUID generated in application code, a natural business key) — since such an ID is *never* null, even for a genuinely new entity, save() can't use nullness to tell new from existing, and might incorrectly attempt an UPDATE on an entity that was never actually inserted (which silently does nothing, since no matching row exists).

Understanding identifier generation strategies, and reaching for Persistable<ID> specifically, matters when:

  • You're choosing between database-generated IDs (IDENTITY/SEQUENCE, simple, works with null-based new/existing detection) and application-assigned IDs (UUIDs generated in Java, natural keys) for a new entity design.
  • You're using manually-assigned IDs and need save() to correctly distinguish new entities from existing ones — Persistable<ID>'s explicit isNew() method is the fix.
  • You're debugging a "save() silently didn't insert anything" bug — this is very often exactly the null-ID-detection problem described above, striking whenever an ID is assigned before the first save rather than generated by the database.

3. Core concept

 @GeneratedValue strategies (database-generated IDs):
   IDENTITY   -- database auto-increment column (MySQL AUTO_INCREMENT, this
                 section's examples' default) -- simple, ID known only AFTER insert
   SEQUENCE   -- a database sequence object, pre-fetched -- allows batching,
                 ID can be known BEFORE the insert
   TABLE      -- a portable, table-based counter -- most portable, slowest
   AUTO       -- JPA provider picks a strategy automatically

 save()'s new-vs-existing detection, DEFAULT (no Persistable):
   id == null  -->  treat as NEW  -->  INSERT
   id != null  -->  treat as EXISTING  -->  UPDATE (merge)
        |
        PROBLEM: manually-assigned IDs are NEVER null, even when genuinely new

 Persistable<ID> -- EXPLICIT override of that detection:
   interface Persistable<ID> {
       ID getId();
       boolean isNew();      -- YOU decide, not inferred from nullness
   }
        |
        v
   save() consults isNew() directly when the entity implements Persistable,
   bypassing the null-check heuristic entirely

Persistable<ID> is the explicit escape hatch whenever an ID's nullness can't be trusted to indicate whether an entity is new.

5. Runnable example

The scenario: an Order entity, evolving from demonstrating the null-check heuristic's failure with a manually-assigned UUID, to Persistable<UUID> fixing it, to a realistic pattern using @PostLoad/constructor logic to manage the isNew flag correctly across the entity's actual lifecycle.

Level 1 — Basic

Show the default behavior breaking for a manually-assigned ID: save() on a genuinely new entity silently performs a no-op UPDATE instead of an INSERT.

import jakarta.persistence.Entity;
import jakarta.persistence.Id;
import org.springframework.boot.SpringApplication;
import org.springframework.boot.autoconfigure.SpringBootApplication;
import org.springframework.context.ConfigurableApplicationContext;
import org.springframework.data.jpa.repository.JpaRepository;

import java.util.UUID;

@SpringBootApplication
public class IdGenerationLevel1 {

    @Entity
    public static class Order {
        @Id
        private UUID id = UUID.randomUUID(); // assigned in JAVA, never null
        private double total;
        protected Order() {}
        public Order(double total) { this.total = total; }
        public UUID getId() { return id; }
    }

    public interface OrderRepository extends JpaRepository<Order, UUID> {}

    public static void main(String[] args) {
        ConfigurableApplicationContext ctx = SpringApplication.run(IdGenerationLevel1.class,
            "--spring.datasource.url=jdbc:h2:mem:idgen1",
            "--spring.jpa.hibernate.ddl-auto=create-drop");

        OrderRepository repo = ctx.getBean(OrderRepository.class);
        Order newOrder = new Order(100.0);
        System.out.println("id before save (already non-null): " + newOrder.getId());

        Order saved = repo.save(newOrder); // save() sees a NON-NULL id -- attempts an UPDATE, not an INSERT!

        long countInDatabase = repo.count();
        System.out.println("rows actually in the database after save() = " + countInDatabase);

        if (countInDatabase != 0)
            throw new AssertionError("Demonstrating the BUG: expected save() to have inserted NOTHING due to the flawed null-check heuristic");
        System.out.println("Confirmed the bug: manually-assigned id caused save() to silently no-op instead of INSERT -- PASS (bug demonstrated)");
        ctx.close();
    }
}

How to run: put spring-boot-starter-data-jpa and com.h2database:h2 on the classpath, then java IdGenerationLevel1.java on JDK 17+.

newOrder.getId() is already a real, non-null UUID before save() is ever called — Hibernate's default new-vs-existing detection sees a non-null ID and assumes the entity already exists, attempting an UPDATE rather than an INSERT. Since no row with that UUID exists yet, the UPDATE matches zero rows and silently does nothing — repo.count() afterward confirms the database genuinely has zero rows, demonstrating the exact bug this card exists to explain and fix.

Level 2 — Intermediate

Fix the bug from Level 1 by implementing Persistable<UUID>, explicitly telling save() whether the entity is new.

import jakarta.persistence.Entity;
import jakarta.persistence.Id;
import jakarta.persistence.Transient;
import org.springframework.boot.SpringApplication;
import org.springframework.boot.autoconfigure.SpringBootApplication;
import org.springframework.context.ConfigurableApplicationContext;
import org.springframework.data.domain.Persistable;
import org.springframework.data.jpa.repository.JpaRepository;

import java.util.UUID;

@SpringBootApplication
public class IdGenerationLevel2 {

    @Entity
    public static class Order implements Persistable<UUID> {
        @Id
        private UUID id = UUID.randomUUID();
        private double total;

        @Transient
        private boolean isNew = true; // explicit flag, defaults to true for freshly-constructed instances

        protected Order() {}
        public Order(double total) { this.total = total; }

        @Override public UUID getId() { return id; }
        @Override public boolean isNew() { return isNew; }

        public void markPersisted() { this.isNew = false; }
    }

    public interface OrderRepository extends JpaRepository<Order, UUID> {}

    public static void main(String[] args) {
        ConfigurableApplicationContext ctx = SpringApplication.run(IdGenerationLevel2.class,
            "--spring.datasource.url=jdbc:h2:mem:idgen2",
            "--spring.jpa.hibernate.ddl-auto=create-drop");

        OrderRepository repo = ctx.getBean(OrderRepository.class);
        Order newOrder = new Order(100.0);

        Order saved = repo.save(newOrder); // save() now consults isNew() -- correctly INSERTs

        long countInDatabase = repo.count();
        System.out.println("rows in the database after save() = " + countInDatabase);

        if (countInDatabase != 1)
            throw new AssertionError("Expected Persistable.isNew()=true to correctly cause an INSERT");
        System.out.println("Persistable<UUID> fixed the bug -- save() correctly performed an INSERT -- PASS");
        ctx.close();
    }
}

How to run: same classpath as Level 1, java IdGenerationLevel2.java.

Order implements Persistable<UUID>, with isNew() returning an explicit boolean field defaulting to true. Because JPA repositories check for Persistable and, when present, consult isNew() directly rather than checking the ID for nullness, save(newOrder) correctly recognizes this as a new entity and performs a genuine INSERT — repo.count() now correctly reports 1, fixing the exact bug demonstrated in Level 1.

Level 3 — Advanced

Use @PostLoad to correctly flip isNew to false for entities loaded back from the database — ensuring a *reloaded* entity (which genuinely already exists) doesn't incorrectly report isNew() == true and trigger an unwanted INSERT attempt on its next save.

import jakarta.persistence.Entity;
import jakarta.persistence.Id;
import jakarta.persistence.PostLoad;
import jakarta.persistence.PostPersist;
import jakarta.persistence.Transient;
import org.springframework.boot.SpringApplication;
import org.springframework.boot.autoconfigure.SpringBootApplication;
import org.springframework.context.ConfigurableApplicationContext;
import org.springframework.data.domain.Persistable;
import org.springframework.data.jpa.repository.JpaRepository;

import java.util.UUID;

@SpringBootApplication
public class IdGenerationLevel3 {

    @Entity
    public static class Order implements Persistable<UUID> {
        @Id
        private UUID id = UUID.randomUUID();
        private double total;
        private String status;

        @Transient
        private boolean isNew = true;

        protected Order() {}
        public Order(double total) { this.total = total; this.status = "pending"; }

        @Override public UUID getId() { return id; }
        @Override public boolean isNew() { return isNew; }
        public String getStatus() { return status; }
        public void setStatus(String status) { this.status = status; }

        // After the entity is genuinely inserted, or loaded back from the database,
        // it's definitely no longer "new" from this point forward.
        @PostPersist
        @PostLoad
        void markNotNew() { this.isNew = false; }
    }

    public interface OrderRepository extends JpaRepository<Order, UUID> {}

    public static void main(String[] args) {
        ConfigurableApplicationContext ctx = SpringApplication.run(IdGenerationLevel3.class,
            "--spring.datasource.url=jdbc:h2:mem:idgen3",
            "--spring.jpa.hibernate.ddl-auto=create-drop");

        OrderRepository repo = ctx.getBean(OrderRepository.class);

        // 1. Genuinely new entity -- isNew() = true, save() should INSERT.
        Order created = repo.save(new Order(100.0));
        System.out.println("after initial save, isNew() = " + created.isNew());

        // 2. Reload it from the database -- @PostLoad marks it isNew=false.
        Order reloaded = repo.findById(created.getId()).orElseThrow();
        System.out.println("after reload, isNew() = " + reloaded.isNew());

        // 3. Modify and save again -- should be an UPDATE, since isNew() is correctly false now.
        reloaded.setStatus("shipped");
        repo.save(reloaded);

        long countInDatabase = repo.count();
        Order finalState = repo.findById(created.getId()).orElseThrow();

        System.out.println("total rows (should still be exactly 1, NOT 2) = " + countInDatabase);
        System.out.println("final status = " + finalState.getStatus());

        if (created.isNew()) throw new AssertionError("Expected isNew() to be false immediately after the initial INSERT");
        if (reloaded.isNew()) throw new AssertionError("Expected a reloaded entity's isNew() to be false");
        if (countInDatabase != 1) throw new AssertionError("Expected the update to affect the SAME row, not create a second one");
        if (!"shipped".equals(finalState.getStatus())) throw new AssertionError("Expected the status update to have persisted");

        System.out.println("@PostPersist/@PostLoad correctly kept isNew() accurate across the entity's full lifecycle -- PASS");
        ctx.close();
    }
}

How to run: same classpath as Level 1 and 2, java IdGenerationLevel3.java.

@PostPersist fires immediately after the entity is actually inserted, and @PostLoad fires whenever an entity is loaded back from the database — both flip isNew to false, since by either point the entity genuinely, verifiably exists in the database. This ensures isNew() stays accurate across the entity's full lifecycle: true only for a genuinely fresh, never-yet-persisted instance; false for anything that's been inserted or loaded — correctly routing every subsequent save() call to the right operation (INSERT only once, UPDATE for everything after).

6. Walkthrough

Trace Level 3's three-step sequence.

  1. repo.save(new Order(100.0)): a brand-new Order is constructed with isNew = true (the field's default value, set at declaration). save() checks Persistable.isNew(), sees true, and performs a genuine INSERT.
  2. @PostPersist fires: immediately after that INSERT succeeds, JPA's lifecycle callback mechanism invokes markNotNew(), setting isNew = false on the *same in-memory instance* that was just saved — this is why created.isNew() correctly reports false right after the save returns.
  3. repo.findById(created.getId()): a completely separate database read occurs, constructing (via the no-arg constructor, then reflective field population, per the previous card's object-mapping model) a *new* Order instance representing the same row.
  4. @PostLoad fires on this freshly-loaded instance: immediately after Hibernate populates its fields from the database row, markNotNew() runs again, setting this new instance's isNew to false too — critical, because this instance's isNew field would otherwise default back to true (its declared default), incorrectly suggesting it's a new entity even though it was just loaded from an existing row.
  5. reloaded.setStatus("shipped") then repo.save(reloaded): because reloaded.isNew() correctly reports false (thanks to step 4), save() performs a genuine UPDATE on the existing row, rather than mistakenly attempting a second INSERT.
  6. Verification: repo.count() confirms exactly one row still exists (not two, which an incorrect second INSERT attempt — or, in a database enforcing unique IDs, an outright constraint violation — would have produced), and finalState.getStatus() confirms the update itself genuinely took effect.
 new Order()            isNew=true  (default)
        |
   save() -> INSERT     (isNew was true)
        |
 @PostPersist -> isNew=false  (on the SAME instance)

 findById(...)           NEW instance, isNew defaults to true again
        |
 @PostLoad -> isNew=false     (corrected immediately after loading)
        |
   save() -> UPDATE      (isNew is now correctly false)

7. Gotchas & takeaways

Gotcha: @Transient on the isNew field (used throughout this card's examples) is essential — without it, JPA would try to map isNew to an actual database column, persisting and reloading whatever boolean value happened to be there, which would defeat the entire purpose of the flag (it needs to reflect the *current in-memory instance's* lifecycle state, not a stored database value). Forgetting @Transient here is an easy, subtle mistake.
  • JPA's @GeneratedValue strategies (IDENTITY, SEQUENCE, TABLE, AUTO) handle database-generated identifiers, where save()'s default null-check-based new-vs-existing detection works correctly, since a not-yet-saved entity's ID is reliably null.
  • Manually-assigned identifiers (UUIDs generated in Java, natural business keys) break that null-check heuristic, since such an ID is never null even for a genuinely new entity — leading to a silently-failed save() that performs a no-op UPDATE instead of the intended INSERT.
  • Persistable<ID>'s explicit isNew() method fixes this by replacing the unreliable heuristic with a definite, application-controlled answer, checked directly by Spring Data's save logic whenever an entity implements the interface.
  • @PostPersist and @PostLoad lifecycle callbacks are the standard way to keep a Persistable entity's isNew flag accurate across its full lifecycle — true only for a genuinely fresh, unsaved instance, false for anything that's ever been inserted or loaded from the database.
