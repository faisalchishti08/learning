LeetCode Patterns › 11. Two Heaps

IPO

← Back to checklist

1. What it is

Given k (number of projects you can pick), w (starting capital), profits[i], and capital[i] (minimum capital needed to start project i), pick up to k projects — each pick's profit adds to your capital, unlocking more projects — to MAXIMIZE final capital. Example: k = 2, w = 0, profits = [1,2,3], capital = [0,1,1] → 4.

2. Why & when

Two competing orders are at play again, like Single-Threaded CPU: projects become AVAILABLE based on capital required (which only grows as you accumulate profit), but among available projects, you should always pick the one with the HIGHEST profit. A min-heap by capital requirement finds what is unlockable; a max-heap by profit picks the best of what is unlocked.

3. Core concept

Key idea: put all projects in a min-heap ordered by capital required. At each of the k steps, move every project whose capital requirement is <= current w from the capital-heap into a max-heap ordered by profit. Pop the max-heap's top (the most profitable currently-startable project), add its profit to w, and repeat.

Steps:

  1. Build a min-heap of [capital, profit] pairs, ordered by capital, containing all projects.
  2. Repeat up to k times: move every project from the capital-heap whose capital <= w into a max-heap ordered by profit.
  3. If the profit-heap is empty (no project is currently affordable), stop early — no further projects can ever become available, since w only grows from completed projects.
  4. Otherwise, pop the profit-heap's top, add its profit to w.
  5. After k iterations (or an early stop), return the final w.

Why it is correct: always picking the highest-profit AVAILABLE project at each step is optimal, because profit gained now only ever increases future capital, which can only unlock MORE projects, never fewer — so there is never a reason to prefer a lower-profit available project over a higher-profit one. Moving newly-affordable projects into the profit-heap before each pick ensures every currently-startable option is considered.

4. Diagram

Projects unlock (move from the capital heap to the profit heap) as accumulated capital grows; the most profitable unlocked project is always picked next.

5. Runnable example

// IPO.java
import java.util.*;

public class IPO {

    // Level 1 -- Brute force: at each of the k steps, linearly scan
    // ALL remaining projects to find the affordable ones, then pick
    // the max-profit among them. Correct, but O(n) per step instead of
    // O(log n), and re-checks affordability for already-considered
    // projects every time.

    // KEY INSIGHT: use a min-heap by capital to efficiently find
    // NEWLY affordable projects (only check the ones not yet moved
    // over), and a max-heap by profit to instantly pick the best
    // available one -- both O(log n) instead of O(n) scans.

    // Level 2 -- Optimal: dual heaps, move-then-pick each round.
    static int findMaximizedCapital(int k, int w, int[] profits, int[] capital) {
        int n = profits.length;
        PriorityQueue<int[]> byCapital = new PriorityQueue<>((a, b) -> a[0] - b[0]);
        for (int i = 0; i < n; i++) byCapital.add(new int[]{capital[i], profits[i]});

        PriorityQueue<Integer> byProfit = new PriorityQueue<>(Collections.reverseOrder());

        for (int round = 0; round < k; round++) {
            while (!byCapital.isEmpty() && byCapital.peek()[0] <= w) {
                byProfit.add(byCapital.poll()[1]);
            }
            if (byProfit.isEmpty()) break;
            w += byProfit.poll();
        }
        return w;
    }

    // Level 3 -- Hardened: the `if (byProfit.isEmpty()) break;` check
    // correctly stops early when no project is currently affordable
    // and none ever will be, instead of looping k times uselessly or
    // crashing on an empty-heap poll.

    public static void main(String[] args) {
        System.out.println(findMaximizedCapital(2, 0, new int[]{1,2,3}, new int[]{0,1,1})); // 4
        System.out.println(findMaximizedCapital(3, 0, new int[]{1,2,3}, new int[]{0,1,2})); // 6
    }
}

How to run: java IPO.java

6. Walkthrough

Trace k = 2, w = 0, profits = [1,2,3], capital = [0,1,1]:

Roundw beforeMoved to byProfit (capital <= w)byProfit topw after
10project0 [cap=0,profit=1]10+1=1
21project1 [cap=1,profit=2], project2 [cap=1,profit=3]3 (highest)1+3=4

Final w = 4, matching the expected output. Time complexity is O(n log n), since each project is moved between heaps at most once, each move costing O(log n); space is O(n) for both heaps.

7. Gotchas & takeaways

Gotcha: moving projects into the profit-heap only ONCE at the start (instead of re-checking the capital-heap before EVERY round) misses projects that only became affordable after an earlier round's profit was added to w.
  • Check byCapital.peek()[0] <= w freshly at the START of every round, since w changes after each pick — projects that were unaffordable last round may be affordable now.
  • Breaking early when byProfit is empty is essential — without it, byProfit.poll() on an empty heap either throws or (with poll(), which returns null) causes a NullPointerException on unboxing.
  • Related problems: Furthest Building You Can Reach (single heap, different resource-tracking greedy), Maximum Average Pass Ratio (max-heap greedy selection by marginal gain instead of raw value).
