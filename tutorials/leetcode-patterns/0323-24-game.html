24 Game

← Back to checklist

1. What it is

Given 4 cards, each showing a number from 1 to 9, determine if they can be combined using +, -, *, /, and parentheses, using EACH card exactly once, to make the value 24. Division is real division (not integer division). Return true or false. Example: cards = [4,1,8,7] → true ((8-4) * (7-1) = 24).

2. Why & when

This is a different flavor of backtracking: instead of choosing among a fixed set of positions, you repeatedly PICK TWO numbers from the current pool, combine them with one of 4 operators, and replace both with the result — shrinking the pool by one each round. Use this shape whenever a problem must explore every way of pairwise-combining a small set of values down to a single result.

3. Core concept

Key idea: with a list of numbers, if only one remains, check if it is close enough to 24 (using a small epsilon, since division introduces floating-point error). Otherwise, try every pair of numbers, every operator between them, replace that pair with the computed result, and recurse on the smaller list.

Steps:

  1. Define backtrack(numbers: List<Double>). Base case: if numbers.size() == 1, return whether Math.abs(numbers.get(0) - 24.0) < 1e-6.
  2. Loop over every pair (i, j) with i < j in numbers: for each of the 4 operators, compute the result of combining numbers[i] and numbers[j] (for - and /, try BOTH orders, since these operators are not commutative; skip division by (near) zero).
  3. Build a NEW list containing every number except i and j, plus the computed result (choose).
  4. Recurse on this new, smaller list. If it returns true, propagate true immediately.
  5. (No explicit un-choose is needed, since a fresh list is built for each recursive call rather than mutating a shared one.)

Why it is correct: with n numbers, any way of reaching a single final value via binary operators corresponds to repeatedly picking two numbers and combining them into one, n - 1 times total — trying EVERY pair and EVERY operator at each step, recursively, explores every possible way of doing this. Because floating-point division can introduce tiny rounding errors (e.g., 1/3 * 3 might not be EXACTLY 1.0), comparing against 24 with a small epsilon tolerance (like 1e-6) instead of exact equality is required for correctness.

5. Runnable example

// TwentyFourGame.java
import java.util.*;

public class TwentyFourGame {

    static final double EPSILON = 1e-6;

    // KEY INSIGHT: repeatedly combine any 2 numbers in the pool into 1
    // via each operator, shrinking the pool by 1 each round -- this
    // explores every possible way of fully parenthesizing an
    // expression over the 4 numbers, without building expression
    // strings at all.

    static boolean judgePoint24(int[] cards) {
        List<Double> numbers = new ArrayList<>();
        for (int card : cards) numbers.add((double) card);
        return backtrack(numbers);
    }

    static boolean backtrack(List<Double> numbers) {
        if (numbers.size() == 1) {
            return Math.abs(numbers.get(0) - 24.0) < EPSILON;
        }

        for (int i = 0; i < numbers.size(); i++) {
            for (int j = i + 1; j < numbers.size(); j++) {
                double a = numbers.get(i), b = numbers.get(j);
                List<Double> rest = new ArrayList<>(numbers);
                rest.remove(j);
                rest.remove(i);

                List<Double> candidates = new ArrayList<>(Arrays.asList(a + b, a - b, b - a, a * b));
                if (Math.abs(b) > EPSILON) candidates.add(a / b);
                if (Math.abs(a) > EPSILON) candidates.add(b / a);

                for (double result : candidates) {
                    rest.add(result);                 // choose
                    if (backtrack(rest)) return true;  // recurse
                    rest.remove(rest.size() - 1);      // un-choose
                }
            }
        }
        return false;
    }

    public static void main(String[] args) {
        System.out.println(judgePoint24(new int[]{4, 1, 8, 7}));
        // true
        System.out.println(judgePoint24(new int[]{1, 2, 1, 2}));
        // false
    }
}

How to run: java TwentyFourGame.java

Final result: true. Time complexity is O(1) in the formal sense, since the input is always exactly 4 numbers — the search explores a small, fixed-size tree (roughly (4·3/2) · 6 choices for the first pairing, then (3·2/2) · 6 for the second, then 6 for the last, all bounded constants). Space is O(1) as well, for the same reason.

7. Gotchas & takeaways

Gotcha: comparing the final result to 24 with EXACT equality (numbers.get(0) == 24.0) can incorrectly return false for a mathematically correct combination, purely due to floating-point rounding from division — always use an epsilon-tolerant comparison for problems involving floating-point division.
  • Representing the "expression built so far" implicitly, as just its NUMERIC VALUE (not a string), is a simplification available whenever the problem only needs a yes/no answer, not the actual expression.
  • Trying both a - b and b - a (and both a / b and b / a) is necessary because subtraction and division are NOT commutative — skipping either order would miss valid combinations.
  • Related problems: Expression Add Operators (a related problem that DOES need the actual expression string, using left-to-right backtracking instead of pairwise combination), Different Ways to Add Parentheses (explores every full parenthesization of a fixed left-to-right expression).
