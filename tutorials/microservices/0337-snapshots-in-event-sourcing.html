"Snapshots in event sourcing"

← Back to checklist

1. What it is

A snapshot in event sourcing is a periodically saved copy of an entity's fully-folded state at a specific event version, stored alongside the event store so that reconstructing current state doesn't require replaying every event from the very beginning every time — instead, you load the nearest snapshot and fold only the (usually much smaller number of) events that happened after it.

2. Why & when

Folding an entity's events from scratch is cheap when there are a handful of events, but an entity with a long history (a years-old bank account with tens of thousands of transactions, a long-lived order with many revisions) becomes slow to reconstruct if every read has to replay its entire history from event one. Snapshots solve this the same way a video game's "save point" does: instead of replaying the whole game from the start, you load the save and continue from there.

Take a snapshot once an entity's event count crosses a threshold where full replay measurably affects read latency — commonly every N events (every 100, every 1000, tuned to the domain), not on every single event, since a snapshot after every event would cost as much as just storing current state directly and would defeat the point of event sourcing. Snapshots are purely a performance optimization; the event stream, never the snapshot, remains the actual source of truth — a lost or corrupted snapshot can always be regenerated by refolding from the beginning.

3. Core concept

A snapshot stores the folded state plus the event version it was taken at. Reconstructing current state becomes: load the latest snapshot at-or-before the requested point (or the entity's genesis if none exists yet), then fold only the events with a version greater than the snapshot's version, starting from the snapshot's state rather than an empty one.

record Snapshot<T>(int atVersion, T state) {}
// currentState = fold(events.filter(v -> v.version() > snapshot.atVersion()), startingFrom: snapshot.state())

5. Runnable example

Scenario: an event-sourced account balance that first reconstructs state by always folding from the very beginning (slow as history grows), then adds a snapshot to skip most of that work, and finally adds automatic periodic snapshotting so the optimization applies without manual intervention.

Level 1 — Basic

// File: FullReplayEveryTime.java -- reconstructs current state by
// folding EVERY event from the beginning, every single time.
import java.util.*;

public class FullReplayEveryTime {
    record Deposited(int version, double amount) {}
    static List<Deposited> events = new ArrayList<>();

    static double currentBalance() { // folds ALL events, from version 1, every call
        double balance = 0;
        for (Deposited e : events) balance += e.amount(); // O(n) in the FULL history, every time
        return balance;
    }

    public static void main(String[] args) {
        for (int i = 1; i <= 1000; i++) events.add(new Deposited(i, 1.0)); // 1000 historical deposits

        long start = System.nanoTime();
        double balance = currentBalance(); // must replay ALL 1000 events
        long elapsedNanos = System.nanoTime() - start;

        System.out.println("Balance: " + balance + ", computed by folding " + events.size()
                + " events (elapsed ~" + elapsedNanos + "ns) -- this cost GROWS with the full history.");
    }
}

How to run: java FullReplayEveryTime.java

currentBalance has no shortcut: every single call folds the entire events list from the start, regardless of how many times the balance has already been computed before. As events grows into the tens or hundreds of thousands, this cost grows right along with it.

Level 2 — Intermediate

// File: SnapshotSkipsMostReplay.java -- a snapshot taken at version 900
// lets us fold only the 100 events AFTER it, instead of all 1000.
import java.util.*;

public class SnapshotSkipsMostReplay {
    record Deposited(int version, double amount) {}
    record Snapshot(int atVersion, double balanceAtThatVersion) {}
    static List<Deposited> events = new ArrayList<>();
    static Snapshot latestSnapshot = null;

    static double currentBalanceUsingSnapshot() {
        double startingBalance = (latestSnapshot != null) ? latestSnapshot.balanceAtThatVersion() : 0;
        int startVersion = (latestSnapshot != null) ? latestSnapshot.atVersion() : 0;

        double balance = startingBalance;
        int foldedCount = 0;
        for (Deposited e : events) {
            if (e.version() <= startVersion) continue; // SKIP everything already captured in the snapshot
            balance += e.amount();
            foldedCount++;
        }
        System.out.println("folded only " + foldedCount + " events (skipped " + startVersion + " via snapshot)");
        return balance;
    }

    public static void main(String[] args) {
        for (int i = 1; i <= 1000; i++) events.add(new Deposited(i, 1.0));
        latestSnapshot = new Snapshot(900, 900.0); // pretend a snapshot was taken earlier, at version 900

        double balance = currentBalanceUsingSnapshot();
        System.out.println("Balance: " + balance + " -- CORRECT, but only 100 events were actually folded, not 1000.");
    }
}

How to run: java SnapshotSkipsMostReplay.java

currentBalanceUsingSnapshot starts from latestSnapshot.balanceAtThatVersion() (900.0) rather than 0, and the loop's if (e.version() <= startVersion) continue; skips every one of the first 900 events. Only the 100 events with version > 900 are actually folded and added, producing the same correct total of 1000.0 while doing roughly a tenth of the work Level 1 did.

Level 3 — Advanced

// File: AutomaticPeriodicSnapshotting.java -- a snapshot is taken
// AUTOMATICALLY every 100 events, so the optimization applies without
// anyone remembering to trigger it manually.
import java.util.*;

public class AutomaticPeriodicSnapshotting {
    record Deposited(int version, double amount) {}
    record Snapshot(int atVersion, double balanceAtThatVersion) {}
    static List<Deposited> events = new ArrayList<>();
    static Snapshot latestSnapshot = null;
    static final int SNAPSHOT_INTERVAL = 100;

    static double foldFromSnapshot() {
        double startingBalance = (latestSnapshot != null) ? latestSnapshot.balanceAtThatVersion() : 0;
        int startVersion = (latestSnapshot != null) ? latestSnapshot.atVersion() : 0;
        double balance = startingBalance;
        for (Deposited e : events) {
            if (e.version() <= startVersion) continue;
            balance += e.amount();
        }
        return balance;
    }

    static void appendDeposit(double amount) {
        int nextVersion = events.size() + 1;
        events.add(new Deposited(nextVersion, amount));

        if (nextVersion % SNAPSHOT_INTERVAL == 0) { // AUTOMATIC -- no manual trigger needed
            double balanceNow = foldFromSnapshot();
            latestSnapshot = new Snapshot(nextVersion, balanceNow);
            System.out.println("  auto-snapshot taken at version " + nextVersion + ", balance=" + balanceNow);
        }
    }

    public static void main(String[] args) {
        for (int i = 0; i < 250; i++) appendDeposit(1.0); // crosses 100 and 200 automatically along the way

        System.out.println("Latest snapshot: version=" + latestSnapshot.atVersion() + ", balance=" + latestSnapshot.balanceAtThatVersion());
        double finalBalance = foldFromSnapshot();
        System.out.println("Final balance (folded from latest snapshot): " + finalBalance
                + " -- only the events after version " + latestSnapshot.atVersion() + " needed folding.");
    }
}

How to run: java AutomaticPeriodicSnapshotting.java

appendDeposit checks, after every append, whether the new version is an exact multiple of SNAPSHOT_INTERVAL (100); if so, it computes the current balance (using whatever snapshot already exists, so this stays cheap even as snapshots accumulate) and stores a new snapshot at that version. Across 250 appends, this fires automatically at versions 100 and 200, so by the end, latestSnapshot is at version 200, and only the 50 events from 201 to 250 would need folding to get the final balance — with nobody having manually decided when to snapshot.

6. Walkthrough

Trace AutomaticPeriodicSnapshotting.main in order. First, the loop calls appendDeposit(1.0) 250 times. Each call appends one Deposited event with an incrementing version and checks nextVersion % 100 == 0.

At the 100th call, nextVersion is 100, so 100 % 100 == 0 is true: foldFromSnapshot() runs (at this point latestSnapshot is still null, so it folds all 100 events from scratch, getting 100.0), and latestSnapshot is set to Snapshot(atVersion=100, balanceAtThatVersion=100.0).

Calls 101 through 199 each append an event but nextVersion % 100 != 0, so no new snapshot is taken during this stretch.

At the 200th call, nextVersion is 200, triggering another snapshot: foldFromSnapshot() now starts from the *existing* snapshot at version 100 (balance 100.0) and folds only events 101 through 200 (100 events, each adding 1.0), arriving at 200.0 — cheaper than a full replay would have been, since even taking a snapshot itself benefits from the previous one. latestSnapshot updates to Snapshot(atVersion=200, balanceAtThatVersion=200.0).

Calls 201 through 250 append more events with no further snapshot (250 is not a multiple of 100 within this run's range... note 250 % 100 = 50, so no trigger).

Finally, main prints the latest snapshot (atVersion=200, balance=200.0) and calls foldFromSnapshot() one more time: it starts from that snapshot's balance and folds only the 50 events from version 201 to 250, arriving at the correct final balance of 250.0 — having folded far fewer than the full 250 events to get there.

appendDeposit x100 -> auto-snapshot @ v100, balance=100.0 (first snapshot: folded from scratch)
appendDeposit x100 -> auto-snapshot @ v200, balance=200.0 (folded only v101-200, using prior snapshot)
appendDeposit x50  -> no snapshot triggered (250 not a multiple of 100)
foldFromSnapshot()  -> starts @ v200/200.0 -> folds v201-250 (50 events) -> 250.0

7. Gotchas & takeaways

A snapshot is purely a cache of a computed fold — it must never be treated as more authoritative than the event stream itself. If a bug in the folding logic produced a bad snapshot, the fix is to correct the logic and regenerate snapshots by refolding from the real events, never to hand-edit a snapshot directly.
  • Snapshots exist purely to bound the cost of reconstructing state as an entity's event history grows long; the event stream remains the actual source of truth.
  • Take snapshots periodically (every N events), not after every single event — snapshotting too often defeats the purpose and approaches the cost of just storing current state directly.
  • Reconstructing state becomes: load the nearest snapshot, then fold only the events after it, starting the fold from the snapshot's state instead of an empty one.
  • A lost or corrupted snapshot is always recoverable by refolding from the beginning of the event store — this safety property is why snapshots can be treated as a pure optimization, never a risk to correctness.
