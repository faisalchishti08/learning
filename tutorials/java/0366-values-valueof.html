values() & valueOf()

← Back to checklist

1. What it is

Every enum automatically gets two static methods generated by the compiler, with no code written by you: values(), which returns an array containing every constant in declaration order, and valueOf(String name), which looks up a constant by its exact name and returns it — or throws IllegalArgumentException if no constant with that exact name exists. Both are essential bridges between an enum's fixed, closed set of values and the more dynamic, string- or loop-based code that often needs to work with them.

2. Why & when

values() is what makes it possible to loop over "every possible value of this enum" without hardcoding the list anywhere else — useful for populating a dropdown, validating input against all legal options, or building a lookup table keyed by enum. Because it returns a new array on every call, you can safely mutate the returned array without corrupting the enum's real internal state.

valueOf(String) is the standard way to convert user input, configuration text, or data read from a file or database back into the correct enum constant. It's strict by design: the name must match exactly, including case (valueOf("monday") fails even though MONDAY exists), which is precisely why it throws rather than silently returning null or an arbitrary default — a typo or wrong-case string is treated as a real error, not something to paper over.

You use values() whenever you need to enumerate every possible constant, and valueOf() whenever you're crossing a text/enum boundary — reading a status from a config file, a command-line argument, or a database column stored as text.

3. Core concept

public class ValuesValueOfDemo {
    enum Day { MONDAY, TUESDAY, WEDNESDAY }

    public static void main(String[] args) {
        for (Day d : Day.values()) { // every constant, in declaration order
            System.out.println(d + " (ordinal " + d.ordinal() + ")");
        }

        Day parsed = Day.valueOf("TUESDAY"); // exact-name lookup
        System.out.println("Parsed: " + parsed);

        try {
            Day.valueOf("tuesday"); // wrong case -- fails
        } catch (IllegalArgumentException e) {
            System.out.println("Failed as expected: " + e.getMessage());
        }
    }
}

How to run: java ValuesValueOfDemo.java

Day.values() returns [MONDAY, TUESDAY, WEDNESDAY] in the order they were declared; the loop prints each with its ordinal. Day.valueOf("TUESDAY") succeeds because the name matches exactly; Day.valueOf("tuesday") fails with IllegalArgumentException, since valueOf never ignores case.

5. Runnable example

Scenario: parsing a status string from a config file, evolved from a naive crash-prone version, to one handling the bad-input case explicitly, to a version that also validates against every known constant up front using values().

Level 1 — Basic

public class ConfigBasic {
    enum LogLevel { DEBUG, INFO, WARN, ERROR }

    public static void main(String[] args) {
        String configured = "INFO"; // imagine this came from a config file
        LogLevel level = LogLevel.valueOf(configured);
        System.out.println("Log level: " + level);
    }
}

How to run: java ConfigBasic.java

The happy path: a correctly-spelled, correctly-cased string parses cleanly into its matching enum constant via valueOf.

Level 2 — Intermediate

public class ConfigIntermediate {
    enum LogLevel { DEBUG, INFO, WARN, ERROR }

    static LogLevel parseLevel(String raw) {
        try {
            return LogLevel.valueOf(raw.trim().toUpperCase()); // normalise before lookup
        } catch (IllegalArgumentException e) {
            System.out.println("Invalid log level '" + raw + "', defaulting to INFO");
            return LogLevel.INFO;
        }
    }

    public static void main(String[] args) {
        System.out.println(parseLevel(" warn "));
        System.out.println(parseLevel("VERBOSE")); // not a real constant
    }
}

How to run: java ConfigIntermediate.java

This handles the real-world messiness valueOf refuses to tolerate on its own: raw.trim().toUpperCase() normalises whitespace and case before the lookup, and a try/catch around valueOf provides a sensible fallback (INFO) instead of letting a config typo crash the whole program.

Level 3 — Advanced

import java.util.Arrays;
import java.util.List;

public class ConfigAdvanced {
    enum LogLevel { DEBUG, INFO, WARN, ERROR }

    static LogLevel parseLevel(String raw) {
        String normalized = raw.trim().toUpperCase();
        for (LogLevel level : LogLevel.values()) { // validate against every real constant
            if (level.name().equals(normalized)) {
                return level;
            }
        }
        List<String> valid = Arrays.stream(LogLevel.values()).map(Enum::name).toList();
        throw new IllegalArgumentException(
                "Invalid log level '" + raw + "'. Valid options: " + valid);
    }

    public static void main(String[] args) {
        System.out.println(parseLevel("error"));
        try {
            parseLevel("VERBOSE");
        } catch (IllegalArgumentException e) {
            System.out.println("Rejected: " + e.getMessage());
        }
    }
}

How to run: java ConfigAdvanced.java

Instead of leaning on valueOf's own exception, this manually walks LogLevel.values() and compares each constant's .name() — functionally similar to valueOf, but it lets the code build a helpful error message listing every *actually valid* option, generated dynamically from values() rather than hardcoded, so it never goes stale if constants are added or removed later.

6. Walkthrough

Execution starts in main. parseLevel("error") runs first: raw.trim().toUpperCase() produces "ERROR". The for (LogLevel level : LogLevel.values()) loop iterates [DEBUG, INFO, WARN, ERROR] in declaration order. DEBUG.name().equals("ERROR") is false, as is INFO and WARN; when level reaches ERROR, "ERROR".equals("ERROR") is true, so the method returns LogLevel.ERROR immediately. main prints ERROR.

parseLevel("VERBOSE") runs next: normalising gives "VERBOSE". The loop checks all four real constants (DEBUG, INFO, WARN, ERROR) and none of their names equal "VERBOSE", so the loop finishes without returning. Control falls to the line after the loop: Arrays.stream(LogLevel.values()).map(Enum::name).toList() builds ["DEBUG", "INFO", "WARN", "ERROR"] by mapping each constant to its name, and this list is embedded in a new IllegalArgumentException's message, which is then thrown.

Back in main, the try block around parseLevel("VERBOSE") catches this exception in the catch (IllegalArgumentException e) block, and prints Rejected: followed by the exception's message.

Expected output:

ERROR
Rejected: Invalid log level 'VERBOSE'. Valid options: [DEBUG, INFO, WARN, ERROR]

7. Gotchas & takeaways

valueOf is case-sensitive and whitespace-sensitive — it never trims or normalises the string you pass it. Always normalise external input (.trim().toUpperCase(), matching your constant naming convention) before calling valueOf, or you'll see spurious IllegalArgumentExceptions from otherwise "obviously correct" input like " info ".
  • values() returns a fresh array of every constant in declaration order, every time it's called — safe to loop over or mutate without side effects on the enum itself.
  • valueOf(String) requires an exact, case-sensitive match to a real constant's name, and throws IllegalArgumentException (not null) when there's no match.
  • Both methods are generated automatically by the compiler for every enum — you never declare them yourself, but they're always available.
  • When parsing external, potentially messy input (config files, CLI args, database text) into an enum, normalise the string yourself and wrap valueOf in a try/catch, or manually search values() for more control over the error message.
  • Building a "list of valid options" error message from LogLevel.values() (rather than a hardcoded string) keeps that message automatically correct even as constants are added, renamed, or removed later.
