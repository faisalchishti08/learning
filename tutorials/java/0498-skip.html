skip()

← Back to checklist

1. What it is

Stream.skip(n) returns a stream with the first n elements of the source discarded, keeping everything after them. If the source has fewer than n elements, the result is simply an empty stream. It's the complement of limit: limit keeps the *front*, skip discards the front and keeps the *rest*.

2. Why & when

skip is how you express "ignore the first N" — skipping a header row before processing data, jumping past already-seen items, or implementing pagination's "offset" half (skip for the offset, limit for the page size). Combined, skip(offset).limit(pageSize) is the standard streams idiom for extracting one page out of a larger sequence.

3. Core concept

import java.util.stream.*;

List<Integer> afterFirstTwo = Stream.of(10, 20, 30, 40, 50)
        .skip(2)
        .toList(); // [30, 40, 50]

// Page 2 of size 2, out of a longer list: skip the first page, then take the next
List<Integer> page2 = Stream.of(10, 20, 30, 40, 50)
        .skip(2)
        .limit(2)
        .toList(); // [30, 40]

skip(n) discards exactly the first n elements (or all of them, if fewer than n exist), leaving the rest untouched and in their original order.

5. Runnable example

Scenario: implementing paginated results for a search API — evolved from a plain header-row skip, through a full skip+limit pagination pattern, to a version that safely handles out-of-range page requests.

Level 1 — Basic

import java.util.*;
import java.util.stream.*;

public class SkipBasic {
    public static void main(String[] args) {
        List<String> csvLines = List.of("name,age", "Alice,30", "Bob,25", "Carol,35");

        List<String> dataRows = csvLines.stream()
                .skip(1) // discard the header row
                .toList();

        System.out.println("Data rows: " + dataRows);
    }
}

How to run: java SkipBasic.java

Expected output:

Data rows: [Alice,30, Bob,25, Carol,35]

.skip(1) discards the first element — the CSV header row "name,age" — leaving only the three actual data rows, in their original order.

Level 2 — Intermediate

import java.util.*;
import java.util.stream.*;

public class SkipLimitPagination {
    public static void main(String[] args) {
        List<String> results = List.of("r1", "r2", "r3", "r4", "r5", "r6", "r7");
        int pageSize = 3;
        int pageNumber = 2; // 1-based

        List<String> page = results.stream()
                .skip((long) (pageNumber - 1) * pageSize) // offset past earlier pages
                .limit(pageSize)                          // take just this page's worth
                .toList();

        System.out.println("Page " + pageNumber + ": " + page);
    }
}

How to run: java SkipLimitPagination.java

Expected output:

Page 2: [r4, r5, r6]

The real-world concern this adds: full pagination needs both an offset (skip) and a page size (limit). For page 2 with pageSize = 3, the offset is (2 - 1) * 3 = 3, so .skip(3) discards r1, r2, r3, and .limit(3) then keeps only the next three: r4, r5, r6 — exactly the second page of results.

Level 3 — Advanced

import java.util.*;
import java.util.stream.*;

public class SkipPaginationSafe {
    record Page<T>(List<T> items, int pageNumber, int totalPages) {}

    static <T> Page<T> paginate(List<T> source, int pageNumber, int pageSize) {
        int totalPages = (int) Math.ceil((double) source.size() / pageSize);
        // Clamp the requested page into the valid range instead of returning garbage for out-of-range input.
        int safePage = Math.max(1, Math.min(pageNumber, Math.max(totalPages, 1)));

        List<T> items = source.stream()
                .skip((long) (safePage - 1) * pageSize)
                .limit(pageSize)
                .toList();

        return new Page<>(items, safePage, totalPages);
    }

    public static void main(String[] args) {
        List<String> results = List.of("r1", "r2", "r3", "r4", "r5", "r6", "r7");

        Page<String> normal = paginate(results, 2, 3);
        System.out.println("Requested page 2: " + normal.items() + " (page " + normal.pageNumber() + " of " + normal.totalPages() + ")");

        Page<String> tooHigh = paginate(results, 99, 3); // way past the end
        System.out.println("Requested page 99: " + tooHigh.items() + " (page " + tooHigh.pageNumber() + " of " + tooHigh.totalPages() + ")");
    }
}

How to run: java SkipPaginationSafe.java

Expected output:

Requested page 2: [r4, r5, r6] (page 2 of 3)
Requested page 99: [r7] (page 3 of 3)

This adds resilience: a naive skip((pageNumber - 1) * pageSize) with an out-of-range pageNumber (like 99) would .skip(...) past the entire source and silently return an empty page with no indication anything was wrong. paginate instead clamps pageNumber into the valid [1, totalPages] range first, so requesting page 99 on a 3-page result set is treated as a request for the actual last page (3), returning its one remaining item (r7) rather than an empty, confusing result.

6. Walkthrough

Execution starts in main of the Level 3 example. results holds seven items, r1 through r7.

paginate(results, 99, 3) is called for the second scenario. Inside, totalPages = Math.ceil(7.0 / 3) = Math.ceil(2.33...) = 3. safePage = Math.max(1, Math.min(99, Math.max(3, 1))): the inner Math.min(99, 3) evaluates to 3 (since 99 exceeds the valid page count), and the outer Math.max(1, 3) stays 3 — so the wildly out-of-range request for page 99 is clamped down to the actual last page, 3.

source.stream().skip((3 - 1) * 3).limit(3) then runs with the *clamped* page number: skip((3-1)*3) = skip(6) discards the first six elements (r1 through r6), leaving only r7. .limit(3) would keep up to three elements, but only one (r7) remains in the stream to take, so the result is [r7].

requested pageNumber = 99, pageSize = 3, source has 7 items
totalPages = ceil(7/3) = 3
safePage   = clamp(99, 1, 3) = 3
skip((3-1)*3) = skip(6)  -> discards r1..r6, leaves [r7]
limit(3)                 -> takes up to 3, only r7 available -> [r7]

paginate returns Page(["r7"], 3, 3) — the clamped page number (3) is reported honestly rather than the originally-requested 99, so the caller can see their request was adjusted. main prints "Requested page 99: [r7] (page 3 of 3)", clearly showing both what was returned and which real page it corresponds to.

7. Gotchas & takeaways

skip(n) where n exceeds the stream's actual size does not throw — it simply produces an empty stream, since there's nothing left after skipping past everything. This is convenient (no bounds-checking exception to catch) but also means an out-of-range page request will silently return an empty result unless you explicitly detect and handle that case, as paginate does in Level 3.
  • skip(n) discards the first n elements and passes through everything after them, in original order.
  • skip(offset).limit(pageSize) is the standard streams pattern for pagination: skip implements the offset, limit implements the page size.
  • If n is larger than the number of elements available, skip simply empties the stream rather than throwing an exception.
  • For robust pagination, clamp the requested page/offset into a valid range before computing skip's argument, rather than trusting caller-supplied page numbers directly.
  • skip and limit are both intermediate operations and compose naturally with filter/map/sorted earlier or later in the same pipeline.
