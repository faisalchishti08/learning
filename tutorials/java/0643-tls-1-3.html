TLS 1.3

← Back to checklist

1. What it is

Java 11 added support for TLS 1.3 (RFC 8446), the latest version of the Transport Layer Security protocol that secures virtually all internet communication. TLS 1.3 is a major overhaul: it removes outdated cryptographic algorithms (RSA key exchange, CBC-mode ciphers, SHA-1 hashing, MD5), reduces the handshake from two round-trips to one (improving connection setup speed by ~30-40%), and mandates forward secrecy for all cipher suites. Java 11's SSLContext and SSLEngine support TLS 1.3 by default when both client and server support it, with transparent negotiation via the TLS supported_versions extension.

2. Why & when

TLS 1.2, while secure when properly configured, has a large attack surface from legacy algorithms (dozens of cipher suites, many of which are weak) and a slow two-round-trip handshake. TLS 1.3 simplifies the protocol to five cipher suites (all AEAD-based), eliminates the vulnerable renegotiation feature, and encrypts more of the handshake (including server certificates). Use TLS 1.3 for all new Java applications — it is the default in Java 11+ when both endpoints support it. The fallback to TLS 1.2 is automatic. You don't need to change any code; the JDK's HttpsURLConnection and HttpClient use TLS 1.3 automatically.

3. Core concept

// TLS 1.3 is automatically negotiated — no code changes needed.
// To check what protocol was used:

HttpClient client = HttpClient.newHttpClient();
HttpRequest req = HttpRequest.newBuilder()
    .uri(URI.create("https://www.google.com"))
    .GET().build();

HttpResponse<String> resp = client.send(req, BodyHandlers.ofString());
// The TLS version is negotiated at the SSL layer — not directly exposed
// But you can check via SSLContext:

SSLContext ctx = SSLContext.getDefault();
SSLParameters params = ctx.getDefaultSSLParameters();
System.out.println("Default protocols: " +
    Arrays.toString(params.getProtocols()));
// Output: [TLSv1.3, TLSv1.2] (TLS 1.3 first if available)

TLS 1.3 is transparent to application code. The JDK's security providers handle negotiation, cipher selection, and session resumption (now called "PSK" — Pre-Shared Key — instead of "session tickets").

5. Runnable example

Scenario: demonstrating TLS 1.3 support in Java 11 by inspecting SSL contexts and making HTTPS connections — starting with basic protocol detection, extending to SSL configuration, and finally discussing TLS 1.3 features.

Level 1 — Basic

// File: TLS13Demo.java
import javax.net.ssl.*;
import java.util.*;

public class TLS13Demo {
    public static void main(String[] args) {
        System.out.println("=== TLS 1.3 Support in Java ===\n");

        SSLContext ctx;
        try {
            ctx = SSLContext.getDefault();
        } catch (Exception e) {
            System.out.println("Error: " + e.getMessage());
            return;
        }

        // Check supported protocols
        SSLParameters params = ctx.getDefaultSSLParameters();
        System.out.println("Default protocols: " +
            Arrays.toString(params.getProtocols()));

        // Check supported cipher suites
        SSLEngine engine = ctx.createSSLEngine();
        String[] ciphers = engine.getEnabledCipherSuites();
        System.out.println("\nCipher suites (" + ciphers.length + " total):");

        // Show only TLS 1.3 ciphers (they have "TLS_" prefix in Java naming)
        System.out.println("\nTLS 1.3 ciphers:");
        for (String c : ciphers) {
            if (c.startsWith("TLS_AES") || c.startsWith("TLS_CHACHA20")) {
                System.out.println("  " + c);
            }
        }
    }
}

Level 3 — Advanced

// File: TLS13Advanced.java
import javax.net.ssl.*;
import java.security.*;
import java.util.*;

public class TLS13Advanced {
    public static void main(String[] args) throws Exception {
        System.out.println("=== TLS 1.3 — Advanced Configuration ===\n");

        // 1. Enforce TLS 1.3 only (no fallback)
        SSLContext tls13Ctx = SSLContext.getInstance("TLSv1.3");
        tls13Ctx.init(null, null, null);

        SSLParameters tls13Only = tls13Ctx.getDefaultSSLParameters();
        System.out.println("1. TLS 1.3-only context:");
        System.out.println("   Protocols: " + Arrays.toString(tls13Only.getProtocols()));

        // 2. Custom cipher preference
        SSLParameters customParams = new SSLParameters();
        customParams.setProtocols(new String[]{"TLSv1.3", "TLSv1.2"});
        // In TLS 1.3, the server selects the cipher; you can't enforce a specific one
        // But you can disable weak ciphers
        System.out.println("\n2. Custom parameters:");
        System.out.println("   Protocols allowed: " +
            Arrays.toString(customParams.getProtocols()));

        // 3. Check algorithm constraints
        System.out.println("\n3. Disabled algorithms (from java.security):");
        String disabled = Security.getProperty("jdk.tls.disabledAlgorithms");
        if (disabled != null) {
            String[] algos = disabled.split(",");
            for (String a : algos) {
                System.out.println("   " + a.trim());
            }
        } else {
            System.out.println("   (none explicitly disabled)");
        }

        // 4. Key TLS 1.3 changes
        System.out.println("\n4. What TLS 1.3 removes:");
        System.out.println("   ❌ RSA key exchange (no forward secrecy)");
        System.out.println("   ❌ CBC-mode ciphers (padding oracle attacks)");
        System.out.println("   ❌ SHA-1 and MD5 hashing");
        System.out.println("   ❌ Static RSA and DH key exchange");
        System.out.println("   ❌ Compression (CRIME attack)");
        System.out.println("   ❌ Renegotiation (protocol complexity)");
        System.out.println("   ❌ Custom DHE groups (use named groups)");

        System.out.println("\n5. What TLS 1.3 adds:");
        System.out.println("   ✅ 1-RTT handshake (0-RTT for resumed)");
        System.out.println("   ✅ Only AEAD ciphers (AES-GCM, ChaCha20-Poly1305)");
        System.out.println("   ✅ Mandatory forward secrecy");
        System.out.println("   ✅ Encrypted Server Name Indication (ESNI)");
        System.out.println("   ✅ Encrypted client certificate");
        System.out.println("   ✅ Simpler state machine (fewer bugs)");

        System.out.println("\n6. Backward compatibility:");
        System.out.println("   Java 11+ clients connect to TLS 1.2 servers seamlessly.");
        System.out.println("   The TLS 'supported_versions' extension handles negotiation.");
        System.out.println("   No application code changes required.");
    }
}

The production-flavoured hard cases: (1) TLS 1.3 is on by default in Java 11+ — it's actually harder to disable than to use. (2) Custom SSLContext — if your application manually configures SSLContext, ensure it doesn't restrict protocols to TLSv1.2-only unless intentionally. (3) Algorithm constraints — the jdk.tls.disabledAlgorithms security property automatically disables weak algorithms; TLS 1.3 mandates strong ones anyway. (4) 0-RTT — TLS 1.3's 0-RTT mode (early data) is supported by the JDK but requires explicit opt-in via SSLEngine/SSLSocket configuration since early data has replay risks.

6. Walkthrough

Tracing a TLS 1.3 handshake in Java 11's HttpClient:

  1. The client opens a TCP connection to www.google.com:443. No TLS yet.
  1. The TLS layer sends ClientHello with the supported_versions extension advertising TLS 1.3 (preferred) and TLS 1.2. It also includes key_share extension with the client's ECDHE public key — this is how TLS 1.3 achieves 1-RTT: the key agreement parameters are sent in the first message.
  1. Google's server responds with ServerHello selecting TLS 1.3, its own key_share, and its certificate (encrypted!). The server sends Finished immediately — it has everything needed after one round-trip.
  1. The client verifies the certificate, computes the shared secret from the combined key shares, derives session keys, and sends Finished. The connection is now encrypted.
  1. HTTP/2 or HTTP/1.1 data flows over the encrypted channel. The entire TLS handshake took 1 round-trip (vs 2 for TLS 1.2).
  1. The HttpClient API doesn't expose the TLS version directly — this is intentional. The security layer is transparent. To verify TLS 1.3 was used, check via -Djavax.net.debug=ssl:handshake JVM flag or inspect the SSLSession from a lower-level API.

7. Gotchas & takeaways

TLS 1.3 changes cipher suite naming — they no longer include the key exchange method (since it's always ECDHE). For example, TLS_AES_128_GCM_SHA256 is a TLS 1.3 cipher. Old-style names like TLS_ECDHE_RSA_WITH_AES_128_GCM_SHA256 are TLS 1.2 only. Don't try to set TLS 1.2 cipher names on a TLS 1.3 connection — they're separate namespaces.
  • TLS 1.3 is enabled by default in Java 11+. No code changes or flag settings are needed. The JDK negotiates the highest TLS version both endpoints support.
  • Some servers don't support TLS 1.3 yet. In that case, the JDK automatically falls back to TLS 1.2. This is transparent — your connection succeeds either way.
  • TLS 1.3 removes static RSA key exchange, which means server private keys alone can't decrypt traffic. This is a security improvement (forward secrecy) but breaks passive network monitoring tools that rely on RSA key decryption.
  • The jdk.tls.disabledAlgorithms security property in java.security can disable specific algorithms. TLS 1.3-relevant entries include TLSv1, TLSv1.1, SSLv3, RC4, DES, MD5, 3DES.
  • TLS 1.3 is not available on Java 8 without additional libraries. Java 8 only supports up to TLS 1.2. Migrating to Java 11+ is the recommended path to TLS 1.3.
