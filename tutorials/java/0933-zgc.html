ZGC

← Back to checklist

1. What it is

ZGC is a low-latency garbage collector, production-ready since Java 15, designed to keep every stop-the-world pause under roughly a millisecond regardless of heap size — whether the heap is 8 GB or 8 TB. It achieves this by making almost every phase concurrent with the running application, including the phase every earlier collector treated as inherently requiring a pause: actually *moving* (relocating) live objects. ZGC accomplishes this using colored pointers (extra metadata bits packed into each object reference itself, marking its GC-relevant state) and a load barrier (a small check inserted into every reference load that can transparently redirect a thread to an object's new location if it has been relocated concurrently) — together, these let application threads keep reading and writing references correctly even while the collector is moving the objects those references point to, underneath them, in parallel.

2. Why & when

ZGC targets applications where pause time is the overriding concern and heap size is large enough that traditional collectors' pauses would be unacceptable — very large in-memory caches, large heaps for analytics or data-grid style services, and any latency-sensitive service where even G1's soft ~200ms target pauses are too long for the service's latency SLA. The tradeoff is real: colored pointers and load barriers add a small but nonzero overhead to *every* reference access in the running application (not just during a pause), and ZGC generally needs somewhat more headroom memory than G1 to do its concurrent relocation safely. For a small heap or a workload that doesn't care about tail latency (a batch job, for instance), this overhead buys nothing and Parallel GC or G1 remains the better choice — ZGC earns its keep specifically when heap size is large and pause time genuinely matters more than raw throughput.

3. Core concept

Traditional collector:  [STOP] -> mark -> relocate/compact -> fix references -> [RESUME]
                                   ^^^^^^^^^^^^^^^^^^^^^^^^^^^^
                                   the actual object-moving work pauses the app

ZGC:                    marking, relocation, and reference fix-up ALL happen
                         CONCURRENTLY with the app running -- pauses are only
                         brief moments to synchronize roots, NOT proportional
                         to heap size or live-set size.

How relocation is made safe while the app runs:
  - Each reference carries extra "colored" metadata bits (marked / relocatable / etc.)
  - A LOAD BARRIER checks these bits on every reference read
  - If an object has moved, the barrier transparently redirects to its new address
    and can even fix up the reference in place ("self-healing")
  - The app thread never observes a stale or dangling reference

The load barrier is the mechanism that makes concurrent relocation *correct*: it is the one small piece of extra work every reference read pays, in exchange for the collector never needing to stop the world just to move memory around.

5. Runnable example

Scenario: observe ZGC's headline property — pause time independent of heap size — starting with a basic ZGC-enabled workload, then scaling up to a genuinely large heap with sustained retention, then directly comparing max pause time against G1 on the identical large-heap workload.

Level 1 — Basic

public class ZgcBaseline {
    public static void main(String[] args) {
        for (int i = 0; i < 3_000_000; i++) {
            byte[] garbage = new byte[64];
        }
        System.out.println("done -- check -Xlog:gc for ZGC pause events, all should be sub-millisecond");
    }
}

How to run: java -Xlog:gc -XX:+UseZGC ZgcBaseline.java (JDK 17+; ZGC is generally available and production-ready from Java 15 onward).

Expected output shape:

[0.015s][info][gc] GC(0) Garbage Collection (Warmup) 4M(0%)->2M(0%) 0.412ms
[0.030s][info][gc] GC(1) Garbage Collection (Warmup) 5M(0%)->2M(0%) 0.389ms
...
done -- check -Xlog:gc for ZGC pause events, all should be sub-millisecond

Even at this small scale, ZGC's individual pause durations are already sub-millisecond, since the actual marking and relocation work is concurrent — the reported pause is only the brief synchronization step, not the bulk of the collection work.

Level 2 — Intermediate

import java.util.*;

public class ZgcLargeHeapRetention {
    public static void main(String[] args) {
        List<byte[]> retained = new ArrayList<>();
        for (int i = 0; i < 2_000_000; i++) {
            retained.add(new byte[500]); // ~1GB retained over the run
        }
        System.out.println("retained: " + retained.size());
        System.out.println("check pause durations -- should remain sub-millisecond despite the large heap");
    }
}

How to run: java -Xlog:gc -Xmx4g -XX:+UseZGC ZgcLargeHeapRetention.java (JDK 17+; a multi-gigabyte heap is where ZGC's design goal becomes meaningfully observable).

Expected output shape:

[0.02s][info][gc] GC(0) Garbage Collection (Warmup) 40M(1%)->10M(0%) 0.512ms
...
[1.80s][info][gc] GC(40) Garbage Collection (Allocation Rate) 950M(23%)->900M(22%) 0.687ms
retained: 2000000
check pause durations -- should remain sub-millisecond despite the large heap

The real-world concern added: retaining roughly a gigabyte of data on a 4 GB heap demonstrates ZGC's central claim directly — pause durations stay in the sub-millisecond range even as live-set size grows substantially, unlike traditional collectors whose pause duration scales with exactly that quantity.

Level 3 — Advanced

import java.util.*;

public class ZgcVsG1PauseComparison {
    public static void main(String[] args) {
        List<byte[]> retained = new ArrayList<>();
        long start = System.nanoTime();
        for (int i = 0; i < 4_000_000; i++) {
            retained.add(new byte[500]);
            if (i % 100_000 == 0) {
                for (int j = 0; j < 50_000; j++) {
                    byte[] churn = new byte[128];
                }
            }
        }
        long elapsedMs = (System.nanoTime() - start) / 1_000_000;
        System.out.println("retained: " + retained.size() + ", elapsed: " + elapsedMs + "ms");
    }
}

How to run: run against a large heap under both collectors and compare *maximum* single-pause duration (not just total time) from -Xlog:gc: java -Xlog:gc -Xmx6g -XX:+UseZGC ZgcVsG1PauseComparison.java versus java -Xlog:gc -Xmx6g -XX:+UseG1GC ZgcVsG1PauseComparison.java (JDK 17+).

Expected output shape (illustrative — exact numbers vary by machine):

(ZGC):  max single pause ~0.8ms across the whole run, total elapsed: 6100ms (slightly higher overhead per-access)
(G1):   max single pause ~180ms (a large mixed collection), total elapsed: 5400ms (lower overhead, larger worst-case pause)

The production-flavored hard case: on the same large-heap, high-retention workload, ZGC's worst-case single pause stays roughly two orders of magnitude smaller than G1's, at the cost of somewhat higher total elapsed time — precisely the throughput-for-latency tradeoff that makes ZGC the right choice specifically when bounding worst-case pause matters more than minimizing total run time.

6. Walkthrough

Tracing ZgcLargeHeapRetention.main end to end:

  1. The loop allocates and retains two million byte[] objects, growing the live set to roughly a gigabyte — under a traditional collector this would eventually force either a large young-to-old promotion wave or an expensive full compaction; under ZGC, the response is structurally different.
  2. As allocation proceeds, ZGC's concurrent marking threads trace reachability across the heap *while* the main thread keeps allocating and running — no stop-the-world mark phase is needed for the bulk of this work, only a brief pause to synchronize GC roots at the very start of a cycle.
  3. Once marking identifies which regions hold reclaimable garbage, ZGC begins concurrently relocating (moving) the surviving objects out of those regions — this is the step every earlier collector needed a pause for, since moving an object while a thread might read a stale reference to its old address is normally unsafe.
  4. ZGC makes this safe via colored pointers and the load barrier: every reference read by the application is checked against the colored metadata bits, and if the referenced object has already been relocated, the barrier transparently redirects the read to the object's new location (and can rewrite the reference in place, "healing" it for future reads) — so the application thread never observes an invalid reference, and never needs to be paused for this to be safe.
  5. Once relocation of a region completes and no live references to its old addresses remain, that region's old memory is reclaimed and becomes available for new allocations — all of this happening concurrently, with the application's retained list growing the entire time without interruption longer than roughly a millisecond at any single point.
  6. The program prints its final retained count and exits, with the surrounding GC log showing pause durations that stay flat and sub-millisecond throughout, regardless of how large the live set became during the run — the direct, observable confirmation of ZGC's core design goal.

7. Gotchas & takeaways

Gotcha: ZGC's low pause times are not free — the load barrier adds a small overhead to every single reference read in the application, and ZGC generally wants more spare heap headroom than G1 to safely relocate concurrently; for small heaps or throughput-bound batch workloads with no latency requirement, this overhead is pure cost with no corresponding benefit, and a simpler collector will finish the same work faster overall.
  • ZGC keeps individual stop-the-world pauses in the sub-millisecond range regardless of heap size, by making marking, relocation, and reference fix-up almost entirely concurrent with the running application.
  • Colored pointers (extra metadata bits in each reference) plus a load barrier (a check on every reference read) are what make concurrently relocating live objects safe — the barrier transparently redirects reads to an object's new location if it has moved.
  • It targets large heaps with strict latency requirements; the load-barrier overhead and extra memory headroom it needs make it a poor fit for small heaps or pure-throughput batch workloads.
  • Compare directly against G1: G1 bounds pause time by limiting how many regions it evacuates per pause, while ZGC removes the pause from evacuation almost entirely, at the cost of small constant overhead on every reference access.
  • See Shenandoah for a similarly concurrent, low-pause alternative that uses a different mechanism (forwarding pointers plus a read/write barrier) to achieve a comparable goal.
