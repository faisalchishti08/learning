Set

← Back to checklist

1. What it is

Set<T> is the Collection subtype that guarantees no duplicate elements — calling add() with a value already present is a no-op; it returns false and the set is unchanged. Unlike List, Set makes no promise about positional order (though specific implementations add their own: HashSet has no defined order, LinkedHashSet preserves insertion order, TreeSet keeps sorted order). Equality for membership is determined by each element's equals()/hashCode() (for HashSet/LinkedHashSet) or by a Comparator/Comparable (for TreeSet).

2. Why & when

Plenty of real data is naturally unique — a user's ID, an email address, a tag applied to a post — and the moment duplicates sneak in, downstream logic (counting distinct visitors, deduplicating tags) gets wrong answers unless something actively filters them out. You could try to enforce uniqueness with a List plus a manual contains() check before every add, but that's an O(n) check on every insert for ArrayList, and it's easy to forget in one code path and not another. Set bakes the uniqueness rule into the data structure itself — add() handles the check internally, in O(1) average time for HashSet. Reach for Set any time "is this already in the collection" matters more than "in what order did these arrive," and pick LinkedHashSet over plain HashSet specifically when you also want insertion order preserved for display purposes.

3. Core concept

Set<String> attendeeIds = new HashSet<>();
attendeeIds.add("user-42");
attendeeIds.add("user-7");
boolean added = attendeeIds.add("user-42"); // duplicate — returns false, set unchanged
System.out.println(added);              // false
System.out.println(attendeeIds.size());  // 2, not 3

The uniqueness check relies on equals()/hashCode(). For String, those are already well-defined by value, so two separately-constructed "user-42" strings are treated as the same element. For a custom class, Set membership only works correctly once equals() and hashCode() are overridden consistently.

5. Runnable example

Scenario: tracking unique attendee IDs for an event, growing from basic dedup to case-insensitive normalization to full set algebra between two events' attendee lists.

Level 1 — Basic

import java.util.*;

public class AttendeesBasic {
    public static void main(String[] args) {
        Set<String> attendees = new HashSet<>();
        attendees.add("user-42");
        attendees.add("user-7");
        boolean addedAgain = attendees.add("user-42"); // duplicate

        System.out.println("attendees: " + attendees.size());
        System.out.println("re-adding user-42 returned: " + addedAgain);
        System.out.println("contains user-7: " + attendees.contains("user-7"));
    }
}

How to run: java AttendeesBasic.java (JDK 17+).

Expected output (HashSet order isn't guaranteed, but size and boolean results are deterministic):

attendees: 2
re-adding user-42 returned: false
contains user-7: true

add("user-42") the second time doesn't throw and doesn't grow the set — it simply reports false because "user-42" is already present, judged by String.equals().

Level 2 — Intermediate

import java.util.*;

public class AttendeesNormalized {

    static Set<String> normalizedEmails = new HashSet<>();

    static boolean registerAttendee(String email) {
        return normalizedEmails.add(email.trim().toLowerCase());
    }

    public static void main(String[] args) {
        System.out.println(registerAttendee("Alice@Example.com"));   // true — new
        System.out.println(registerAttendee(" alice@example.com "));  // false — same person, different casing/whitespace
        System.out.println(registerAttendee("bob@example.com"));      // true — new

        System.out.println("unique attendees: " + normalizedEmails.size());
    }
}

How to run: java AttendeesNormalized.java.

Expected output:

true
false
true
unique attendees: 2

The real-world concern added: raw string equality isn't always the right notion of "duplicate." "Alice@Example.com" and " alice@example.com " are different String values but the same email address once trimmed and lowercased — so registerAttendee normalizes before inserting, letting HashSet's built-in equals() check catch the duplicate correctly.

Level 3 — Advanced

import java.util.*;

public class AttendeesSetAlgebra {
    public static void main(String[] args) {
        Set<String> conferenceDayOne = new HashSet<>(Set.of("alice", "bob", "carol", "dave"));
        Set<String> conferenceDayTwo = new HashSet<>(Set.of("carol", "dave", "erin"));

        // Union: everyone who attended either day. Copy first — addAll mutates in place.
        Set<String> union = new HashSet<>(conferenceDayOne);
        union.addAll(conferenceDayTwo);
        System.out.println("attended either day: " + new TreeSet<>(union)); // TreeSet just for stable printed order

        // Intersection: attended BOTH days.
        Set<String> intersection = new HashSet<>(conferenceDayOne);
        intersection.retainAll(conferenceDayTwo);
        System.out.println("attended both days: " + new TreeSet<>(intersection));

        // Difference: day-one-only attendees.
        Set<String> dayOneOnly = new HashSet<>(conferenceDayOne);
        dayOneOnly.removeAll(conferenceDayTwo);
        System.out.println("day one only: " + new TreeSet<>(dayOneOnly));
    }
}

How to run: java AttendeesSetAlgebra.java.

Expected output:

attended either day: [alice, bob, carol, dave, erin]
attended both days: [carol, dave]
day one only: [alice, bob]

This adds the production-flavored hard case: real set algebra across two independently collected Sets — union via addAll, intersection via retainAll, and difference via removeAll. Each operation is performed on a defensive copy (new HashSet<>(conferenceDayOne)) so the original conferenceDayOne/conferenceDayTwo sets are never mutated by computing the others — the same defensive-copy discipline from Collection applies here too.

6. Walkthrough

Tracing AttendeesSetAlgebra.main:

  1. Two HashSets are built from Set.of(...) literals — conferenceDayOne has four attendees, conferenceDayTwo has three, with "carol" and "dave" appearing in both.
  2. union starts as a fresh copy of conferenceDayOne; union.addAll(conferenceDayTwo) inserts every element of day two into it. Elements already present ("carol", "dave") are silently skipped by Set's own duplicate rule — addAll doesn't need special-case logic for that, it's automatic.
  3. intersection starts as a copy of conferenceDayOne, then retainAll(conferenceDayTwo) removes every element not also present in conferenceDayTwo, leaving only "carol" and "dave" — the classic intersection operation, expressed with a single method call.
  4. dayOneOnly starts as a copy of conferenceDayOne, then removeAll(conferenceDayTwo) strips out anything also in conferenceDayTwo, leaving "alice" and "bob" — the set difference.
  5. Each result is wrapped in new TreeSet<>(...) purely so the printed output has a stable, sorted order for this walkthrough — HashSet itself makes no ordering promise, so printing it directly could show elements in any order across JVM runs.

7. Gotchas & takeaways

Gotcha: HashSet makes no guarantee whatsoever about iteration order, and that order can even change between runs of the same program. If you need predictable order, use LinkedHashSet (insertion order) or TreeSet (sorted order) — never rely on HashSet's apparent order, even if it looks stable in casual testing.
  • Set<T> guarantees no duplicate elements; add() on an already-present value returns false and changes nothing.
  • Membership for HashSet/LinkedHashSet relies on equals()/hashCode() — custom classes must override both consistently for Set behavior to work correctly.
  • Normalize values (trim, lowercase, canonicalize) *before* inserting when "duplicate" means something looser than raw equals().
  • addAll, retainAll, and removeAll implement union, intersection, and difference respectively — always on a defensive copy if the originals must stay intact.
  • HashSet has undefined iteration order; reach for LinkedHashSet or TreeSet when order matters.
