Custom class loaders

← Back to checklist

1. What it is

A custom class loader is a subclass of java.lang.ClassLoader that supplies its own logic for finding and defining classes — typically by overriding findClass(String name) (the recommended extension point, which preserves the standard parent delegation behavior in loadClass()) to locate bytecode from some non-standard source: a network location, an encrypted file, a database blob, or a dynamically-generated array of bytes never backed by any file at all. Once bytecode bytes are obtained, defineClass(name, bytes, offset, length) — a protected final method inherited from ClassLoader — turns those raw bytes into an actual, usable Class object.

2. Why & when

Reach for a custom class loader whenever classes need to come from somewhere the standard classpath/module-path mechanism doesn't reach, or when you need genuine isolation between groups of classes — a plugin system where each plugin's classes should be loadable and unloadable independently (see class unloading), a hot-reloading development tool that needs to load a fresh version of a class without restarting the JVM, an application server hosting multiple independently-deployed applications that must not see each other's classes even if they happen to share class names, or a framework that generates bytecode at runtime (dynamic proxies, ORMs generating entity subclasses) and needs somewhere to load that generated bytecode from. The overwhelming majority of ordinary application code never needs a custom loader at all — this is a specialized tool for specific architectural needs, not a general-purpose technique.

3. Core concept

class CustomSourceClassLoader extends ClassLoader {
    @Override
    protected Class<?> findClass(String name) throws ClassNotFoundException {
        byte[] bytecode = loadBytecodeFromSomewhere(name); // your own logic: network, DB, decryption, etc.
        if (bytecode == null) throw new ClassNotFoundException(name);
        return defineClass(name, bytecode, 0, bytecode.length); // turn raw bytes into a real Class object
    }
}

Overriding findClass() (rather than loadClass()) means the inherited loadClass() still runs its normal parent-delegation logic first, and only calls your findClass() as the last resort, after every ancestor has failed — giving you custom loading behavior without accidentally breaking delegation's core JDK protection.

5. Runnable example

Scenario: a simple plugin system loading plugin classes from an arbitrary directory outside the normal classpath, growing from a basic custom loader reading .class files from a chosen directory, to loading multiple plugins each with their own isolated loader instance, to a version that supports reloading a plugin by discarding the old loader and creating a fresh one.

Level 1 — Basic

import java.io.*;
import java.nio.file.*;

public class DirectoryClassLoader extends ClassLoader {
    private final Path directory;

    public DirectoryClassLoader(Path directory, ClassLoader parent) {
        super(parent); // preserves normal parent delegation for everything except what THIS loader finds itself
        this.directory = directory;
    }

    @Override
    protected Class<?> findClass(String name) throws ClassNotFoundException {
        try {
            Path classFile = directory.resolve(name.replace('.', '/') + ".class");
            byte[] bytecode = Files.readAllBytes(classFile);
            return defineClass(name, bytecode, 0, bytecode.length);
        } catch (IOException e) {
            throw new ClassNotFoundException(name, e);
        }
    }

    public static void main(String[] args) throws Exception {
        // Assumes a compiled Plugin1.class already exists in the given directory (e.g. "plugins/").
        Path pluginDir = Path.of("plugins");
        DirectoryClassLoader loader = new DirectoryClassLoader(pluginDir, DirectoryClassLoader.class.getClassLoader());

        Class<?> pluginClass = loader.loadClass("Plugin1");
        Object instance = pluginClass.getDeclaredConstructor().newInstance();
        System.out.println("loaded and instantiated: " + instance);
        System.out.println("loader used: " + pluginClass.getClassLoader());
    }
}

How to run: first compile a trivial Plugin1.java (e.g. public class Plugin1 { public String toString() { return "Plugin1 instance"; } }) into a plugins/ subdirectory relative to where you run from (javac -d plugins Plugin1.java), then javac DirectoryClassLoader.java && java DirectoryClassLoader (JDK 17+).

Expected output shape:

loaded and instantiated: Plugin1 instance
loader used: DirectoryClassLoader@...

The class is loaded from an arbitrary directory that isn't on the normal application classpath at all — findClass() reads the raw .class file bytes directly from disk and hands them to defineClass(), demonstrating the minimal custom-loader pattern.

Level 2 — Intermediate

import java.nio.file.*;
import java.util.*;

public class MultiplePluginsIsolatedLoaders {
    public static void main(String[] args) throws Exception {
        // Two SEPARATE plugin directories, each loaded through its OWN DirectoryClassLoader instance --
        // giving each plugin its own isolated class namespace, even if they define same-named classes.
        Map<String, DirectoryClassLoader> loaders = new LinkedHashMap<>();
        loaders.put("pluginA", new DirectoryClassLoader(Path.of("plugins/a"), MultiplePluginsIsolatedLoaders.class.getClassLoader()));
        loaders.put("pluginB", new DirectoryClassLoader(Path.of("plugins/b"), MultiplePluginsIsolatedLoaders.class.getClassLoader()));

        for (var entry : loaders.entrySet()) {
            Class<?> pluginClass = entry.getValue().loadClass("PluginEntry"); // same class NAME in both directories
            Object instance = pluginClass.getDeclaredConstructor().newInstance();
            System.out.println(entry.getKey() + " -> " + instance + " (loader: " + pluginClass.getClassLoader() + ")");
        }
    }
}

How to run: compile two different PluginEntry.java files (with different toString() messages) into plugins/a/ and plugins/b/ respectively, then javac MultiplePluginsIsolatedLoaders.java && java MultiplePluginsIsolatedLoaders.

Expected output shape (two distinct instances, from two distinct PluginEntry classes, despite sharing a name):

pluginA -> PluginEntry from A (loader: DirectoryClassLoader@1b6d3586)
pluginB -> PluginEntry from B (loader: DirectoryClassLoader@4554617c)

The real-world concern added: two plugins with a same-named class (PluginEntry) are loaded through two separate DirectoryClassLoader instances, giving each an isolated namespace — exactly the pattern real plugin systems use to let independently-developed plugins avoid class-name collisions with each other.

Level 3 — Advanced

import java.nio.file.*;

public class ReloadablePlugin {
    static DirectoryClassLoader currentLoader;
    static Object currentInstance;

    static void loadOrReload(Path pluginDir, ClassLoader parent) throws Exception {
        // Discard the OLD loader entirely and create a FRESH one -- this is what makes
        // reloading possible: the JVM treats classes from the new loader as genuinely
        // DIFFERENT types from the old ones, so there's no stale static state or cached
        // class-level data left over from the previous version.
        currentLoader = new DirectoryClassLoader(pluginDir, parent);
        Class<?> pluginClass = currentLoader.loadClass("ReloadableEntry");
        currentInstance = pluginClass.getDeclaredConstructor().newInstance();
    }

    public static void main(String[] args) throws Exception {
        ClassLoader parent = ReloadablePlugin.class.getClassLoader();

        loadOrReload(Path.of("plugins/v1"), parent);
        System.out.println("version 1: " + currentInstance);
        DirectoryClassLoader firstLoader = currentLoader;

        // Simulate a code change: reload from a DIFFERENT directory representing "version 2"
        // of the same logical plugin.
        loadOrReload(Path.of("plugins/v2"), parent);
        System.out.println("version 2: " + currentInstance);

        System.out.println("first loader still referenced? " + (firstLoader != currentLoader));
        System.out.println("(the OLD loader and its classes are now eligible for garbage collection,");
        System.out.println(" since nothing in this program still references them)");
    }
}

How to run: compile a ReloadableEntry.java with one message into plugins/v1/ and a different message into plugins/v2/, then javac ReloadablePlugin.java && java ReloadablePlugin.

Expected output shape (two distinct outputs, confirming a genuine reload rather than a cached, stale version):

version 1: ReloadableEntry v1 message
version 2: ReloadableEntry v2 message
first loader still referenced? true
(the OLD loader and its classes are now eligible for garbage collection,
 since nothing in this program still references them)

This adds the production-flavored hard case: implementing hot-reload by *discarding the entire old class loader* and creating a fresh one, rather than trying to somehow "redefine" an already-loaded class in place (which the JVM generally does not allow for normal class loading — see class unloading for what actually makes the old classes eligible for garbage collection once nothing references their loader anymore). This is the standard technique behind development-time hot-reloading tools and certain plugin frameworks: a "new version" of a class is, from the JVM's perspective, an entirely different, unrelated type loaded by an entirely different loader instance — there's no in-place mutation of an existing class involved.

6. Walkthrough

Tracing ReloadablePlugin.main's two calls to loadOrReload:

  1. loadOrReload(Path.of("plugins/v1"), parent) constructs a brand-new DirectoryClassLoader (call it loader-1) and uses it to load ReloadableEntry from the v1 directory — this produces a Class object whose identity is tied specifically to loader-1, per the (name, loader) identity rule covered in the classloader hierarchy tutorial.
  2. currentInstance now holds an instance of "loader-1's ReloadableEntry," and main prints its message, confirming version 1's behavior.
  3. firstLoader = currentLoader captures a reference to loader-1 specifically so the later comparison can demonstrate that a *new* loader is genuinely created on reload, not the same one reused.
  4. loadOrReload(Path.of("plugins/v2"), parent) constructs an entirely new DirectoryClassLoader (loader-2) and loads ReloadableEntry from the v2 directory through it — even though the class name is identical to before, this produces a completely distinct Class object, tied to loader-2's identity, with no relationship to loader-1's earlier ReloadableEntry at the JVM level.
  5. currentInstance is reassigned to hold an instance of this new, "version 2" class — main's second print confirms the updated behavior took effect.
  6. firstLoader != currentLoader evaluates to true, confirming loader-1 and loader-2 are genuinely separate ClassLoader instances, and that currentLoader (the field actually used going forward) no longer refers to loader-1 at all.
  7. Because nothing in the running program retains any reference to loader-1 (it was only ever held by the now-reassigned currentLoader field, briefly captured in the local firstLoader variable which itself goes out of scope after main returns), loader-1 and every class it defined (ReloadableEntry version 1) become eligible for garbage collection — this is precisely the mechanism class unloading describes in more depth: an entire class loader and everything it defined can be reclaimed together, once nothing references the loader itself.

7. Gotchas & takeaways

Gotcha: overriding findClass() preserves normal parent delegation (your logic only runs after every ancestor has failed to find the class); overriding loadClass() directly bypasses that safety net entirely, and is almost always the wrong extension point unless you specifically intend to break delegation for a well-understood reason — accidentally overriding loadClass() instead of findClass() is one of the most common mistakes when first writing a custom class loader.
  • Extend ClassLoader and override findClass() to supply custom logic for locating bytecode, then call the inherited defineClass() to turn raw bytes into a usable Class object.
  • Custom loaders are the standard mechanism behind plugin systems (isolated per-plugin loaders, as shown), hot-reloading tools (discard the old loader, create a fresh one), application servers (isolating deployed applications from each other), and frameworks that generate bytecode dynamically.
  • Each distinct loader instance defines its own separate namespace — same-named classes loaded through different loader instances are genuinely different, incompatible types at the JVM level.
  • "Reloading" a class in practice means creating an entirely new loader and loading a fresh Class object through it, not mutating an existing, already-loaded class in place.
  • An entire class loader (and every class it defined) becomes eligible for garbage collection once nothing in the running program references it anymore — see class unloading for exactly what conditions must hold for this to actually happen.
