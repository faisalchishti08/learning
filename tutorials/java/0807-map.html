Map

1. What it is

Map<K, V> stores key-value pairs where each key maps to at most one value and keys are unique (inserting a value for an existing key replaces the old one). Unlike Collection, Map is a separate root interface — it doesn't extend Collection or Iterable directly, because iterating "a map" is ambiguous (keys? values? both?) until you ask for one of its three views: keySet(), values(), or entrySet(), each of which *is* a Collection/Iterable. Core operations are put(k, v), get(k), containsKey(k), remove(k), plus the newer convenience methods getOrDefault, putIfAbsent, computeIfAbsent, computeIfPresent, compute, and merge. HashMap is the standard general-purpose implementation.

2. Why & when

Any time data is naturally "look something up by a name" — a user's session by ID, a word's frequency by the word itself, a cache entry by its key — a Map is the direct fit; the alternative, a List<Pair<K,V>> scanned linearly for a matching key, is O(n) per lookup versus a HashMap's O(1) average. Reach for Map whenever the access pattern is "given a key, find the associated value" rather than "walk everything in order." Use HashMap by default; reach for LinkedHashMap when insertion or access order must be preserved for display or LRU purposes, and TreeMap when keys must stay sorted for range queries.

3. Core concept

Map<String, Integer> wordCounts = new HashMap<>();
wordCounts.put("the", 1);
wordCounts.merge("the", 1, Integer::sum); // "the" -> 2, using a combining function
wordCounts.merge("cat", 1, Integer::sum); // "cat" wasn't present, so it's inserted as 1

wordCounts.getOrDefault("dog", 0); // 0 — no exception, no null check needed
wordCounts.computeIfAbsent("bird", k -> 0); // inserts "bird" -> 0 only if absent

for (Map.Entry<String, Integer> entry : wordCounts.entrySet()) {
    System.out.println(entry.getKey() + " -> " + entry.getValue());
}

merge is the idiomatic way to express "increment a counter, or initialize it if this is the first time" in a single call, without a manual containsKey/get/put sequence.

5. Runnable example

Scenario: a word-frequency counter, growing from basic counting to grouped-by-length buckets to a sorted top-N report.

Level 1 — Basic

import java.util.*;

public class WordCountBasic {
    public static void main(String[] args) {
        String[] words = {"the", "cat", "sat", "on", "the", "mat", "the", "cat"};
        Map<String, Integer> counts = new HashMap<>();

        for (String word : words) {
            counts.merge(word, 1, Integer::sum);
        }

        for (Map.Entry<String, Integer> entry : counts.entrySet()) {
            System.out.println(entry.getKey() + ": " + entry.getValue());
        }
    }
}

How to run: java WordCountBasic.java (JDK 17+).

Expected output (HashMap iteration order isn't guaranteed — some run may print these lines in a different order):

the: 3
cat: 2
sat: 1
on: 1
mat: 1

merge(word, 1, Integer::sum) is the whole counting logic: if word isn't a key yet, it's inserted with value 1; if it already is, the combining function Integer::sum adds 1 to the existing count.

Level 2 — Intermediate

import java.util.*;

public class WordCountGrouped {
    public static void main(String[] args) {
        String[] words = {"the", "cat", "sat", "on", "the", "mat", "an", "at"};

        // Group words by length: Map<Integer, List<String>>, built with computeIfAbsent.
        Map<Integer, List<String>> byLength = new HashMap<>();
        for (String word : words) {
            byLength.computeIfAbsent(word.length(), len -> new ArrayList<>()).add(word);
        }

        System.out.println("2-letter words: " + byLength.getOrDefault(2, List.of()));
        System.out.println("3-letter words: " + byLength.getOrDefault(3, List.of()));
        System.out.println("5-letter words (none exist): " + byLength.getOrDefault(5, List.of()));
    }
}

How to run: java WordCountGrouped.java.

Expected output:

2-letter words: [on, an, at]
3-letter words: [the, cat, sat, the, mat]
5-letter words (none exist): []

The real-world concern added: grouping, not just counting — computeIfAbsent(key, k -> new ArrayList<>()) returns the existing list for key if present, or creates, inserts, and returns a new empty list if not, in one call. Combined with getOrDefault, a query for a length that never occurred (5) safely returns an empty list instead of null or a NoSuchElementException.

Level 3 — Advanced

import java.util.*;
import java.util.stream.*;

public class WordCountTopN {
    public static void main(String[] args) {
        String[] words = {"the", "cat", "sat", "on", "the", "mat", "the", "cat", "purred"};
        Map<String, Integer> counts = new HashMap<>();
        for (String word : words) {
            counts.merge(word, 1, Integer::sum);
        }

        // Sort entrySet by value descending to build a "top N" report — a Map itself has no order,
        // but its entrySet can be copied into a sortable List.
        List<Map.Entry<String, Integer>> ranked = new ArrayList<>(counts.entrySet());
        ranked.sort(Map.Entry.<String, Integer>comparingByValue().reversed());

        System.out.println("top 3 words:");
        ranked.stream().limit(3).forEach(e -> System.out.println("  " + e.getKey() + ": " + e.getValue()));

        // computeIfPresent: only touch a key that's ALREADY there, e.g. to cap a count.
        counts.computeIfPresent("the", (k, v) -> Math.min(v, 2));
        System.out.println("'the' capped at 2: " + counts.get("the"));

        // compute: unconditional read-modify-write, key may or may not exist beforehand.
        counts.compute("newword", (k, v) -> (v == null) ? 1 : v + 1);
        System.out.println("'newword' after compute: " + counts.get("newword"));
    }
}

How to run: java WordCountTopN.java.

Expected output:

top 3 words:
  the: 3
  cat: 2
  purred: 1
'the' capped at 2: 2
'newword' after compute: 1

This adds the production-flavored hard case: a Map has no inherent order, so a "top N" report requires copying entrySet() into a List and sorting that copy — Map.Entry.comparingByValue() is the idiomatic comparator for this. computeIfPresent and compute round out the family of atomic-style update methods: computeIfPresent only fires its function when the key already exists (useful for capping or transforming existing values without accidentally inserting new ones), while compute always runs, receiving null for a missing key so the same call can express both "insert if absent" and "update if present."

6. Walkthrough

Tracing WordCountTopN.main:

  1. The for loop over words calls counts.merge(word, 1, Integer::sum) for each entry, building up frequency counts exactly as in Level 1 — final state: {the=3, cat=2, sat=1, on=1, mat=1, purred=1} (order not guaranteed by HashMap).
  2. ranked is built by copying counts.entrySet() — a live view over the map — into a fresh ArrayList<Map.Entry<String, Integer>>, decoupling the sort from the map's own (undefined) iteration order.
  3. ranked.sort(Map.Entry.<String, Integer>comparingByValue().reversed()) sorts that list by each entry's value, descending — the (3) first, then cat (2), then the four remaining words tied at 1, in whatever relative order they landed in during the copy.
  4. ranked.stream().limit(3) takes the first three entries of the now-sorted list and prints them — the top 3 report.
  5. counts.computeIfPresent("the", (k, v) -> Math.min(v, 2)) looks up "the" (present, value 3), runs the function, and stores its result (2) back — capping the count. Because "the" was present, the function ran; had the key been absent, computeIfPresent would have done nothing at all.
  6. counts.compute("newword", (k, v) -> (v == null) ? 1 : v + 1) looks up "newword" (absent, so v is null inside the lambda), takes the v == null branch, and stores 1 — demonstrating compute's unconditional insert-or-update behavior in a single call.

7. Gotchas & takeaways

Gotcha: HashMap (like HashSet) makes no guarantee about iteration order — printing counts directly, or iterating entrySet() without sorting, can show entries in any order, and that order isn't guaranteed to stay stable even across runs of the same program. Any report requiring a specific order (alphabetical, by value, by insertion) must sort explicitly or use LinkedHashMap/TreeMap instead.
  • Map<K, V> associates unique keys with values; it does not extend Collection — use keySet(), values(), or entrySet() to iterate.
  • merge, computeIfAbsent, computeIfPresent, and compute replace manual containsKey/get/put sequences with single atomic-style calls.
  • getOrDefault avoids null checks for missing keys; it never inserts, unlike computeIfAbsent.
  • To sort a map's contents, copy entrySet() into a List and sort that — the map itself has no order to rely on unless it's a LinkedHashMap or TreeMap.
  • Reach for HashMap by default; switch to LinkedHashMap for order-preserving iteration, or TreeMap when the keys themselves must stay sorted.
