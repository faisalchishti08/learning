List

1. What it is

List<T> is the Collection subtype that adds two guarantees neither Set nor Queue make: elements keep a stable, positional order (whatever order you inserted them, that's the order you get back), and duplicates are allowed. On top of the Collection methods, List adds index-based operations — get(int index), set(int index, T value), add(int index, T value), remove(int index), and indexOf(Object). The two workhorse implementations are ArrayList (backed by a resizable array, fast random access) and LinkedList (backed by a doubly-linked chain of nodes, fast insertion at the ends).

2. Why & when

Plenty of real data is inherently ordered and allows repeats — a task list can have two tasks with the same name, and their order (which comes first) matters to the user. A Set would silently collapse the duplicate task names into one; a Map would need an artificial key just to preserve order. List exists precisely for this shape of data: sequential, position-addressable, duplicate-tolerant. Reach for List whenever "the Nth item" is a meaningful question to ask, or when the same value can legitimately appear more than once. Use ArrayList by default — it's backed by an array and gives O(1) get(index) — and only switch to LinkedList when you specifically need cheap insertion/removal at both ends without random-access reads.

3. Core concept

List<String> todos = new ArrayList<>();
todos.add("Buy milk");        // index 0
todos.add("Walk the dog");    // index 1
todos.add("Buy milk");        // index 2 — duplicates are fine
todos.get(1);                 // "Walk the dog" — direct index access
todos.set(0, "Buy oat milk"); // replace index 0 in place
todos.add(1, "Call mom");     // insert at index 1, shifting the rest right

After the calls above, todos is ["Buy oat milk", "Call mom", "Walk the dog", "Buy milk"] — insertion order preserved, duplicates ("Buy milk" appearing under two different values now) tolerated, every element addressable by its position.

5. Runnable example

Scenario: a simple task-list app, growing from basic add/get to safe editing to pagination.

Level 1 — Basic

import java.util.*;

public class TodoListBasic {
    public static void main(String[] args) {
        List<String> todos = new ArrayList<>();
        todos.add("Buy milk");
        todos.add("Walk the dog");
        todos.add("Buy milk"); // duplicate task, allowed

        System.out.println("all todos: " + todos);
        System.out.println("first task: " + todos.get(0));
        System.out.println("total: " + todos.size());
    }
}

How to run: java TodoListBasic.java (JDK 17+).

Expected output:

all todos: [Buy milk, Walk the dog, Buy milk]
first task: Buy milk
total: 3

get(0) returns the element at position 0 in O(1) time because ArrayList is backed by a real array — no traversal needed. Notice "Buy milk" appears twice: List never rejects a duplicate the way Set would.

Level 2 — Intermediate

import java.util.*;

public class TodoListEditable {
    public static void main(String[] args) {
        List<String> todos = new ArrayList<>(List.of("Buy milk", "Walk the dog", "Call mom"));

        todos.set(0, "Buy oat milk");     // replace index 0 in place
        todos.add(1, "Reply to email");   // insert at index 1, shifting the rest right
        todos.remove("Call mom");         // remove by value

        int index = todos.indexOf("Walk the dog");
        System.out.println("todos now: " + todos);
        System.out.println("'Walk the dog' is now at index: " + index);

        try {
            todos.get(99);
        } catch (IndexOutOfBoundsException e) {
            System.out.println("caught: " + e.getMessage());
        }
    }
}

How to run: java TodoListEditable.java.

Expected output:

todos now: [Buy oat milk, Reply to email, Walk the dog]
'Walk the dog' is now at index: 2
caught: Index 99 out of bounds for length 3

The real-world concern added: editing in place (set), inserting mid-list (add(index, value), which shifts every later element right by one), and handling the very real case of an out-of-range index — get on an invalid index throws IndexOutOfBoundsException rather than returning null, so production code that indexes into a List needs to validate bounds first or catch this.

Level 3 — Advanced

import java.util.*;

public class TodoListPaginated {
    public static void main(String[] args) {
        List<String> todos = new ArrayList<>(List.of(
            "Buy milk", "Walk the dog", "Call mom", "Pay rent",
            "Read book", "Clean garage", "Write report"
        ));

        // Sort with a Comparator: shortest tasks first.
        todos.sort(Comparator.comparingInt(String::length));
        System.out.println("sorted by length: " + todos);

        // Pagination via subList — a LIVE VIEW backed by the same list, not a copy.
        int pageSize = 3;
        for (int page = 0; page * pageSize < todos.size(); page++) {
            int from = page * pageSize;
            int to = Math.min(from + pageSize, todos.size());
            List<String> pageView = todos.subList(from, to);
            System.out.println("page " + page + ": " + pageView);
        }

        // Gotcha demonstration: mutating the subList view mutates the backing list too.
        List<String> firstPage = todos.subList(0, pageSize);
        firstPage.clear();
        System.out.println("after clearing first page view, full list: " + todos);
    }
}

How to run: java TodoListPaginated.java.

Expected output:

sorted by length: [Call mom, Pay rent, Buy milk, Read book, Write report, Walk the dog, Clean garage]
page 0: [Call mom, Pay rent, Buy milk]
page 1: [Read book, Write report, Walk the dog]
page 2: [Clean garage]
after clearing first page view, full list: [Read book, Write report, Walk the dog, Clean garage]

This adds the production-flavored hard case: subList(from, to) returns a live view, not a copy — mutating the view (firstPage.clear()) mutates the backing todos list too, immediately dropping its first three elements. This is a common footgun: pagination code that treats a subList result as a safe, independent snapshot can accidentally corrupt the original list. If an independent copy is needed, wrap it explicitly: new ArrayList<>(todos.subList(from, to)).

6. Walkthrough

Tracing TodoListPaginated.main:

  1. todos starts with seven tasks in the order written.
  2. todos.sort(Comparator.comparingInt(String::length)) reorders the list in place by string length, shortest first; ties (like "Buy milk" and "Call mom", both 8 characters) keep their relative order because List.sort is a stable sort.
  3. The for loop computes page boundaries and calls todos.subList(from, to) for each page. subList doesn't copy elements — it returns a List object that reads and writes directly into the same backing array as todos, just windowed to [from, to).
  4. Three pages print, matching the sorted order sliced into groups of three (and a partial final group of one).
  5. firstPage = todos.subList(0, pageSize) grabs a second, independent view over the current first three elements. Calling firstPage.clear() removes those three elements from the underlying todos list itself, because the view and the backing list share the same storage.
  6. Printing todos afterward shows only four elements remaining — the first three were removed via the view, even though the code never called todos.remove(...) directly.

7. Gotchas & takeaways

Gotcha: subList(from, to) returns a live, writable view backed by the original list — not a snapshot. Structural changes made through the view (or, in the other direction, directly on the backing list while a view is held) can throw ConcurrentModificationException on the view, or silently corrupt data if you assumed independence. Copy explicitly (new ArrayList<>(list.subList(a, b))) whenever you need an isolated snapshot.
  • List<T> preserves insertion order and allows duplicate elements — the two properties Set does not offer.
  • ArrayList gives O(1) indexed get/set because it's backed by a real array; use it by default.
  • LinkedList gives O(1) insertion/removal at either end but O(n) indexed access — pick it only when that tradeoff actually matches your access pattern.
  • add(index, value) and remove(index) are O(n) on ArrayList because they shift every later element.
  • subList is a live view sharing storage with its backing list — mutate it deliberately, or copy it if you need isolation.
