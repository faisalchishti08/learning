Heap

1. What it is

The heap is the runtime data area where the JVM allocates every object and array — everything created with new, every autoboxed wrapper, every string built at runtime, every array. It's shared by all threads in the JVM (unlike each thread's own private stack), and it's the memory region the garbage collector manages: objects live there until nothing reachable references them anymore, at which point the collector reclaims their space. Modern JVM garbage collectors typically subdivide the heap further (a "young generation" for newly-created, usually short-lived objects, and an "old generation" for objects that have survived several collection cycles), but from the Java language's perspective, the heap is simply "where all your objects live."

2. Why & when

Understanding the heap matters for reasoning about memory usage, garbage collection behavior, and OutOfMemoryError: Java heap space — the specific error that occurs when the JVM cannot allocate a new object because the heap (even after attempting a full garbage collection) has no room left, distinct from other, differently-caused OutOfMemoryError variants (like running out of metaspace, which is a completely separate memory region with its own limits). You'll size heap limits explicitly (-Xms for initial size, -Xmx for maximum size) for production JVMs to control how much memory the process can consume, and you'll reason about heap contents specifically when diagnosing memory leaks (objects unexpectedly still reachable, preventing collection) or excessive garbage collection pauses (too many short-lived allocations, or a heap sized too small for the actual working set of live objects).

3. Core concept

// Every one of these allocates memory on the HEAP, regardless of where the code runs:
Object obj = new Object();          // heap
int[] array = new int[1000];         // heap (arrays are objects too)
String s = "built at runtime " + n;   // heap (unlike compile-time-constant string literals, which are interned)

// Only PRIMITIVE LOCAL VARIABLES and object REFERENCES themselves live on the stack --
// what a reference POINTS TO always lives on the heap.

The distinction that matters: a local variable of a primitive type, or a reference variable itself, lives in a stack frame; the *object* a reference points to always lives on the heap, shared across all threads.

5. Runnable example

Scenario: observing heap allocation and exhaustion directly, growing from a basic allocation demonstration with reported heap statistics, to triggering and catching OutOfMemoryError: Java heap space deliberately, to demonstrating that clearing references lets the heap reclaim space via garbage collection.

Level 1 — Basic

public class ObservingHeapUsage {
    public static void main(String[] args) {
        Runtime runtime = Runtime.getRuntime();
        System.out.println("max heap: " + (runtime.maxMemory() / 1_000_000) + "MB");
        System.out.println("used before allocation: " + usedMemoryMB(runtime) + "MB");

        int[][] arrays = new int[100][];
        for (int i = 0; i < arrays.length; i++) {
            arrays[i] = new int[100_000]; // each allocated on the HEAP
        }

        System.out.println("used after allocating 100 arrays of 100,000 ints: " + usedMemoryMB(runtime) + "MB");
        System.out.println("arrays[0].length = " + arrays[0].length);
    }

    static long usedMemoryMB(Runtime runtime) {
        return (runtime.totalMemory() - runtime.freeMemory()) / 1_000_000;
    }
}

How to run: java ObservingHeapUsage.java (JDK 17+).

Expected output shape (exact numbers vary by machine and default heap sizing):

max heap: 4096MB
used before allocation: 8MB
used after allocating 100 arrays of 100,000 ints: 46MB
arrays[0].length = 100000

Every new int[100_000] call allocates on the shared heap, and Runtime's memory statistics make this growth directly observable.

Level 2 — Intermediate

import java.util.*;

public class ExhaustingTheHeap {
    public static void main(String[] args) {
        List<int[]> holder = new ArrayList<>(); // keeps every array REACHABLE -- prevents collection
        long allocatedMB = 0;
        try {
            while (true) {
                holder.add(new int[1_000_000]); // ~4MB per array (int = 4 bytes)
                allocatedMB += 4;
            }
        } catch (OutOfMemoryError e) {
            System.out.println("caught OutOfMemoryError after allocating roughly " + allocatedMB + "MB");
            System.out.println("message: " + e.getMessage());
        }
    }
}

How to run: java -Xmx256m ExhaustingTheHeap.java (explicitly capping the heap to 256MB makes this run quickly and deterministically; JDK 17+).

Expected output shape (exact allocated amount depends on the capped heap size and JVM overhead):

caught OutOfMemoryError after allocating roughly 240MB
message: Java heap space

The real-world concern added: deliberately exhausting the heap by keeping every allocated array reachable (via holder, preventing garbage collection from reclaiming any of them) — the exact error message "Java heap space" specifically identifies this as heap exhaustion, distinct from other OutOfMemoryError causes like exhausting metaspace or hitting a thread-creation limit.

Level 3 — Advanced

import java.util.*;
import java.lang.ref.*;

public class ClearingReferencesReclaimsHeap {
    public static void main(String[] args) throws InterruptedException {
        Runtime runtime = Runtime.getRuntime();
        List<WeakReference<int[]>> tracked = new ArrayList<>();
        List<int[]> holder = new ArrayList<>(); // STRONG references, keeping arrays alive for now

        for (int i = 0; i < 50; i++) {
            int[] arr = new int[1_000_000]; // ~4MB each
            holder.add(arr);
            tracked.add(new WeakReference<>(arr));
        }
        System.out.println("used with 50 arrays strongly held: " + usedMemoryMB(runtime) + "MB");

        holder.clear(); // drop the ONLY strong references -- arrays become eligible for collection
        System.gc();
        Thread.sleep(200);

        long stillAlive = tracked.stream().filter(ref -> ref.get() != null).count();
        System.out.println("arrays still alive after clearing holder and requesting GC: " + stillAlive + " (expected 0)");
        System.out.println("used after clearing and GC: " + usedMemoryMB(runtime) + "MB (should have dropped significantly)");
    }

    static long usedMemoryMB(Runtime runtime) {
        return (runtime.totalMemory() - runtime.freeMemory()) / 1_000_000;
    }
}

How to run: java ClearingReferencesReclaimsHeap.java (JDK 17+).

Expected output shape (System.gc() is best-effort, but reliably effective for this simple, clearly-unreachable-after-clearing scenario):

used with 50 arrays strongly held: 214MB
arrays still alive after clearing holder and requesting GC: 0 (expected 0)
used after clearing and GC: 12MB (should have dropped significantly)

This adds the production-flavored hard case: using WeakReference to directly observe that once the *only* strong references (holder) are cleared, the garbage collector genuinely reclaims the heap space those arrays occupied — demonstrating the fundamental heap lifecycle: allocate while reachable, reclaim once unreachable, with Runtime's memory statistics making the actual space recovery directly measurable.

6. Walkthrough

Tracing ClearingReferencesReclaimsHeap.main:

  1. The loop allocates 50 arrays of 1,000,000 ints each (roughly 4MB apiece, accounting for JVM array overhead) and adds each one to holder (a strong reference, which by itself keeps every array reachable from a GC root, since holder is a local variable of main, which is itself on the call stack for the whole duration of the method) as well as recording a WeakReference to each in tracked (which, being weak, does *not* by itself keep anything alive).
  2. usedMemoryMB reports the heap usage growing to reflect roughly 200MB of allocated array data, all currently reachable via holder.
  3. holder.clear() removes every element from the list — since holder was the *only* strong reference keeping these 50 arrays reachable (the WeakReferences in tracked don't count), all 50 arrays instantly become eligible for garbage collection, even though nothing has explicitly "deleted" them (Java has no manual delete/free; eligibility for collection is purely a function of reachability).
  4. System.gc() requests a collection cycle; the garbage collector traces reachability from GC roots (including main's remaining local variables, now only tracked and runtime, neither of which strongly reference the arrays) and finds all 50 arrays are unreachable, reclaiming their heap space.
  5. The tracked.stream().filter(ref -> ref.get() != null).count() check confirms this directly: every WeakReference.get() call now returns null, since the collector has cleared each weak reference upon reclaiming its referent, confirming stillAlive is 0.
  6. The final usedMemoryMB reading reflects the heap usage dropping back down close to its pre-allocation baseline, providing direct, measured evidence that the memory really was reclaimed, not merely marked as "logically gone" without any actual space recovery.

7. Gotchas & takeaways

Gotcha: System.gc() is only a *request*, not a guarantee — the JVM specification explicitly permits an implementation to ignore it entirely. In practice, most JVMs do perform a collection when asked, especially for small, clearly-reclaimable heaps like these examples, but production code should never rely on System.gc() actually running at a specific moment; let the collector run on its own schedule based on actual memory pressure instead.
  • The heap is the single, shared memory region where every object and array lives, managed by the garbage collector, and is distinct from each thread's own private stack, which holds only primitive locals and reference variables (not the objects they point to).
  • -Xmx sets the maximum heap size; exceeding it (with no reclaimable garbage available) produces OutOfMemoryError: Java heap space, a distinct error from other OutOfMemoryError causes tied to different memory regions.
  • An object becomes eligible for garbage collection the instant nothing reachable from any GC root still references it — there's no manual deallocation in Java; reachability alone determines collectibility.
  • WeakReference is a useful diagnostic and architectural tool for observing (or intentionally not preventing) an object's collectibility, as demonstrated throughout this and related tutorials.
  • See method area / metaspace for the separate memory region holding class metadata (as opposed to object instances), which has its own independent size limits and its own distinct OutOfMemoryError cause.
