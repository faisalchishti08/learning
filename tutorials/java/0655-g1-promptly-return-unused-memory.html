G1 promptly return unused memory

← Back to checklist

1. What it is

G1 promptly return unused memory, added in Java 12 (JEP 346), makes the G1 garbage collector give committed-but-unused heap memory back to the operating system during idle periods, not just during full GCs. Before this change, G1 only returned memory to the OS when a full garbage collection ran — and G1 is specifically designed to *avoid* full GCs whenever possible, so an application whose memory usage spiked once and then dropped could sit there holding onto all that committed memory indefinitely, never giving it back, even while completely idle. From Java 12 onward, G1 periodically checks heap usage during idle periods and, if the JVM has been idle for a configurable amount of time (-XX:G1PeriodicGCInterval), can trigger a concurrent cycle specifically to identify and uncommit unused regions back to the OS.

2. Why & when

This matters a lot for memory-density-sensitive environments: containers, cloud VMs billed by allocated memory, and any host running many JVMs side by side. Without prompt memory return, a JVM that briefly needed a large heap (say, during a startup burst or an occasional batch job) would keep that memory "reserved" from the OS's perspective forever, starving other processes on the same machine even though the JVM itself was using almost none of it. This was a well-known pain point for G1 versus other collectors (like the older CMS, which had this idle-return behavior already) and a real obstacle to running G1 efficiently in memory-constrained containers. Reach for the relevant flags whenever you're running G1 in a container or shared host and want the JVM's memory footprint to shrink back down after a usage spike, instead of staying pinned at its peak.

3. Core concept

# Enable periodic GC to reclaim idle memory (disabled by default: interval 0 = off)
java -XX:+UseG1GC -XX:G1PeriodicGCInterval=60000 MyApp
# Above: if the JVM is idle for 60 seconds, G1 may trigger a GC cycle
# specifically to find and uncommit unused heap regions.

# Control how "idle" is measured and how aggressive the returned GC is:
java -XX:+UseG1GC \
     -XX:G1PeriodicGCInterval=60000 \
     -XX:G1PeriodicGCLoadThreshold=0 \
     MyApp

G1PeriodicGCInterval (milliseconds, default 0 = disabled) is the main knob — set it to a nonzero value to opt in to periodic idle-memory reclamation; G1PeriodicGCLoadThreshold controls how low CPU load must be for a periodic GC to be considered worth triggering.

5. Runnable example

Scenario: a program that briefly allocates a large amount of memory (simulating a startup burst or batch job) and then goes idle — first observing the default behavior with periodic GC disabled, then enabling it and watching memory get returned, then building a small monitor that reports the JVM's committed memory over time to make the effect visible from within the program itself.

Level 1 — Basic

// File: MemorySpike.java
import java.util.ArrayList;
import java.util.List;

public class MemorySpike {
    public static void main(String[] args) throws InterruptedException {
        List<byte[]> temp = new ArrayList<>();
        for (int i = 0; i < 2000; i++) {
            temp.add(new byte[1024 * 1024]); // 1 MB each => ~2 GB peak
        }
        System.out.println("Peak allocation done: " + temp.size() + " MB held.");

        temp.clear();       // drop all references
        temp = null;        // make it eligible for collection
        System.out.println("Cleared references. Now sleeping (idle)...");

        Thread.sleep(90_000); // stay idle for 90 seconds
        System.out.println("Done sleeping.");
    }
}

How to run (default, periodic GC disabled): java -Xmx3g MemorySpike.java

While this runs, check the process's resident memory in another terminal (e.g. ps -o rss -p <pid> on macOS/Linux). Expected: RSS stays near its peak throughout the 90-second sleep — the JVM doesn't proactively return the memory.

Level 2 — Intermediate

How to run with periodic idle GC enabled:

java -Xmx3g -XX:+UseG1GC -XX:G1PeriodicGCInterval=10000 -Xlog:gc MemorySpike.java

Expected console output is the same program messages, but the GC log now shows a periodic GC cycle firing roughly 10 seconds after the program goes idle:

Peak allocation done: 2000 MB held.
Cleared references. Now sleeping (idle)...
[92.341s][info][gc] GC(5) Pause Young (Concurrent Start) (G1 Periodic Collection) 2048M->12M(3072M) 8.234ms
Done sleeping.

Checking RSS in another terminal this time shows it drop noticeably a few seconds into the sleep, once G1PeriodicGCInterval (10 seconds of idleness) elapses and the periodic collection runs — proof the memory is being returned to the OS during the idle period, not just held.

Level 3 — Advanced

// File: MemoryMonitor.java
import java.lang.management.ManagementFactory;
import java.lang.management.MemoryMXBean;
import java.lang.management.MemoryUsage;
import java.util.ArrayList;
import java.util.List;

public class MemoryMonitor {
    static void report(String label) {
        MemoryMXBean mem = ManagementFactory.getMemoryMXBean();
        MemoryUsage heap = mem.getHeapMemoryUsage();
        System.out.printf("%-25s committed=%dMB used=%dMB%n",
            label, heap.getCommitted() / (1024 * 1024), heap.getUsed() / (1024 * 1024));
    }

    public static void main(String[] args) throws InterruptedException {
        report("startup");

        List<byte[]> temp = new ArrayList<>();
        for (int i = 0; i < 1500; i++) {
            temp.add(new byte[1024 * 1024]);
        }
        report("after spike");

        temp.clear();
        temp = null;
        report("after clear (refs dropped)");

        for (int i = 0; i < 6; i++) {
            Thread.sleep(15_000);
            report("idle +" + ((i + 1) * 15) + "s");
        }
    }
}

How to run: java -Xmx2g -XX:+UseG1GC -XX:G1PeriodicGCInterval=10000 MemoryMonitor.java

Expected output (values approximate):

startup                  committed=64MB used=8MB
after spike               committed=1600MB used=1536MB
after clear (refs dropped) committed=1600MB used=45MB
idle +15s                 committed=1600MB used=40MB
idle +30s                 committed=180MB used=38MB
idle +45s                 committed=175MB used=37MB
idle +60s                 committed=175MB used=36MB
idle +75s                 committed=175MB used=36MB
idle +90s                 committed=175MB used=35MB

Level 3 uses MemoryMXBean to read the heap's committed size directly from inside the JVM — used drops immediately after clearing references (the objects become garbage-collectible), but committed (the memory actually reserved from the OS) stays high until enough idle time passes for G1PeriodicGCInterval to trigger a periodic collection that uncommits the now-unused regions, visible as the sharp drop between "idle +15s" and "idle +30s".

6. Walkthrough

  1. main calls report("startup") first. ManagementFactory.getMemoryMXBean().getHeapMemoryUsage() reads the JVM's current heap statistics; at startup both committed and used are small since almost nothing has been allocated yet.
  2. The loop allocates 1500 one-megabyte byte[] objects, each added to temp, keeping all of them reachable — the heap grows to accommodate this, and G1 commits more regions from the OS as needed. report("after spike") shows both committed and used near their peak (~1.5 GB).
  3. temp.clear() removes every reference from the list, and temp = null drops the list itself — none of those 1500 byte arrays are reachable from any GC root anymore, making them garbage. But no collection has necessarily run yet at this exact point, so report("after clear") can still show used reflecting some not-yet-collected garbage, though typically a young collection will have already reclaimed most of it, dropping used sharply while committed stays unchanged — the regions are still reserved from the OS even though nothing meaningful occupies them.
  4. The final loop calls Thread.sleep(15_000) six times, calling report(...) after each. During these sleeps, the application makes no allocations at all — this is exactly the "idle" condition G1PeriodicGCInterval is designed to detect.
  5. Once 10 seconds of idleness have elapsed (the configured G1PeriodicGCInterval), G1's periodic-GC mechanism triggers a GC cycle specifically to check whether committed regions are actually needed. Finding large swaths of committed-but-empty regions, it uncommits them, handing that memory back to the operating system.
  6. This shows up in the output as the sharp drop in committed between the "idle +15s" and "idle +30s" checkpoints — used was already low, but committed only falls once the periodic GC actually runs and performs the uncommit.
  7. Subsequent idle checkpoints show committed staying near its new, lower value, since there's nothing new to reclaim — the JVM's OS-visible memory footprint has settled down to roughly what the application is actually using, rather than remaining pinned at its historical peak.
startup: committed=64MB
   │ allocate 1.5GB
after spike: committed=1600MB, used=1536MB
   │ drop references
after clear: committed=1600MB (unchanged!), used=45MB (dropped)
   │ idle 10s+ → G1PeriodicGCInterval fires → uncommit unused regions
idle +30s: committed=180MB (dropped!), used=38MB

7. Gotchas & takeaways

G1PeriodicGCInterval defaults to 0, meaning periodic idle-memory return is OFF by default — you must explicitly opt in with a nonzero millisecond value. Simply upgrading to Java 12 does not automatically shrink idle JVMs' memory footprint; the JEP added the *capability*, not a new default behavior.
  • committed memory (reserved from the OS) and used memory (actually holding live objects) are different numbers — dropping references shrinks used quickly but doesn't shrink committed without a GC cycle that specifically uncommits regions.
  • G1PeriodicGCInterval is measured from the last GC activity — the JVM must be genuinely idle (no allocation pressure triggering other GCs) for that long before a periodic cycle fires.
  • G1PeriodicGCLoadThreshold lets you require low CPU load before triggering a periodic GC, avoiding extra GC work when the machine is already busy with other things.
  • This feature is especially valuable in containerized/cloud deployments where committed-but-idle memory directly costs money or starves co-located processes.
  • Use MemoryMXBean.getHeapMemoryUsage().getCommitted() (or OS-level RSS monitoring) to verify the effect — total heap usage graphs from application-level metrics alone won't show committed-vs-used distinctions.
