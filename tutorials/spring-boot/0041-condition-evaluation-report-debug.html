Condition evaluation report (debug)

← Back to checklist

1. What it is

The condition evaluation report is a diagnostic log block printed by Spring Boot at startup when you enable debug mode. It lists every auto-configuration class that was considered and, for each one, whether it was applied or not — and the exact reason either way.

Enable it with any of these:

# application.properties
debug=true
# command line
java -jar myapp.jar --debug
# application.yml
debug: true

The report appears in the console under the heading CONDITIONS EVALUATION REPORT and has two sections:

  • Positive matches — auto-configurations that were applied (conditions passed).
  • Negative matches — auto-configurations that were skipped (conditions failed), with the failing condition shown.

2. Why & when

The most common Spring Boot debugging question is: *"why does my app have / not have this bean?"* The condition evaluation report answers it definitively and directly — no source-reading, no guessing.

Use it when:

  • An expected auto-configured bean is missing (check "Negative matches" to see which condition failed).
  • An unexpected bean appeared in the context (check "Positive matches" to see which auto-config added it).
  • You're writing a custom auto-configuration and want to verify it applies under the right conditions.
  • You upgrade Spring Boot versions and want to see if any auto-configuration changed.

Turn it off in production (it is verbose) and use it as a temporary debug tool.

3. Core concept

Analogy: the condition evaluation report is like a flight manifest at the gate. Before the plane departs (the app starts), the gate agent (Spring's ConditionEvaluator) checks every passenger (auto-configuration class) against the passenger list (conditions). The manifest records who boarded (positive matches) and who was turned away and why (negative matches).

The report is generated by ConditionEvaluationReportLogger, which:

  1. Listens on Spring's ApplicationStartingEvent and ConditionEvaluationReport internally.
  2. Accumulates condition results as each auto-configuration is processed.
  3. Prints the full report as an INFO log after context refresh.

A negative match entry looks like:

DataSourceAutoConfiguration:
   Did not match:
      - @ConditionalOnClass did not find required class 'javax.sql.DataSource' (OnClassCondition)

A positive match entry looks like:

JacksonAutoConfiguration matched:
   - @ConditionalOnClass found required class 'com.fasterxml.jackson.databind.ObjectMapper' (OnClassCondition)

4. Diagram

The --debug report prints two sections: green positive matches (beans applied) and red negative matches (beans skipped with reason).

5. Runnable example

// ConditionReportDemo.java
// How to run: java ConditionReportDemo.java  (JDK 17+)
// Simulates building a condition evaluation report similar to Spring Boot's --debug output.

import java.util.*;

public class ConditionReportDemo {

    record ConditionResult(String autoConfig, boolean passed, List<String> reasons) {}

    public static void main(String[] args) {
        // Simulate condition evaluation results collected during startup
        List<ConditionResult> results = List.of(
            new ConditionResult("JacksonAutoConfiguration", true, List.of(
                "@ConditionalOnClass found required class 'ObjectMapper' (OnClassCondition)"
            )),
            new ConditionResult("DispatcherServletAutoConfiguration", true, List.of(
                "@ConditionalOnWebApplication — is a web application (OnWebApplicationCondition)",
                "@ConditionalOnClass found required class 'DispatcherServlet' (OnClassCondition)"
            )),
            new ConditionResult("DataSourceAutoConfiguration", false, List.of(
                "@ConditionalOnClass did not find required class 'javax.sql.DataSource' (OnClassCondition)"
            )),
            new ConditionResult("MongoAutoConfiguration", false, List.of(
                "@ConditionalOnClass did not find required class 'MongoClient' (OnClassCondition)"
            )),
            new ConditionResult("SecurityAutoConfiguration", false, List.of(
                "@ConditionalOnClass found 'AbstractSecurityWebApplicationInitializer'",
                "@ConditionalOnMissingBean (types: SecurityFilterChain) — bean found [securityFilterChain]"
            ))
        );

        System.out.println("============================");
        System.out.println("CONDITIONS EVALUATION REPORT");
        System.out.println("============================\n");

        System.out.println("Positive matches:");
        System.out.println("-----------------");
        results.stream().filter(ConditionResult::passed).forEach(r -> {
            System.out.println("   " + r.autoConfig() + " matched:");
            r.reasons().forEach(reason -> System.out.println("      - " + reason));
            System.out.println();
        });

        System.out.println("Negative matches:");
        System.out.println("-----------------");
        results.stream().filter(r -> !r.passed()).forEach(r -> {
            System.out.println("   " + r.autoConfig() + ":");
            System.out.println("      Did not match:");
            r.reasons().forEach(reason -> System.out.println("         - " + reason));
            System.out.println();
        });
    }
}

6. Walkthrough

  • results is a list of simulated condition evaluations, modelled after what ConditionEvaluationReport collects internally.
  • Each ConditionResult holds: which auto-config was evaluated, whether it passed (all conditions true), and the list of reason strings Spring builds from each @Conditional annotation.
  • Positive matches stream and print with "matched:" — these are auto-configurations that actually applied beans to the context.
  • Negative matches print with "Did not match:" — the reasons tell you exactly which condition failed and why (class missing, bean already present, property not set, etc.).
  • SecurityAutoConfiguration shows a negative match with two reason lines: the first condition passed but the second failed (a SecurityFilterChain bean already existed) — illustrating that multiple conditions are evaluated and any failure stops the config.

7. Gotchas & takeaways

The condition evaluation report only covers auto-configuration classes. It does not show why a user-defined @Bean was or wasn't created — for that, use ApplicationContext.getBeanDefinitionNames() or actuator's /actuator/beans endpoint.
The debug=true flag increases log verbosity across the whole application, not just auto-configuration. In a noisy app this can scroll the report off screen. Pipe to a file: java -jar myapp.jar --debug > startup.log 2>&1, then search for CONDITIONS EVALUATION REPORT.
  • debug=true in application.properties persists across restarts; --debug on the command line is one-time. Prefer --debug for ad-hoc debugging.
  • Negative match ≠ problem. Most negative matches are expected (e.g. MongoAutoConfiguration skipped because you don't use Mongo).
  • The report is also available programmatically via ConditionEvaluationReport.get(beanFactory).
  • Spring Boot Actuator's /actuator/conditions endpoint exposes the same data as JSON for tooling integration.
  • Unexplained bean conflicts? Look in "Positive matches" to find which auto-config registered the conflicting bean.
