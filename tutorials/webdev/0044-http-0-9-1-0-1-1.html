HTTP/0.9, 1.0, 1.1

← Back to checklist

1. What it is

HTTP has evolved through three major versions before HTTP/2. Understanding the progression shows *why* each feature was added and what pain it solved:

  • HTTP/0.9 (1991): the original, one-trick protocol. A single line asks for a document, the server streams back raw HTML, then closes the connection. No headers, no status codes, no images — just text.
  • HTTP/1.0 (1996): added the request/response structure we still recognise: method + path + version on line 1, headers, blank line, body. Also added status codes (200, 404…) and Content-Type, making it possible to serve images, CSS, and other file types. Still: one request per connection.
  • HTTP/1.1 (1997, revised 2014): the version the web ran on for two decades. Added persistent connections (keep-alive by default), the mandatory Host header (enabling virtual hosting), chunked transfer encoding, range requests, and richer caching controls.

2. Why & when

The web's early simplicity became a bottleneck very quickly. A 1991 web page was a single HTML document. A 1996 page had images. A 2000 page had dozens of assets. Each version of HTTP answered the load of its era:

ProblemFixed by
Need headers, status codesHTTP/1.0
New TCP connection per request is slowHTTP/1.1 (persistent)
Many servers on one IPHTTP/1.1 (Host header)
Unknown body length upfrontHTTP/1.1 (chunked encoding)

You will see HTTP/1.1 in server logs, curl output, and anywhere TLS certificates are inspected. Most debugging tools still show 1.1-style headers even when the underlying transport is HTTP/2.

3. Core concept

Think of the evolution like restaurant ordering. HTTP/0.9 is a walk-up counter where you shout one item, get it, and the counter closes. HTTP/1.0 is a proper counter with a receipt (status code) and a paper bag label (Content-Type) — but the queue still moves one customer at a time and they lock the door after each one. HTTP/1.1 finally keeps the door open: the customer stays seated, orders multiple items in sequence, and the waiter remembers which table they're at (Host header).

Key differences at a glance:

HTTP/0.9  →  only: GET /path\r\n   (no version string, no headers)
HTTP/1.0  →  GET /path HTTP/1.0    (version; closes after each response)
HTTP/1.1  →  GET /path HTTP/1.1    (Host header required; keep-alive default)
             Host: example.com

Persistent connections (HTTP/1.1) mean the TCP handshake cost is paid once and amortised over many requests on that same connection.

The Host header lets one server IP serve foo.com and bar.com — the client says which hostname it wanted. Without it, virtual hosting (thousands of sites on one server) would be impossible.

Chunked transfer encoding lets the server start sending a response before it knows the total size — critical for dynamically generated pages.

5. Runnable example

This Node.js script shows an HTTP/1.1 server responding with the protocol-version information it receives, and a second request on the same keep-alive connection.

// save as http-versions.js  —  node http-versions.js  (no installs)
const http = require("http");
const net  = require("net");

const server = http.createServer((req, res) => {
  const info = `Protocol: ${req.httpVersion}  Method: ${req.method}  URL: ${req.url}`;
  console.log("Server sees:", info);
  res.writeHead(200, { "Content-Type": "text/plain", "Connection": "keep-alive" });
  res.end(info);
});

server.listen(3100, () => {
  // Make two requests on ONE persistent connection using the low-level socket
  const socket = net.connect(3100, "localhost", () => {
    // Request 1
    socket.write(
      "GET /first HTTP/1.1\r\nHost: localhost\r\nConnection: keep-alive\r\n\r\n"
    );
  });

  let buf = "";
  let reqCount = 0;
  socket.on("data", (chunk) => {
    buf += chunk.toString();
    // Each HTTP/1.1 response ends with a blank line before body; body ends at \n
    if (buf.includes("\r\n\r\n")) {
      reqCount++;
      const body = buf.split("\r\n\r\n").pop();
      console.log(`Client got response ${reqCount}: "${body.trim()}"`);
      buf = "";
      if (reqCount === 1) {
        // Request 2 on the SAME socket — no new TCP handshake
        socket.write(
          "GET /second HTTP/1.1\r\nHost: localhost\r\nConnection: close\r\n\r\n"
        );
      } else {
        socket.end();
        server.close();
      }
    }
  });
});

How to run: node http-versions.js — watch both requests travel over the same TCP socket.

Expected output:

Server sees: Protocol: 1.1  Method: GET  URL: /first
Client got response 1: "Protocol: 1.1  Method: GET  URL: /first"
Server sees: Protocol: 1.1  Method: GET  URL: /second
Client got response 2: "Protocol: 1.1  Method: GET  URL: /second"

6. Walkthrough

  • req.httpVersion — Node exposes the HTTP version the client sent; HTTP/1.1 requests always include this string after the method and path.
  • "Connection": "keep-alive" in the response header — tells the client the server is willing to reuse this socket.
  • net.connect(3100, ...) — opens a raw TCP socket so we can manually write HTTP/1.1 request text, making the protocol fully visible.
  • First socket.write(...) — a textbook HTTP/1.1 request: method, path, version on line 1; Host header (mandatory in 1.1); Connection: keep-alive; blank line to end headers.
  • After response 1 arrives the code writes a second request on the *same* socket — no TCP handshake, no DNS lookup. This is persistent connections in action.
  • Connection: close on request 2 signals "I'm done after this one", so the server closes the socket cleanly.

7. Gotchas & takeaways

HTTP/1.0 servers exist in the wild (some embedded devices, old proxies). If you send an HTTP/1.1 request with a Host header and get a garbled response, the server may be HTTP/1.0-only and confused by unfamiliar headers.
In HTTP/1.1 the Host header is mandatory. A missing Host results in 400 Bad Request. This is the one header you can never omit.
  • 0.9 → 1.0 → 1.1 each added structure to solve real congestion/scaling problems.
  • HTTP/1.1 from 1997 (updated RFC 7230–7235 in 2014) remained the dominant version until HTTP/2 adoption in ~2016.
  • Persistent connections save TCP handshake time (and TLS handshake on HTTPS) across requests.
  • Chunked transfer encoding (Transfer-Encoding: chunked) lets servers stream responses without knowing the final size — essential for server-side rendering.
  • Virtual hosting at scale (millions of sites on shared IPs) is only possible because of the mandatory Host header.
